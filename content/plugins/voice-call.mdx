# Voice Call プラグイン

# Voice Call (plugin)

OpenClawのための音声通話プラグイン。発信通知と着信ポリシーを用いたマルチターン会話をサポートします。

現在のプロバイダー:

* `twilio` (Programmable Voice + Media Streams)
* `telnyx` (Call Control v2)
* `plivo` (Voice API + XML transfer + GetInput speech)
* `mock` (dev/no network)

クイックメンタルモデル:

* プラグインをインストール
* Gatewayを再起動
* `plugins.entries.voice-call.config`で設定
* `openclaw voicecall ...`または`voice_call`ツールを使用

## 実行場所 (ローカル vs リモート)

Voice Callプラグインは**Gatewayプロセス内で実行**されます。

リモートGatewayを使用する場合は、**Gatewayを実行しているマシン**にプラグインをインストール/設定し、その後Gatewayを再起動して読み込みます。

## インストール

### Option A: npmからインストール (推奨)

```bash
openclaw plugins install @openclaw/voice-call
```

その後、Gatewayを再起動します。

### Option B: ローカルフォルダからインストール (開発用、コピーなし)

```bash
openclaw plugins install ./extensions/voice-call
cd ./extensions/voice-call && pnpm install
```

その後、Gatewayを再起動します。

## 設定

`plugins.entries.voice-call.config`に設定を行います:

```json5
{
  plugins: {
    entries: {
      "voice-call": {
        enabled: true,
        config: {
          provider: "twilio", // または "telnyx" | "plivo" | "mock"
          fromNumber: "+15550001234",
          toNumber: "+15550005678",

          twilio: {
            accountSid: "ACxxxxxxxx",
            authToken: "...",
          },

          telnyx: {
            apiKey: "...",
            connectionId: "...",
            // TelnyxのミッションコントロールポータルからのWebhook公開鍵
            // (Base64文字列; TELNYX_PUBLIC_KEYでも設定可能)。
            publicKey: "...",
          },

          plivo: {
            authId: "MAxxxxxxxxxxxxxxxxxxxx",
            authToken: "...",
          },

          // Webhookサーバー
          serve: {
            port: 3334,
            path: "/voice/webhook",
          },

          // Webhookセキュリティ (トンネル/プロキシ用に推奨)
          webhookSecurity: {
            allowedHosts: ["voice.example.com"],
            trustedProxyIPs: ["100.64.0.1"],
          },

          // 公開露出 (いずれかを選択)
          // publicUrl: "https://example.ngrok.app/voice/webhook",
          // tunnel: { provider: "ngrok" },
          // tailscale: { mode: "funnel", path: "/voice/webhook" }

          outbound: {
            defaultMode: "notify", // notify | conversation
          },

          streaming: {
            enabled: true,
            streamPath: "/voice/stream",
            preStartTimeoutMs: 5000,
            maxPendingConnections: 32,
            maxPendingConnectionsPerIp: 4,
            maxConnections: 128,
          },
        },
      },
    },
  },
}
```

注意:

* Twilio/Telnyxは**公開アクセス可能な**Webhook URLを必要とします。
* Plivoも**公開アクセス可能な**Webhook URLを必要とします。
* `mock`はローカル開発用プロバイダーです（ネットワーク呼び出しなし）。
* Telnyxは`telnyx.publicKey`（または`TELNYX_PUBLIC_KEY`）を必要としますが、`skipSignatureVerification`がtrueの場合は不要です。
* `skipSignatureVerification`はローカルテスト用のみです。
* ngrokの無料プランを使用する場合は、`publicUrl`を正確なngrok URLに設定してください; シグネチャ検証は常に強制されます。
* `tunnel.allowNgrokFreeTierLoopbackBypass: true`は、TwilioのWebhookが無効なシグネチャを持つ場合に**のみ**、`tunnel.provider="ngrok"`で`serve.bind`がループバック（ngrokローカルエージェント）の場合に許可します。ローカル開発用に使用してください。
* Ngrokの無料プランのURLは変更される可能性があり、間に合う動作を追加することがあります; `publicUrl`が変動すると、Twilioのシグネチャが失敗します。プロダクションでは、安定したドメインまたはTailscaleファンネルを好むべきです。
* ストリーミングセキュリティのデフォルト:
  * `streaming.preStartTimeoutMs`は、有効な`start`フレームを送信しないソケットを閉じます。
  * `streaming.maxPendingConnections`は、認証されていないプレスタートソケットの合計を制限します。
  * `streaming.maxPendingConnectionsPerIp`は、ソースIPごとの認証されていないプレスタートソケットを制限します。
  * `streaming.maxConnections`は、オープンなメディアストリームソケットの合計（保留中 + アクティブ）を制限します。

## ステールコールリーパー

`staleCallReaperSeconds`を使用して、端末Webhookを受信しないコールを終了します
（例えば、完了しない通知モードのコール）。デフォルトは`0`
（無効）。

推奨範囲:

* **プロダクション:** 通知スタイルのフローには`120`–`300`秒。
* この値は**`maxDurationSeconds`よりも高く**設定してください、そうすれば通常のコールが
  完了できます。良い出発点は`maxDurationSeconds + 30–60`秒です。

例:

```json5
{
  plugins: {
    entries: {
      "voice-call": {
        config: {
          maxDurationSeconds: 300,
          staleCallReaperSeconds: 360,
        },
      },
    },
  },
}
```

## Webhookセキュリティ

プロキシやトンネルがGatewayの前にある場合、プラグインは
シグネチャ検証のために公開URLを再構築します。これらのオプションは、どの転送された
ヘッダーが信頼されるかを制御します。

`webhookSecurity.allowedHosts`は、転送ヘッダーからのホストをホワイトリストします。

`webhookSecurity.trustForwardingHeaders`は、ホワイトリストなしで転送ヘッダーを信頼します。

`webhookSecurity.trustedProxyIPs`は、リクエストのリモートIPがリストと一致する場合にのみ
転送ヘッダーを信頼します。

Webhookリプレイ保護はTwilioとPlivoに対して有効です。リプレイされた有効なWebhook
リクエストは確認されますが、副作用のためにスキップされます。

Twilioの会話ターンには、`<Gather>`コールバック内にターンごとのトークンが含まれているため、
ステール/リプレイされたスピーチコールバックは新しい保留中のトランスクリプトターンを満たすことができません。

安定した公開ホストの例:

```json5
{
  plugins: {
    entries: {
      "voice-call": {
        config: {
          publicUrl: "https://voice.example.com/voice/webhook",
          webhookSecurity: {
            allowedHosts: ["voice.example.com"],
          },
        },
      },
    },
  },
}
```

## TTS for calls

Voice Callはコアの`messages.tts`設定（OpenAIまたはElevenLabs）を使用して
通話中のスピーチをストリーミングします。プラグインの設定で**同じ形状**で上書きできます — 
`messages.tts`と深くマージされます。

```json5
{
  tts: {
    provider: "elevenlabs",
    elevenlabs: {
      voiceId: "pMsXgVXv3BLzUgSXRplE",
      modelId: "eleven_multilingual_v2",
    },
  },
}
```

注意:

* **Edge TTSは音声通話には無視されます**（電話音声はPCMが必要; Edge出力は信頼性がありません）。
* Twilioメディアストリーミングが有効な場合はコアTTSが使用されます; そうでない場合はプロバイダーのネイティブボイスにフォールバックします。

### さらなる例

コアTTSのみを使用（上書きなし）:

```json5
{
  messages: {
    tts: {
      provider: "openai",
      openai: { voice: "alloy" },
    },
  },
}
```

通話のためだけにElevenLabsに上書き（他の場所ではコアデフォルトを維持）:

```json5
{
  plugins: {
    entries: {
      "voice-call": {
        config: {
          tts: {
            provider: "elevenlabs",
            elevenlabs: {
              apiKey: "elevenlabs_key",
              voiceId: "pMsXgVXv3BLzUgSXRplE",
              modelId: "eleven_multilingual_v2",
            },
          },
        },
      },
    },
  },
}
```

通話のためだけにOpenAIモデルを上書き（深くマージの例）:

```json5
{
  plugins: {
    entries: {
      "voice-call": {
        config: {
          tts: {
            openai: {
              model: "gpt-4o-mini-tts",
              voice: "marin",
            },
          },
        },
      },
    },
  },
}
```

## 着信通話

着信ポリシーのデフォルトは`disabled`です。着信通話を有効にするには、次のように設定します:

```json5
{
  inboundPolicy: "allowlist",
  allowFrom: ["+15550001234"],
  inboundGreeting: "こんにちは！どのようにお手伝いできますか？",
}
```

自動応答はエージェントシステムを使用します。調整には:

* `responseModel`
* `responseSystemPrompt`
* `responseTimeoutMs`

## CLI

```bash
openclaw voicecall call --to "+15555550123" --message "OpenClawからこんにちは"
openclaw voicecall continue --call-id <id> --message "何か質問はありますか？"
openclaw voicecall speak --call-id <id> --message "少々お待ちください"
openclaw voicecall end --call-id <id>
openclaw voicecall status --call-id <id>
openclaw voicecall tail
openclaw voicecall expose --mode funnel
```

## エージェントツール

ツール名: `voice_call`

アクション:

* `initiate_call` (message, to?, mode?)
* `continue_call` (callId, message)
* `speak_to_user` (callId, message)
* `end_call` (callId)
* `get_status` (callId)

このリポジトリには、`skills/voice-call/SKILL.md`に一致するスキルドキュメントが含まれています。

## Gateway RPC

* `voicecall.initiate` (`to?`, `message`, `mode?`)
* `voicecall.continue` (`callId`, `message`)
* `voicecall.speak` (`callId`, `message`)
* `voicecall.end` (`callId`)
* `voicecall.status` (`callId`)
