# secrets

# `openclaw secrets`

`openclaw secrets`を使用して、プレーンテキストからSecretRefsに資格情報を移行し、アクティブなシークレットのランタイムを健全に保ちます。

コマンドの役割:

* `reload`: gateway RPC (`secrets.reload`)で、完全成功時のみ参照を再解決し、ランタイムスナップショットを交換します（設定の書き込みはなし）。
* `audit`: プレーンテキスト、未解決の参照、および優先順位のずれを検出するための設定 + 認証ストア + レガシー残留物（`.env`, `auth.json`）の読み取り専用スキャン。
* `configure`: プロバイダーのセットアップ + ターゲットマッピング + プレフライトのためのインタラクティブプランナー（TTYが必要）。
* `apply`: 保存されたプランを実行します（検証のみのための`--dry-run`）、その後、移行されたプレーンテキストの残留物を削除します。

推奨されるオペレーターのループ:

```bash
openclaw secrets audit --check
openclaw secrets configure
openclaw secrets apply --from /tmp/openclaw-secrets-plan.json --dry-run
openclaw secrets apply --from /tmp/openclaw-secrets-plan.json
openclaw secrets audit --check
openclaw secrets reload
```

CI/ゲート用の終了コードの注意:

* `audit --check`は、発見があった場合に`1`を返し、未解決の参照がある場合は`2`を返します。

* シークレットガイド: [Secrets Management](/gateway/secrets)

## ランタイムスナップショットの再読み込み

シークレット参照を再解決し、ランタイムスナップショットを原子的に交換します。

```bash
openclaw secrets reload
openclaw secrets reload --json
```

注意事項:

* gateway RPCメソッド`secrets.reload`を使用します。
* 解決に失敗した場合、gatewayは最後の既知の良好なスナップショットを保持し、エラーを返します（部分的なアクティベーションはなし）。
* JSONレスポンスには`warningCount`が含まれます。

## 監査

OpenClawの状態をスキャンして:

* プレーンテキストのシークレットストレージ
* 未解決の参照
* 優先順位のずれ（`auth-profiles`が設定参照を隠す）
* レガシー残留物（`auth.json`、OAuthの範囲外のメモ）

```bash
openclaw secrets audit
openclaw secrets audit --check
openclaw secrets audit --json
```

終了動作:

* `--check`は、発見があった場合にゼロ以外で終了します。
* 未解決の参照は、優先度の高いゼロ以外のコードで終了します。

レポートの形状のハイライト:

* `status`: `clean | findings | unresolved`
* `summary`: `plaintextCount`, `unresolvedRefCount`, `shadowedRefCount`, `legacyResidueCount`
* 発見コード:
  * `PLAINTEXT_FOUND`
  * `REF_UNRESOLVED`
  * `REF_SHADOWED`
  * `LEGACY_RESIDUE`

## 設定（インタラクティブヘルパー）

プロバイダー + SecretRefの変更をインタラクティブに構築し、プレフライトを実行し、オプションで適用します。

```bash
openclaw secrets configure
openclaw secrets configure --plan-out /tmp/openclaw-secrets-plan.json
openclaw secrets configure --apply --yes
openclaw secrets configure --providers-only
openclaw secrets configure --skip-provider-setup
openclaw secrets configure --json
```

フロー:

* 最初にプロバイダーのセットアップ（`secrets.providers`エイリアスの`add/edit/remove`）。
* 次に資格情報のマッピング（フィールドを選択し、`{source, provider, id}`参照を割り当て）。
* 最後にプレフライトとオプションの適用。

フラグ:

* `--providers-only`: `secrets.providers`のみを設定し、資格情報のマッピングをスキップします。
* `--skip-provider-setup`: プロバイダーのセットアップをスキップし、既存のプロバイダーに資格情報をマッピングします。

注意事項:

* インタラクティブなTTYが必要です。
* `--providers-only`と`--skip-provider-setup`を組み合わせることはできません。
* `configure`は`openclaw.json`内のシークレットを持つフィールドを対象とします。
* 移行する予定のすべてのシークレットを持つフィールドを含めてください（例えば、`models.providers.*.apiKey`と`skills.entries.*.apiKey`の両方）ので、監査がクリーンな状態に到達できます。
* 適用前にプレフライト解決を実行します。
* 生成されたプランはデフォルトでスクラブオプションを有効にします（`scrubEnv`, `scrubAuthProfilesForProviderTargets`, `scrubLegacyAuthJson`がすべて有効）。
* 適用パスは移行されたプレーンテキスト値に対して一方向です。
* `--apply`なしでも、CLIはプレフライト後に`このプランを今すぐ適用しますか？`とプロンプトを表示します。
* `--apply`（および`--yes`なし）では、CLIは追加の不可逆的な移行確認を促します。

プロバイダーの安全性に関する注意:

* Homebrewのインストールは、しばしば`/opt/homebrew/bin/*`の下にシンボリックリンクされたバイナリを公開します。
* 信頼できるパッケージマネージャーパスのために必要な場合にのみ`allowSymlinkCommand: true`を設定し、`trustedDirs`と組み合わせて使用します（例えば`["/opt/homebrew"]`）。

## 保存されたプランを適用

以前に生成されたプランを適用またはプレフライトします。

```bash
openclaw secrets apply --from /tmp/openclaw-secrets-plan.json
openclaw secrets apply --from /tmp/openclaw-secrets-plan.json --dry-run
openclaw secrets apply --from /tmp/openclaw-secrets-plan.json --json
```

プラン契約の詳細（許可されたターゲットパス、検証ルール、および失敗の意味）:

* [Secrets Apply Plan Contract](/gateway/secrets-plan-contract)

`apply`が更新する可能性のあるもの:

* `openclaw.json`（SecretRefターゲット + プロバイダーのアップサート/削除）
* `auth-profiles.json`（プロバイダーターゲットのスクラブ）
* レガシー`auth.json`の残留物
* `~/.openclaw/.env`にある、値が移行された既知のシークレットキー

## なぜロールバックバックアップがないのか

`secrets apply`は、古いプレーンテキスト値を含むロールバックバックアップを意図的に書き込みません。

安全性は、厳格なプレフライト + 原子的な適用に由来し、失敗時には最善の努力でメモリ内の復元を行います。

## 例

```bash
# 最初に監査し、次に設定し、最後にクリーンを確認します:
openclaw secrets audit --check
openclaw secrets configure
openclaw secrets audit --check
```

部分的な移行後に`audit --check`がプレーンテキストの発見を報告し続ける場合、スキルキー（`skills.entries.*.apiKey`）や他の報告されたターゲットパスも移行されたことを確認してください。
