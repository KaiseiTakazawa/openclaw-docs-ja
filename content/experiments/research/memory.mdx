# ワークスペースメモリ研究

# Workspace Memory v2 (offline): research notes

ターゲット: Clawdスタイルのワークスペース（`agents.defaults.workspace`、デフォルトは`~/.openclaw/workspace`）で、「メモリ」は1日ごとに1つのMarkdownファイル（`memory/YYYY-MM-DD.md`）として保存され、安定したファイルの小さなセット（例: `memory.md`, `SOUL.md`）が追加されます。

この文書は、Markdownを正規の、レビュー可能な真実のソースとして保持しつつ、派生インデックスを介して**構造化されたリコール**（検索、エンティティ要約、信頼度更新）を追加する**オフラインファースト**のメモリアーキテクチャを提案します。

## なぜ変更するのか？

現在のセットアップ（1日1ファイル）は、以下の点で優れています：

* 「追加のみ」のジャーナリング
* 人間による編集
* Gitによる耐久性と監査可能性
* 低摩擦のキャプチャ（「ただ書き留めるだけ」）

しかし、以下の点では弱いです：

* 高リコールの取得（「Xについて何を決めたか？」、「前回Yを試したのはいつ？」）
* エンティティ中心の回答（「アリスについて教えて」/「城について」/「warelayについて」）を多くのファイルを再読せずに得ること
* 意見/好みの安定性（そして変更時の証拠）
* 時間的制約（「2025年11月の間に何が真実だったか？」）と対立解決

## 設計目標

* **オフライン**: ネットワークなしで動作; ノートパソコン/Castleで実行可能; クラウド依存なし。
* **説明可能**: 取得したアイテムは帰属可能（ファイル + 位置）で、推論から分離可能であるべき。
* **低儀式**: 日々のログはMarkdownのままで、重いスキーマ作業は不要。
* **インクリメンタル**: v1はFTSのみで有用; セマンティック/ベクトルおよびグラフはオプションのアップグレード。
* **エージェントフレンドリー**: 「トークン予算内でのリコール」を容易にする（小さな事実の束を返す）。

## ノーススターモデル（Hindsight × Letta）

組み合わせる2つの要素：

1. **Letta/MemGPTスタイルの制御ループ**

* 小さな「コア」を常にコンテキスト内に保持（ペルソナ + 主要なユーザ事実）
* その他はコンテキスト外でツールを介して取得
* メモリの書き込みは明示的なツール呼び出し（追加/置換/挿入）であり、永続化され、次のターンで再注入される

2. **Hindsightスタイルのメモリ基盤**

* 観察されたものと信じられているもの、要約されたものを分離
* 保持/リコール/反映をサポート
* 証拠と共に進化する信頼性を持つ意見
* エンティティを意識した取得 + 時間的クエリ（完全な知識グラフがなくても）

## 提案されたアーキテクチャ（Markdown真実のソース + 派生インデックス）

### 正規ストア（gitフレンドリー）

`~/.openclaw/workspace`を正規の人間可読メモリとして保持します。

提案されたワークスペースのレイアウト：

```
~/.openclaw/workspace/
  memory.md                    # 小さな: 耐久性のある事実 + 好み（コア的）
  memory/
    YYYY-MM-DD.md              # 日次ログ（追加; ナラティブ）
  bank/                        # 「型付き」メモリページ（安定、レビュー可能）
    world.md                   # 世界に関する客観的事実
    experience.md              # エージェントが行ったこと（第一人称）
    opinions.md                # 主観的な好み/判断 + 信頼度 + 証拠のポイント
    entities/
      Peter.md
      The-Castle.md
      warelay.md
      ...
```

注意：

* **日次ログは日次ログのまま**。JSONに変換する必要はありません。
* `bank/`ファイルは**キュレーションされており**、反映作業によって生成され、手動で編集することもできます。
* `memory.md`は「小さく + コア的」であり続けます：Clawdが毎回見るべきものです。

### 派生ストア（機械リコール）

ワークスペースの下に派生インデックスを追加します（必ずしもgitで追跡されるわけではありません）：

```
~/.openclaw/workspace/.memory/index.sqlite
```

以下でバックアップします：

* 事実 + エンティティリンク + 意見メタデータのためのSQLiteスキーマ
* レキシカルリコールのためのSQLite **FTS5**（高速、コンパクト、オフライン）
* セマンティックリコールのためのオプションの埋め込みテーブル（依然としてオフライン）

インデックスは常に**Markdownから再構築可能**です。

## 保持 / リコール / 反映（運用ループ）

### 保持: 日次ログを「事実」に正規化

ここで重要なHindsightの洞察：**ナラティブで自己完結した事実**を保存します。

`memory/YYYY-MM-DD.md`の実用的なルール：

* 日の終わり（またはその間）に、2〜5の箇条書きを持つ`## Retain`セクションを追加します：
  * ナラティブ（ターン間のコンテキストが保持される）
  * 自己完結（後で独立して意味を持つ）
  * タイプ + エンティティの言及でタグ付け

例：

```
## Retain
- W @Peter: 現在マラケシュにいる（2025年11月27日〜12月1日）アンディの誕生日のため。
- B @warelay: Baileys WSのクラッシュを修正しました。connection.updateハンドラをtry/catchでラップしました（memory/2025-11-27.mdを参照）。
- O(c=0.95) @Peter: WhatsAppでの簡潔な返信を好む（<1500文字）；長いコンテンツはファイルに入れる。
```

最小限の解析：

* タイププレフィックス: `W`（世界）、`B`（経験/伝記）、`O`（意見）、`S`（観察/要約; 通常は生成される）
* エンティティ: `@Peter`, `@warelay`など（スラグは`bank/entities/*.md`にマッピング）
* 意見の信頼度: `O(c=0.0..1.0)`はオプション

著者に考えさせたくない場合：反映作業はログの残りからこれらの箇条書きを推測できますが、明示的な`## Retain`セクションを持つことが最も簡単な「品質レバー」です。

### リコール: 派生インデックスに対するクエリ

リコールは以下をサポートするべきです：

* **レキシカル**: 「正確な用語/名前/コマンドを見つける」（FTS5）
* **エンティティ**: 「Xについて教えて」（エンティティページ + エンティティリンクされた事実）
* **時間的**: 「11月27日頃に何が起こったか」/「先週から」
* **意見**: 「ピーターは何を好むか？」（信頼度 + 証拠付き）

返却形式はエージェントフレンドリーで、ソースを引用するべきです：

* `kind`（`world|experience|opinion|observation`）
* `timestamp`（ソース日、または存在する場合は抽出された時間範囲）
* `entities`（`["Peter","warelay"]`）
* `content`（ナラティブな事実）
* `source`（`memory/2025-11-27.md#L12`など）

### 反映: 安定したページを生成し、信念を更新

反映はスケジュールされたジョブ（毎日またはハートビート`ultrathink`）で：

* 最近の事実から`bank/entities/*.md`を更新（エンティティ要約）
* 強化/矛盾に基づいて`bank/opinions.md`の信頼度を更新
* オプションで`memory.md`の編集を提案（「コア的」耐久性のある事実）

意見の進化（シンプルで説明可能）：

* 各意見には：
  * ステートメント
  * 信頼度 `c ∈ [0,1]`
  * last_updated
  * 証拠リンク（支持する + 矛盾する事実ID）
* 新しい事実が到着したとき：
  * エンティティの重複 + 類似性によって候補意見を見つける（最初はFTS、後で埋め込み）
  * 小さなデルタで信頼度を更新; 大きなジャンプには強い矛盾 + 繰り返しの証拠が必要

## CLI統合: スタンドアロン vs 深い統合

推奨: **OpenClawへの深い統合**、しかし分離可能なコアライブラリを保持。

### なぜOpenClawに統合するのか？

* OpenClawはすでに以下を知っています：
  * ワークスペースパス（`agents.defaults.workspace`）
  * セッションモデル + ハートビート
  * ロギング + トラブルシューティングパターン
* エージェント自体にツールを呼び出させたい：
  * `openclaw memory recall "…" --k 25 --since 30d`
  * `openclaw memory reflect --since 7d`

### なぜまだライブラリを分けるのか？

* ゲートウェイ/ランタイムなしでメモリロジックをテスト可能に保つ
* 他のコンテキスト（ローカルスクリプト、将来のデスクトップアプリなど）から再利用

形状：
メモリツールは小さなCLI + ライブラリレイヤーを意図していますが、これは探索的なものに過ぎません。

## “S-Collide” / SuCo: いつ使用するか（研究）

「S-Collide」が**SuCo（サブスペース衝突）**を指す場合：これは、学習された/構造化された衝突をサブスペースで使用することによって強いリコール/レイテンシトレードオフをターゲットとするANN取得アプローチです（論文: arXiv 2411.14754, 2024）。

`~/.openclaw/workspace`の実用的な見解：

* **SuCoで始めない**。
* SQLite FTS + （オプションの）シンプルな埋め込みから始めれば、すぐにほとんどのUXの利点が得られます。
* SuCo/HNSW/ScaNNクラスのソリューションは次のような場合にのみ検討します：
  * コーパスが大きい（数万/数十万のチャンク）
  * ブルートフォースの埋め込み検索が遅すぎる
  * リコールの質がレキシカル検索によって意味的にボトルネックされている

オフラインフレンドリーな代替案（複雑さの増加に伴い）：

* SQLite FTS5 + メタデータフィルター（ゼロML）
* 埋め込み + ブルートフォース（チャンク数が少ない場合は驚くほど機能する）
* HNSWインデックス（一般的で堅牢; ライブラリバインディングが必要）
* SuCo（研究グレード; 組み込むことができる堅実な実装があれば魅力的）

オープンクエスチョン：

* あなたのマシン（ノートパソコン + デスクトップ）での「パーソナルアシスタントメモリ」に最適な**オフライン埋め込みモデル**は何ですか？
  * すでにOllamaを持っている場合：ローカルモデルで埋め込む; そうでなければ、ツールチェーンに小さな埋め込みモデルを出荷します。

## 最小限の有用なパイロット

最小限でありながら有用なバージョンが必要な場合：

* `bank/`エンティティページと日次ログに`## Retain`セクションを追加します。
* 引用付きでリコールのためにSQLite FTSを使用します（パス + 行番号）。
* リコールの質やスケールが要求する場合にのみ埋め込みを追加します。
