# RPC アダプター

OpenClaw は JSON-RPC を介して外部 CLI を統合します。現在、2つのパターンが使用されています。

## パターン A: HTTP デーモン (signal-cli)

* `signal-cli` は HTTP 上で JSON-RPC を使用してデーモンとして実行されます。
* イベントストリームは SSE (`/api/v1/events`) です。
* ヘルスプローブ: `/api/v1/check`。
* `channels.signal.autoStart=true` の場合、OpenClaw がライフサイクルを管理します。

設定とエンドポイントについては [Signal](/channels/signal) を参照してください。

## パターン B: stdio 子プロセス (レガシー: imsg)

> **注意:** 新しい iMessage の設定には、代わりに [BlueBubbles](/channels/bluebubbles) を使用してください。

* OpenClaw は `imsg rpc` を子プロセスとして生成します（レガシー iMessage 統合）。
* JSON-RPC は stdin/stdout 上で行ごとに区切られた形式で（1行につき1つの JSON オブジェクト）使用されます。
* TCP ポートは不要で、デーモンも必要ありません。

使用されるコアメソッド:

* `watch.subscribe` → 通知 (`method: "message"`)
* `watch.unsubscribe`
* `send`
* `chats.list` (プローブ/診断)

レガシー設定とアドレッシングについては [iMessage](/channels/imessage) を参照してください（`chat_id` が推奨されます）。

## アダプターガイドライン

* Gateway がプロセスを管理します（プロバイダーのライフサイクルに関連付けられた開始/停止）。
* RPC クライアントを堅牢に保つ: タイムアウト、終了時の再起動。
* 表示文字列よりも安定した ID（例: `chat_id`）を優先してください。
