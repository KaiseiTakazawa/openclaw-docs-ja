# メッセージ

# メッセージ

このページでは、OpenClawが受信メッセージ、セッション、キューイング、ストリーミング、および推論の可視性をどのように処理するかをまとめています。

## メッセージフロー（高レベル）

```
受信メッセージ
  -> ルーティング/バインディング -> セッションキー
  -> キュー（実行中の場合）
  -> エージェント実行（ストリーミング + ツール）
  -> アウトバウンド返信（チャネル制限 + チャンク化）
```

主要な設定は構成にあります：

* `messages.*` はプレフィックス、キューイング、およびグループの動作に使用されます。
* `agents.defaults.*` はブロックストリーミングとチャンク化のデフォルトに使用されます。
* チャネルのオーバーライド（`channels.whatsapp.*`、`channels.telegram.*`など）は、制限とストリーミングの切り替えに使用されます。

完全なスキーマについては、[Configuration](/gateway/configuration)を参照してください。

## 受信メッセージの重複排除

チャネルは再接続後に同じメッセージを再配信することがあります。OpenClawは、チャネル/アカウント/ピア/セッション/メッセージIDでキー付けされた短命のキャッシュを保持しており、重複配信が別のエージェント実行をトリガーしないようにしています。

## 受信メッセージのデバウンス

**同じ送信者**からの迅速な連続メッセージは、`messages.inbound`を介して単一のエージェントターンにバッチ処理されることがあります。デバウンスはチャネル + 会話ごとにスコープされ、最も最近のメッセージが返信スレッド/IDに使用されます。

設定（グローバルデフォルト + チャネルごとのオーバーライド）：

```json5
{
  messages: {
    inbound: {
      debounceMs: 2000,
      byChannel: {
        whatsapp: 5000,
        slack: 1500,
        discord: 1500,
      },
    },
  },
}
```

注意：

* デバウンスは**テキストのみ**のメッセージに適用されます。メディア/添付ファイルは即座にフラッシュされます。
* コントロールコマンドはデバウンスをバイパスするため、独立したものとして残ります。

## セッションとデバイス

セッションはゲートウェイが所有し、クライアントが所有するものではありません。

* ダイレクトチャットはエージェントのメインセッションキーに統合されます。
* グループ/チャネルはそれぞれ独自のセッションキーを持ちます。
* セッションストアとトランスクリプトはゲートウェイホスト上に存在します。

複数のデバイス/チャネルが同じセッションにマッピングできますが、履歴はすべてのクライアントに完全に同期されません。推奨事項：長い会話には1つの主要デバイスを使用して、文脈の分岐を避けてください。Control UIとTUIは常にゲートウェイに基づいたセッショントランスクリプトを表示するため、真実の源です。

詳細については、[Session management](/concepts/session)を参照してください。

## 受信ボディと履歴コンテキスト

OpenClawは**プロンプトボディ**と**コマンドボディ**を分離します：

* `Body`: エージェントに送信されるプロンプトテキスト。これにはチャネルエンベロープやオプションの履歴ラッパーが含まれる場合があります。
* `CommandBody`: 指令/コマンド解析のための生のユーザーテキスト。
* `RawBody`: 互換性のために保持されている`CommandBody`のレガシーエイリアス。

チャネルが履歴を提供する場合、共有ラッパーを使用します：

* `[あなたの最後の返信からのチャットメッセージ - コンテキスト用]`
* `[現在のメッセージ - これに応答]`

**非ダイレクトチャット**（グループ/チャネル/ルーム）では、**現在のメッセージボディ**は送信者ラベルでプレフィックスされます（履歴エントリに使用されるのと同じスタイル）。これにより、リアルタイムメッセージとキュー/履歴メッセージがエージェントプロンプト内で一貫性を保ちます。

履歴バッファは**保留中のみ**です：実行をトリガーしなかったグループメッセージ（たとえば、メンションゲートされたメッセージ）を含み、セッショントランスクリプトに既にあるメッセージは**除外**されます。

指令の除去は**現在のメッセージ**セクションにのみ適用されるため、履歴はそのまま残ります。履歴をラップするチャネルは、`CommandBody`（または`RawBody`）を元のメッセージテキストに設定し、`Body`を結合されたプロンプトとして保持する必要があります。履歴バッファは、`messages.groupChat.historyLimit`（グローバルデフォルト）や、`channels.slack.historyLimit`や`channels.telegram.accounts.<id>.historyLimit`のようなチャネルごとのオーバーライドを介して設定可能です（無効にするには`0`を設定）。

## キューイングとフォローアップ

実行中の場合、受信メッセージはキューに入れられたり、現在の実行に誘導されたり、フォローアップターンのために収集されたりすることができます。

* `messages.queue`（および`messages.queue.byChannel`）を介して設定します。
* モード：`interrupt`、`steer`、`followup`、`collect`、およびバックログバリアント。

詳細については、[Queueing](/concepts/queue)を参照してください。

## ストリーミング、チャンク化、およびバッチ処理

ブロックストリーミングは、モデルがテキストブロックを生成する際に部分的な返信を送信します。チャンク化はチャネルのテキスト制限を尊重し、フェンスで囲まれたコードを分割しないようにします。

主要な設定：

* `agents.defaults.blockStreamingDefault`（`on|off`、デフォルトはオフ）
* `agents.defaults.blockStreamingBreak`（`text_end|message_end`）
* `agents.defaults.blockStreamingChunk`（`minChars|maxChars|breakPreference`）
* `agents.defaults.blockStreamingCoalesce`（アイドルベースのバッチ処理）
* `agents.defaults.humanDelay`（ブロック返信間の人間のような一時停止）
* チャネルオーバーライド：`*.blockStreaming`および`*.blockStreamingCoalesce`（非Telegramチャネルは明示的に`*.blockStreaming: true`が必要）

詳細については、[Streaming + chunking](/concepts/streaming)を参照してください。

## 推論の可視性とトークン

OpenClawはモデルの推論を公開または非公開にできます：

* `/reasoning on|off|stream`は可視性を制御します。
* 推論コンテンツは、モデルによって生成された場合、トークン使用量にカウントされます。
* Telegramはドラフトバブルへの推論ストリームをサポートしています。

詳細については、[Thinking + reasoning directives](/tools/thinking)および[Token use](/reference/token-use)を参照してください。

## プレフィックス、スレッド、および返信

アウトバウンドメッセージのフォーマットは`messages`に集中しています：

* `messages.responsePrefix`、`channels.<channel>.responsePrefix`、および`channels.<channel>.accounts.<id>.responsePrefix`（アウトバウンドプレフィックスのカスケード）、さらに`channels.whatsapp.messagePrefix`（WhatsAppの受信プレフィックス）
* 返信スレッドは`replyToMode`およびチャネルごとのデフォルトを介して行われます

詳細については、[Configuration](/gateway/configuration#messages)およびチャネルドキュメントを参照してください。
