# メモリ

OpenClawのメモリは**エージェントワークスペース内のプレーンMarkdown**です。ファイルが真実の源であり、モデルはディスクに書き込まれたものだけを「記憶」します。

メモリ検索ツールは、アクティブなメモリプラグイン（デフォルト: `memory-core`）によって提供されます。メモリプラグインを無効にするには、`plugins.slots.memory = "none"`を使用します。

## メモリファイル（Markdown）

デフォルトのワークスペースレイアウトは、2つのメモリレイヤーを使用します：

* `memory/YYYY-MM-DD.md`
  * 日次ログ（追加のみ）。
  * セッション開始時に今日と昨日を読み込みます。
* `MEMORY.md`（オプション）
  * 精選された長期メモリ。
  * **メインのプライベートセッションでのみ読み込み**（グループコンテキストでは決して読み込まない）。

これらのファイルはワークスペースの下に存在します（`agents.defaults.workspace`、デフォルトは`~/.openclaw/workspace`）。完全なレイアウトについては[Agent workspace](/concepts/agent-workspace)を参照してください。

## メモリツール

OpenClawは、これらのMarkdownファイル用に2つのエージェント向けツールを公開しています：

* `memory_search` — インデックスされたスニペットに対する意味的リコール。
* `memory_get` — 特定のMarkdownファイル/行範囲のターゲットリード。

`memory_get`は、**ファイルが存在しない場合に優雅に劣化します**（例えば、最初の書き込み前の今日の日次ログ）。組み込みマネージャーとQMDバックエンドは、`ENOENT`をスローする代わりに`{ text: "", path }`を返すため、エージェントは「まだ何も記録されていない」と処理し、ツール呼び出しをtry/catchロジックでラップせずにワークフローを続けることができます。

## メモリを書くタイミング

* 決定、好み、持続的な事実は`MEMORY.md`に書き込みます。
* 日常のメモや実行中のコンテキストは`memory/YYYY-MM-DD.md`に書き込みます。
* 誰かが「これを覚えておいて」と言ったら、メモを取ります（RAMに保持しないでください）。
* この領域はまだ進化中です。モデルにメモリを保存するようにリマインドするのが役立ちます；モデルは何をすべきかを知っています。
* 何かをしっかりと記憶させたい場合は、**ボットにメモリに書き込むように依頼してください**。

## 自動メモリフラッシュ（プレコンパクションピング）

セッションが**自動コンパクションに近づくと**、OpenClawは**サイレントでエージェント的なターン**をトリガーし、コンテキストがコンパクト化される**前に**モデルに持続的なメモリを書き込むようにリマインドします。デフォルトのプロンプトは、モデルが*応答する可能性がある*ことを明示的に示しますが、通常は`NO_REPLY`が正しい応答であり、ユーザーはこのターンを決して見ることはありません。

これは`agents.defaults.compaction.memoryFlush`によって制御されます：

```json5
{
  agents: {
    defaults: {
      compaction: {
        reserveTokensFloor: 20000,
        memoryFlush: {
          enabled: true,
          softThresholdTokens: 4000,
          systemPrompt: "セッションがコンパクションに近づいています。持続的なメモリを今すぐ保存してください。",
          prompt: "memory/YYYY-MM-DD.mdに持続的なメモを記入してください；保存するものがない場合はNO_REPLYで応答してください。",
        },
      },
    },
  },
}
```

詳細：

* **ソフト閾値**：フラッシュは、セッショントークンの見積もりが`contextWindow - reserveTokensFloor - softThresholdTokens`を超えたときにトリガーされます。
* **デフォルトでサイレント**：プロンプトには`NO_REPLY`が含まれているため、何も配信されません。
* **2つのプロンプト**：ユーザープロンプトとシステムプロンプトがリマインダーを追加します。
* **コンパクションサイクルごとに1回のフラッシュ**（`sessions.json`で追跡）。
* **ワークスペースは書き込み可能でなければならない**：セッションが`workspaceAccess: "ro"`または`"none"`でサンドボックス実行される場合、フラッシュはスキップされます。

完全なコンパクションライフサイクルについては、[セッション管理 + コンパクション](/reference/session-management-compaction)を参照してください。

## ベクトルメモリ検索

OpenClawは`MEMORY.md`と`memory/*.md`に小さなベクトルインデックスを構築できるため、意味的クエリは言葉が異なっていても関連するメモを見つけることができます。

デフォルト：

* デフォルトで有効。
* メモリファイルの変更を監視します（デバウンス）。
* メモリ検索は`agents.defaults.memorySearch`で構成します（トップレベルの`memorySearch`ではありません）。
* デフォルトではリモート埋め込みを使用します。`memorySearch.provider`が設定されていない場合、OpenClawは自動的に選択します：
  1. `memorySearch.local.modelPath`が構成されていてファイルが存在する場合は`local`。
  2. OpenAIキーが解決できる場合は`openai`。
  3. Geminiキーが解決できる場合は`gemini`。
  4. Voyageキーが解決できる場合は`voyage`。
  5. Mistralキーが解決できる場合は`mistral`。
  6. それ以外の場合、メモリ検索は構成されるまで無効のままです。
* ローカルモードはnode-llama-cppを使用し、`pnpm approve-builds`が必要な場合があります。
* SQLite内でのベクトル検索を加速するためにsqlite-vecを使用します（利用可能な場合）。

リモート埋め込みには、埋め込みプロバイダーのAPIキーが**必要**です。OpenClawは、認証プロファイル、`models.providers.*.apiKey`、または環境変数からキーを解決します。Codex OAuthはチャット/完了のみをカバーし、メモリ検索の埋め込みには**適合しません**。Geminiの場合は`GEMINI_API_KEY`または`models.providers.google.apiKey`を使用します。Voyageの場合は`VOYAGE_API_KEY`または`models.providers.voyage.apiKey`を使用します。Mistralの場合は`MISTRAL_API_KEY`または`models.providers.mistral.apiKey`を使用します。カスタムのOpenAI互換エンドポイントを使用する場合は、`memorySearch.remote.apiKey`（およびオプションの`memorySearch.remote.headers`）を設定します。

### QMDバックエンド（実験的）

`memory.backend = "qmd"`を設定すると、組み込みのSQLiteインデクサーを[QMD](https://github.com/tobi/qmd)に置き換えます：BM25 + ベクトル + 再ランキングを組み合わせたローカルファーストの検索サイドカーです。Markdownは真実の源のままで、OpenClawは取得のためにQMDにシェルアウトします。重要なポイント：

**前提条件**

* デフォルトで無効。設定ごとにオプトイン（`memory.backend = "qmd"`）。
* QMD CLIを別途インストールする必要があります（`bun install -g https://github.com/tobi/qmd`またはリリースを取得）し、`qmd`バイナリがゲートウェイの`PATH`にあることを確認してください。
* QMDは拡張機能を許可するSQLiteビルドが必要です（macOSでは`brew install sqlite`）。
* QMDはBun + `node-llama-cpp`を介して完全にローカルで実行され、初回使用時にHuggingFaceからGGUFモデルを自動ダウンロードします（別途Ollamaデーモンは必要ありません）。
* ゲートウェイは、`XDG_CONFIG_HOME`と`XDG_CACHE_HOME`を設定することにより、`~/.openclaw/agents/<agentId>/qmd/`の下に自己完結型のQMDホームを実行します。
* OSサポート：macOSとLinuxはBun + SQLiteがインストールされていればすぐに動作します。WindowsはWSL2を介して最もサポートされています。

**サイドカーの実行方法**

* ゲートウェイは、`~/.openclaw/agents/<agentId>/qmd/`の下に自己完結型のQMDホームを作成します（設定 + キャッシュ + SQLite DB）。
* コレクションは、`memory.qmd.paths`から`qmd collection add`を介して作成され（デフォルトのワークスペースメモリファイルを含む）、その後`qmd update` + `qmd embed`が起動時と設定可能な間隔で実行されます（`memory.qmd.update.interval`、デフォルトは5分）。
* ゲートウェイは、起動時にQMDマネージャーを初期化するため、最初の`memory_search`呼び出しの前でも定期的な更新タイマーがアームされます。
* ブートリフレッシュはデフォルトでバックグラウンドで実行されるため、チャットの起動がブロックされません。以前のブロッキング動作を維持するには、`memory.qmd.update.waitForBootSync = true`を設定します。
* 検索は`memory.qmd.searchMode`を介して実行されます（デフォルトは`qmd search --json`；`vsearch`と`query`もサポート）。選択したモードがQMDビルドでフラグを拒否する場合、OpenClawは`qmd query`で再試行します。QMDが失敗するか、バイナリが欠落している場合、OpenClawは自動的に組み込みのSQLiteマネージャーにフォールバックし、メモリツールが機能し続けます。
* OpenClawは現在、QMD埋め込みバッチサイズの調整を公開していません；バッチ動作はQMD自体によって制御されます。
* **最初の検索は遅くなる可能性があります**：QMDは最初の`qmd query`実行時にローカルGGUFモデル（再ランキング/クエリ拡張）をダウンロードする場合があります。
  * OpenClawはQMDを実行するときに`XDG_CONFIG_HOME`/`XDG_CACHE_HOME`を自動的に設定します。
  * モデルを手動で事前ダウンロードしたい場合（OpenClawが使用するのと同じインデックスをウォームアップするため）、エージェントのXDGディレクトリを使用して一度限りのクエリを実行します。

    OpenClawのQMD状態はあなたの**状態ディレクトリ**の下に存在します（デフォルトは`~/.openclaw`）。
    OpenClawが使用するのと同じインデックスを指すには、次のXDG変数をエクスポートします：

    ```bash
    # OpenClawが使用するのと同じ状態ディレクトリを選択
    STATE_DIR="${OPENCLAW_STATE_DIR:-$HOME/.openclaw}"

    export XDG_CONFIG_HOME="$STATE_DIR/agents/main/qmd/xdg-config"
    export XDG_CACHE_HOME="$STATE_DIR/agents/main/qmd/xdg-cache"

    # （オプション）インデックスのリフレッシュ + 埋め込みを強制
    qmd update
    qmd embed

    # 初回モデルダウンロードをウォームアップ/トリガー
    qmd query "test" -c memory-root --json >/dev/null 2>&1
    ```

**設定サーフェス（`memory.qmd.*`）**

* `command`（デフォルトは`qmd`）：実行可能パスをオーバーライドします。
* `searchMode`（デフォルトは`search`）：`memory_search`をバックアップするQMDコマンドを選択します（`search`、`vsearch`、`query`）。
* `includeDefaultMemory`（デフォルトは`true`）：`MEMORY.md` + `memory/**/*.md`を自動インデックスします。
* `paths[]`：追加のディレクトリ/ファイルを追加します（`path`、オプションの`pattern`、オプションの安定した`name`）。
* `sessions`：セッションJSONLインデックスにオプトインします（`enabled`、`retentionDays`、`exportDir`）。
* `update`：リフレッシュの頻度とメンテナンスの実行を制御します：
  （`interval`、`debounceMs`、`onBoot`、`waitForBootSync`、`embedInterval`、`commandTimeoutMs`、`updateTimeoutMs`、`embedTimeoutMs`）。
* `limits`：リコールペイロードを制限します（`maxResults`、`maxSnippetChars`、`maxInjectedChars`、`timeoutMs`）。
* `scope`：[`session.sendPolicy`](/gateway/configuration#session)と同じスキーマです。デフォルトはDM専用（すべてを`deny`、直接チャットを`allow`）；グループ/チャンネルでQMDヒットを表示するために緩和します。
  * `match.keyPrefix`は**正規化された**セッションキー（小文字化され、先頭の`agent:<id>:`が削除されたもの）に一致します。例：`discord:channel:`。
  * `match.rawKeyPrefix`は**生の**セッションキー（小文字化され、`agent:<id>:`を含む）に一致します。例：`agent:main:discord:`。
  * レガシー：`match.keyPrefix: "agent:..."`は依然として生のキー接頭辞として扱われますが、明確さのために`rawKeyPrefix`を優先してください。
* `scope`が検索を拒否すると、OpenClawは派生した`channel`/`chatType`で警告を記録し、空の結果のデバッグが容易になります。
* ワークスペース外から取得されたスニペットは、`memory_search`の結果に`qmd/<collection>/<relative-path>`として表示されます；`memory_get`はそのプレフィックスを理解し、構成されたQMDコレクションルートから読み取ります。
* `memory.qmd.sessions.enabled = true`のとき、OpenClawはセッションのトランスクリプト（ユーザー/アシスタントターン）を`~/.openclaw/agents/<id>/qmd/sessions/`の専用QMDコレクションにエクスポートし、`memory_search`が最近の会話をリコールできるようにします。これは組み込みのSQLiteインデックスに触れることなく行われます。
* `memory_search`のスニペットには、`memory.citations`が`auto`/`on`のときに`Source: <path#line>`のフッターが含まれます；`memory.citations = "off"`を設定すると、パスメタデータが内部に保持されます（エージェントは`memory_get`のためにパスを受け取りますが、スニペットテキストはフッターを省略し、システムプロンプトはエージェントに引用しないよう警告します）。

**例**

```json5
memory: {
  backend: "qmd",
  citations: "auto",
  qmd: {
    includeDefaultMemory: true,
    update: { interval: "5m", debounceMs: 15000 },
    limits: { maxResults: 6, timeoutMs: 4000 },
    scope: {
      default: "deny",
      rules: [
        { action: "allow", match: { chatType: "direct" } },
        // 正規化されたセッションキー接頭辞（`agent:<id>:`を削除）。
        { action: "deny", match: { keyPrefix: "discord:channel:" } },
        // 生のセッションキー接頭辞（`agent:<id>:`を含む）。
        { action: "deny", match: { rawKeyPrefix: "agent:main:discord:" } },
      ]
    },
    paths: [
      { name: "docs", path: "~/notes", pattern: "**/*.md" }
    ]
  }
}
```

**引用とフォールバック**

* `memory.citations`はバックエンドに関係なく適用されます（`auto`/`on`/`off`）。
* `qmd`が実行されると、`status().backend = "qmd"`をタグ付けし、診断がどのエンジンが結果を提供したかを示します。QMDサブプロセスが終了するか、JSON出力が解析できない場合、検索マネージャーは警告を記録し、組み込みプロバイダー（既存のMarkdown埋め込み）を返します。

### 追加のメモリパス

デフォルトのワークスペースレイアウトの外部にMarkdownファイルをインデックスしたい場合は、明示的なパスを追加します：

```json5
agents: {
  defaults: {
    memorySearch: {
      extraPaths: ["../team-docs", "/srv/shared-notes/overview.md"]
    }
  }
}
```

ノート：

* パスは絶対またはワークスペース相対で指定できます。
* ディレクトリは`.md`ファイルのために再帰的にスキャンされます。
* インデックスされるのはMarkdownファイルのみです。
* シンボリックリンクは無視されます（ファイルまたはディレクトリ）。

### Gemini埋め込み（ネイティブ）

プロバイダーを`gemini`に設定して、Gemini埋め込みAPIを直接使用します：

```json5
agents: {
  defaults: {
    memorySearch: {
      provider: "gemini",
      model: "gemini-embedding-001",
      remote: {
        apiKey: "YOUR_GEMINI_API_KEY"
      }
    }
  }
}
```

ノート：

* `remote.baseUrl`はオプション（デフォルトはGemini APIのベースURL）。
* `remote.headers`を使用して、必要に応じて追加のヘッダーを追加できます。
* デフォルトモデル：`gemini-embedding-001`。

**カスタムOpenAI互換エンドポイント**（OpenRouter、vLLM、またはプロキシ）を使用する場合は、OpenAIプロバイダーとともに`remote`構成を使用できます：

```json5
agents: {
  defaults: {
    memorySearch: {
      provider: "openai",
      model: "text-embedding-3-small",
      remote: {
        baseUrl: "https://api.example.com/v1/",
        apiKey: "YOUR_OPENAI_COMPAT_API_KEY",
        headers: { "X-Custom-Header": "value" }
      }
    }
  }
}
```

APIキーを設定したくない場合は、`memorySearch.provider = "local"`を使用するか、`memorySearch.fallback = "none"`を設定します。

フォールバック：

* `memorySearch.fallback`は`openai`、`gemini`、`voyage`、`mistral`、`local`、または`none`に設定できます。
* フォールバックプロバイダーは、プライマリ埋め込みプロバイダーが失敗した場合にのみ使用されます。

バッチインデックス（OpenAI + Gemini + Voyage）：

* デフォルトで無効。大規模コーパスのインデックス作成を有効にするには、`agents.defaults.memorySearch.remote.batch.enabled = true`を設定します（OpenAI、Gemini、およびVoyage）。
* デフォルトの動作はバッチの完了を待ちます；必要に応じて`remote.batch.wait`、`remote.batch.pollIntervalMs`、および`remote.batch.timeoutMinutes`を調整します。
* `remote.batch.concurrency`を設定して、並行して提出するバッチジョブの数を制御します（デフォルト：2）。
* バッチモードは、`memorySearch.provider = "openai"`または`"gemini"`のときに適用され、対応するAPIキーを使用します。
* Geminiバッチジョブは非同期埋め込みバッチエンドポイントを使用し、Gemini Batch APIの利用可能性が必要です。

OpenAIバッチが速くて安価な理由：

* 大規模なバックフィルの場合、OpenAIは通常、サポートされている中で最も速いオプションです。なぜなら、単一のバッチジョブで多くの埋め込みリクエストを提出し、OpenAIに非同期で処理させることができるからです。
* OpenAIはバッチAPIワークロードに対して割引価格を提供しているため、大規模なインデックス作成は通常、同じリクエストを同期的に送信するよりも安価です。
* 詳細についてはOpenAIバッチAPIのドキュメントと価格を参照してください：
  * [https://platform.openai.com/docs/api-reference/batch](https://platform.openai.com/docs/api-reference/batch)
  * [https://platform.openai.com/pricing](https://platform.openai.com/pricing)

設定例：

```json5
agents: {
  defaults: {
    memorySearch: {
      provider: "openai",
      model: "text-embedding-3-small",
      fallback: "openai",
      remote: {
        batch: { enabled: true, concurrency: 2 }
      },
      sync: { watch: true }
    }
  }
}
```

ツール：

* `memory_search` — ファイル + 行範囲を持つスニペットを返します。
* `memory_get` — パスによってメモリファイルの内容を読み取ります。

ローカルモード：

* `agents.defaults.memorySearch.provider = "local"`を設定します。
* `agents.defaults.memorySearch.local.modelPath`を提供します（GGUFまたは`hf:` URI）。
* オプション：`agents.defaults.memorySearch.fallback = "none"`を設定してリモートフォールバックを回避します。

### メモリツールの動作

* `memory_search`は、`MEMORY.md` + `memory/**/*.md`からMarkdownチャンク（約400トークンのターゲット、80トークンのオーバーラップ）を意味的に検索します。スニペットテキスト（約700文字に制限）、ファイルパス、行範囲、スコア、プロバイダー/モデル、およびローカルからリモート埋め込みにフォールバックしたかどうかを返します。完全なファイルペイロードは返されません。
* `memory_get`は特定のメモリMarkdownファイル（ワークスペース相対）を読み取り、オプションで開始行からN行を読み取ります。`MEMORY.md` / `memory/`の外のパスは拒否されます。
* 両方のツールは、`memorySearch.enabled`がエージェントに対して真で解決される場合にのみ有効です。

### インデックスされるもの（およびそのタイミング）

* ファイルタイプ：Markdownのみ（`MEMORY.md`、`memory/**/*.md`）。
* インデックスストレージ：エージェントごとのSQLiteが`~/.openclaw/memory/<agentId>.sqlite`に保存されます（`agents.defaults.memorySearch.store.path`で構成可能、`{agentId}`トークンをサポート）。
* 新鮮さ：`MEMORY.md` + `memory/`のウォッチャーがインデックスを汚れた状態にマークします（デバウンス1.5秒）。同期はセッション開始時、検索時、または間隔でスケジュールされ、非同期で実行されます。セッションのトランスクリプトは、バックグラウンド同期をトリガーするためにデルタしきい値を使用します。
* 再インデックスのトリガー：インデックスは埋め込み**プロバイダー/モデル + エンドポイントフィンガープリント + チャンクパラメータ**を保存します。これらのいずれかが変更されると、OpenClawは自動的にリセットし、インデックス全体を再インデックスします。

### ハイブリッド検索（BM25 + ベクトル）

有効にすると、OpenClawは次のものを組み合わせます：

* **ベクトル類似性**（意味的マッチ、言葉が異なる場合があります）
* **BM25キーワード関連性**（ID、環境変数、コードシンボルなどの正確なトークン）

プラットフォームでフルテキスト検索が利用できない場合、OpenClawはベクトルのみの検索にフォールバックします。

#### なぜハイブリッド？

ベクトル検索は「これは同じ意味です」に優れています：

* 「Mac Studioゲートウェイホスト」対「ゲートウェイを実行しているマシン」
* 「デバウンスファイルの更新」対「すべての書き込みでインデックス作成を避ける」

しかし、正確で高信号のトークンには弱い場合があります：

* ID（`a828e60`、`b3b9895a…`）
* コードシンボル（`memorySearch.query.hybrid`）
* エラーストリング（"sqlite-vec unavailable"）

BM25（フルテキスト）はその逆：正確なトークンに強く、パラフレーズには弱いです。ハイブリッド検索は実用的な中間地帯です：**両方のリトリーバルシグナルを使用**して、「自然言語」クエリと「干し草の中の針」クエリの両方に対して良い結果を得ることができます。

#### 結果をマージする方法（現在の設計）

実装スケッチ：

1. 両方の側から候補プールを取得します：

* **ベクトル**：コサイン類似性による上位`maxResults * candidateMultiplier`。
* **BM25**：FTS5 BM25ランクによる上位`maxResults * candidateMultiplier`（低いほど良い）。

2. BM25ランクを0..1のスコアに変換します：

* `textScore = 1 / (1 + max(0, bm25Rank))`

3. チャンクIDで候補をユニオンし、重み付けスコアを計算します：

* `finalScore = vectorWeight * vectorScore + textWeight * textScore`

ノート：

* `vectorWeight` + `textWeight`は設定解決時に1.0に正規化されるため、重みはパーセンテージとして機能します。
* 埋め込みが利用できない場合（またはプロバイダーがゼロベクトルを返す場合）、BM25を実行し、キーワードマッチを返します。
* FTS5を作成できない場合、ベクトルのみの検索を維持します（ハードフェイルなし）。

これは「IR理論的に完璧」ではありませんが、シンプルで迅速であり、実際のメモのリコール/精度を改善する傾向があります。後でより洗練されたものを求める場合、一般的な次のステップは、結果の混合前に逆順位融合（RRF）やスコア正規化（min/maxまたはzスコア）です。

#### ポストプロセッシングパイプライン

ベクトルとキーワードスコアをマージした後、2つのオプションのポストプロセッシングステージが結果リストを洗練し、エージェントに到達する前に精製します：

```
ベクトル + キーワード → 重み付けマージ → 時間的減衰 → ソート → MMR → 上位K結果
```

両方のステージは**デフォルトでオフ**であり、独立して有効にできます。

#### MMR再ランキング（多様性）

ハイブリッド検索が結果を返すと、複数のチャンクが類似または重複するコンテンツを含む場合があります。例えば、「ホームネットワークの設定」を検索すると、同じルーター設定をすべて言及する異なる日次メモからほぼ同一のスニペットが5つ返されるかもしれません。

**MMR（最大限の限界関連性）**は、関連性と多様性のバランスを取るために結果を再ランキングし、上位の結果が同じ情報を繰り返すのではなく、クエリの異なる側面をカバーすることを保証します。

動作方法：

1. 結果は元の関連性（ベクトル + BM25重み付けスコア）によってスコア付けされます。
2. MMRは、選択された結果に対する最大の類似性を最小化することを最大化する結果を反復的に選択します：`λ × relevance − (1−λ) × max_similarity_to_selected`。
3. 結果間の類似性は、トークン化されたコンテンツに対するジャッカードテキスト類似性を使用して測定されます。

`lambda`パラメータはトレードオフを制御します：

* `lambda = 1.0` → 純粋な関連性（多様性ペナルティなし）
* `lambda = 0.0` → 最大の多様性（関連性を無視）
* デフォルト：`0.7`（バランスの取れた、わずかな関連性バイアス）

**例 — クエリ：「ホームネットワークの設定」**

これらのメモリファイルが与えられた場合：

```
memory/2026-02-10.md  → "Omadaルーターを設定し、IoTデバイス用にVLAN 10を設定"
memory/2026-02-08.md  → "Omadaルーターを設定し、IoTをVLAN 10に移動"
memory/2026-02-05.md  → "192.168.10.2でAdGuard DNSを設定"
memory/network.md     → "ルーター：Omada ER605、AdGuard：192.168.10.2、VLAN 10：IoT"
```

MMRなし — 上位3結果：

```
1. memory/2026-02-10.md  (スコア: 0.92)  ← ルーター + VLAN
2. memory/2026-02-08.md  (スコア: 0.89)  ← ルーター + VLAN（ほぼ重複！）
3. memory/network.md     (スコア: 0.85)  ← 参照ドキュメント
```

MMRあり（λ=0.7） — 上位3結果：

```
1. memory/2026-02-10.md  (スコア: 0.92)  ← ルーター + VLAN
2. memory/network.md     (スコア: 0.85)  ← 参照ドキュメント（多様性あり！）
3. memory/2026-02-05.md  (スコア: 0.78)  ← AdGuard DNS（多様性あり！）
```

2月8日の近似重複が除外され、エージェントは3つの異なる情報を得ます。

**有効にするタイミング**：`memory_search`が冗長または近似重複のスニペットを返すことに気付いた場合、特に日次メモで同じ情報を繰り返すことが多い場合。

#### 時間的減衰（最近のブースト）

日次メモを持つエージェントは、時間の経過とともに数百の日時付きファイルを蓄積します。減衰がない場合、6か月前のうまく書かれたメモが、同じトピックに関する昨日の更新よりも上位にランク付けされる可能性があります。

**時間的減衰**は、各結果の年齢に基づいてスコアに指数的な乗数を適用し、最近のメモが自然に高くランク付けされる一方で、古いメモは減衰します：

```
decayedScore = score × e^(-λ × ageInDays)
```

ここで、`λ = ln(2) / halfLifeDays`です。

デフォルトの半減期が30日の場合：

* 今日のメモ：**元のスコアの100%**
* 7日前：**約84%**
* 30日前：**50%**
* 90日前：**12.5%**
* 180日前：**約1.6%**

**エバーグリーンファイルは決して減衰しません：**

* `MEMORY.md`（ルートメモリファイル）
* `memory/`内の非日時ファイル（例：`memory/projects.md`、`memory/network.md`）
* これらは常に通常のランクを持つ持続的な参照情報を含みます。

**日時付きの日次ファイル**（`memory/YYYY-MM-DD.md`）は、ファイル名から抽出された日付を使用します。他のソース（例：セッションのトランスクリプト）は、ファイルの変更時刻（`mtime`）にフォールバックします。

**例 — クエリ：「ロッドの勤務スケジュールは？」**

これらのメモリファイルが与えられた場合（今日が2月10日）：

```
memory/2025-09-15.md  → "ロッドは月曜日から金曜日まで働き、10時にスタンドアップ、14時にペアリング"  （148日前）
memory/2026-02-10.md  → "ロッドは14:15にスタンドアップ、14:45にゼブと1:1"    （今日）
memory/2026-02-03.md  → "ロッドは新しいチームを始め、スタンドアップが14:15に移動"        （7日前）
```

減衰なし：

```
1. memory/2025-09-15.md  (スコア: 0.91)  ← 最良の意味的マッチですが、古い！
2. memory/2026-02-10.md  (スコア: 0.82)
3. memory/2026-02-03.md  (スコア: 0.80)
```

減衰あり（半減期=30）：

```
1. memory/2026-02-10.md  (スコア: 0.82 × 1.00 = 0.82)  ← 今日、減衰なし
2. memory/2026-02-03.md  (スコア: 0.80 × 0.85 = 0.68)  ← 7日前、軽い減衰
3. memory/2025-09-15.md  (スコア: 0.91 × 0.03 = 0.03)  ← 148日前、ほぼ消失
```

古い9月のメモは、最良の生の意味的マッチを持っているにもかかわらず、底に落ちます。

**有効にするタイミング**：エージェントが数か月の日時付きメモを持っていて、古い、古くなった情報が最近のコンテキストよりも上位にランク付けされることがある場合。30日の半減期は、日次メモが多いワークフローに適しています；古いメモを頻繁に参照する場合は、増やしてください（例：90日）。

#### 設定

両方の機能は`memorySearch.query.hybrid`の下で構成されます：

```json5
agents: {
  defaults: {
    memorySearch: {
      query: {
        hybrid: {
          enabled: true,
          vectorWeight: 0.7,
          textWeight: 0.3,
          candidateMultiplier: 4,
          // 多様性：冗長な結果を減少させる
          mmr: {
            enabled: true,    // デフォルト：false
            lambda: 0.7       // 0 = 最大の多様性、1 = 最大の関連性
          },
          // 最近性：新しいメモをブースト
          temporalDecay: {
            enabled: true,    // デフォルト：false
            halfLifeDays: 30  // スコアが30日ごとに半分になる
          }
        }
      }
    }
  }
}
```

いずれかの機能を独立して有効にできます：

* **MMRのみ** — 多くの類似メモがあるが、年齢は問題ではない場合に便利です。
* **時間的減衰のみ** — 最近性が重要だが、結果がすでに多様な場合に便利です。
* **両方** — 大規模で長期的な日次メモ履歴を持つエージェントに推奨されます。

### 埋め込みキャッシュ

OpenClawは、**チャンク埋め込み**をSQLiteにキャッシュできるため、再インデックス作成や頻繁な更新（特にセッションのトランスクリプト）で変更されていないテキストを再埋め込みすることがありません。

構成：

```json5
agents: {
  defaults: {
    memorySearch: {
      cache: {
        enabled: true,
        maxEntries: 50000
      }
    }
  }
}
```

### セッションメモリ検索（実験的）

オプションで**セッションのトランスクリプト**をインデックス化し、`memory_search`を介してそれらを表示できます。これは実験的なフラグの背後にあります。

```json5
agents: {
  defaults: {
    memorySearch: {
      experimental: { sessionMemory: true },
      sources: ["memory", "sessions"]
    }
  }
}
```

ノート：

* セッションインデックスは**オプトイン**（デフォルトでオフ）です。
* セッションの更新はデバウンスされ、デルタしきい値を超えると**非同期でインデックス化**されます（最善の努力）。
* `memory_search`はインデックス化中にブロックされることはありません；結果はバックグラウンド同期が終了するまでわずかに古くなる可能性があります。
* 結果にはスニペットのみが含まれます；`memory_get`はメモリファイルに制限されます。
* セッションインデックスはエージェントごとに隔離されます（そのエージェントのセッションログのみがインデックス化されます）。
* セッションログはディスクに保存されます（`~/.openclaw/agents/<agentId>/sessions/*.jsonl`）。ファイルシステムへのアクセス権を持つプロセス/ユーザーはそれらを読み取ることができるため、ディスクアクセスを信頼の境界として扱います。厳格な隔離を求める場合は、エージェントを別のOSユーザーまたはホストの下で実行してください。

デルタしきい値（デフォルト値を示す）：

```json5
agents: {
  defaults: {
    memorySearch: {
      sync: {
        sessions: {
          deltaBytes: 100000,   // 約100 KB
          deltaMessages: 50     // JSONL行
        }
      }
    }
  }
}
```

### SQLiteベクトル加速（sqlite-vec）

sqlite-vec拡張が利用可能な場合、OpenClawは埋め込みをSQLiteの仮想テーブル（`vec0`）に保存し、データベース内でベクトル距離クエリを実行します。これにより、すべての埋め込みをJSに読み込むことなく、検索が迅速になります。

構成（オプション）：

```json5
agents: {
  defaults: {
    memorySearch: {
      store: {
        vector: {
          enabled: true,
          extensionPath: "/path/to/sqlite-vec"
        }
      }
    }
  }
}
```

ノート：

* `enabled`はデフォルトでtrueです；無効にすると、検索は保存された埋め込みに対してプロセス内のコサイン類似性にフォールバックします。
* sqlite-vec拡張が欠落しているか、読み込むことに失敗した場合、OpenClawはエラーをログに記録し、JSフォールバック（ベクトルテーブルなし）を続行します。
* `extensionPath`はバンドルされたsqlite-vecパスをオーバーライドします（カスタムビルドや非標準のインストール場所に便利です）。

### ローカル埋め込みの自動ダウンロード

* デフォルトのローカル埋め込みモデル：`hf:ggml-org/embeddinggemma-300m-qat-q8_0-GGUF/embeddinggemma-300m-qat-Q8_0.gguf`（約0.6 GB）。
* `memorySearch.provider = "local"`の場合、`node-llama-cpp`は`modelPath`を解決します；GGUFが欠落している場合、**自動的に**キャッシュにダウンロードします（または設定されている場合は`local.modelCacheDir`）、その後読み込みます。ダウンロードは再試行時に再開されます。
* ネイティブビルドの要件：`pnpm approve-builds`を実行し、`node-llama-cpp`を選択し、次に`pnpm rebuild node-llama-cpp`を実行します。
* フォールバック：ローカルセットアップが失敗し、`memorySearch.fallback = "openai"`の場合、リモート埋め込み（`openai/text-embedding-3-small`、オーバーライドされていない限り）に自動的に切り替え、その理由を記録します。

### カスタムOpenAI互換エンドポイントの例

```json5
agents: {
  defaults: {
    memorySearch: {
      provider: "openai",
      model: "text-embedding-3-small",
      remote: {
        baseUrl: "https://api.example.com/v1/",
        apiKey: "YOUR_REMOTE_API_KEY",
        headers: {
          "X-Organization": "org-id",
          "X-Project": "project-id"
        }
      }
    }
  }
}
```

ノート：

* `remote.*`は`models.providers.openai.*`よりも優先されます。
* `remote.headers`はOpenAIのヘッダーとマージされ、リモートがキーの競合に勝ちます。`remote.headers`を省略すると、OpenAIのデフォルトが使用されます。
