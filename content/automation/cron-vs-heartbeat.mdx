# Cron vs Heartbeat
# Cron vs Heartbeat: どちらをいつ使うべきか
ハートビートとクロンジョブは、どちらもタスクをスケジュールに従って実行することができます。このガイドは、あなたのユースケースに最適なメカニズムを選ぶ手助けをします。
## 迅速な判断ガイド
| ユースケース | 推奨 | 理由 |
| ------------------------------------ | ------------------- | ---------------------------------------- |
| 30分ごとに受信トレイを確認 | ハートビート | 他のチェックとバッチ処理、コンテキストに応じた対応 |
| 毎日午前9時にレポートを送信 | クロン（孤立） | 正確なタイミングが必要 |
| カレンダーを監視して今後のイベントを確認 | ハートビート | 定期的な認識に自然に適している |
| 毎週の詳細分析を実行 | クロン（孤立） | 独立したタスクで、異なるモデルを使用できる |
| 20分後にリマインド | クロン（メイン、`--at`） | 正確なタイミングでのワンショット |
| プロジェクトの健康状態をバックグラウンドでチェック | ハートビート | 既存のサイクルに乗っかる |
## ハートビート: 定期的な認識
ハートビートは、\*\*メインセッション\*\*で定期的に実行されます（デフォルト: 30分）。これはエージェントが物事を確認し、重要なことを浮き彫りにするために設計されています。
### ハートビートを使用するタイミング
\* \*\*複数の定期チェック\*\*: 受信トレイ、カレンダー、天気、通知、プロジェクトのステータスを確認する5つの別々のクロンジョブの代わりに、1つのハートビートでこれらすべてをバッチ処理できます。
\* \*\*コンテキストに応じた意思決定\*\*: エージェントは完全なメインセッションのコンテキストを持っているため、何が緊急か、何が待てるかを賢く判断できます。
\* \*\*会話の連続性\*\*: ハートビートの実行は同じセッションで行われるため、エージェントは最近の会話を記憶し、自然にフォローアップできます。
\* \*\*低オーバーヘッドの監視\*\*: 1つのハートビートが多くの小さなポーリングタスクを置き換えます。
### ハートビートの利点
\* \*\*複数のチェックをバッチ処理\*\*: 1回のエージェントのターンで受信トレイ、カレンダー、通知を一緒に確認できます。
\* \*\*APIコールを削減\*\*: 1つのハートビートは5つの孤立したクロンジョブよりも安価です。
\* \*\*コンテキストに応じた対応\*\*: エージェントはあなたが何に取り組んでいるかを知っており、それに応じて優先順位を付けることができます。
\* \*\*スマートサプレッション\*\*: 何も注意を要しない場合、エージェントは `HEARTBEAT_OK` と応答し、メッセージは配信されません。
\* \*\*自然なタイミング\*\*: キューの負荷に基づいてわずかにずれますが、ほとんどの監視には問題ありません。
### ハートビートの例: HEARTBEAT.md チェックリスト
```md
# ハートビートチェックリスト
- 緊急メッセージのためにメールを確認
- 次の2時間のイベントのためにカレンダーを確認
- バックグラウンドタスクが終了した場合、結果を要約
- 8時間以上アイドルの場合、簡単なチェックインを送信
```
エージェントは各ハートビートでこれを読み取り、すべての項目を1回のターンで処理します。
### ハートビートの設定
```json5
{
agents: {
defaults: {
heartbeat: {
every: "30m", // インターバル
target: "last", // 明示的なアラート配信ターゲット（デフォルトは "none"）
activeHours: { start: "08:00", end: "22:00" }, // オプション
},
},
},
}
```
完全な設定については [Heartbeat](/gateway/heartbeat) を参照してください。
## クロン: 正確なスケジューリング
クロンジョブは正確な時間に実行され、メインコンテキストに影響を与えずに孤立したセッションで実行できます。
定期的な時間の先頭でのスケジュールは、決定論的な
ジョブごとのオフセットによって自動的に5分のウィンドウ内に分散されます。
### クロンを使用するタイミング
\* \*\*正確なタイミングが必要\*\*: "毎週月曜日の午前9時にこれを送信"（"午前9時頃"ではなく）。
\* \*\*独立したタスク\*\*: 会話のコンテキストを必要としないタスク。
\* \*\*異なるモデル/思考\*\*: より強力なモデルが必要な重い分析。
\* \*\*ワンショットリマインダー\*\*: `--at` を使用した "20分後にリマインド"。
\* \*\*騒がしい/頻繁なタスク\*\*: メインセッションの履歴を混乱させるタスク。
\* \*\*外部トリガー\*\*: エージェントが他にアクティブであるかどうかに関係なく実行されるべきタスク。
### クロンの利点
\* \*\*正確なタイミング\*\*: タイムゾーンサポートを持つ5フィールドまたは6フィールド（秒）クロン式。
\* \*\*組み込みの負荷分散\*\*: 定期的な時間の先頭でのスケジュールはデフォルトで最大5分間ずらされます。
\* \*\*ジョブごとの制御\*\*: `--stagger` でずらしをオーバーライドするか、`--exact` で正確なタイミングを強制します。
\* \*\*セッションの隔離\*\*: `cron:` で実行され、メインの履歴を汚染しません。
\* \*\*モデルのオーバーライド\*\*: ジョブごとに安価またはより強力なモデルを使用できます。
\* \*\*配信制御\*\*: 孤立したジョブはデフォルトで `announce`（要約）; 必要に応じて `none` を選択します。
\* \*\*即時配信\*\*: アナウンスモードはハートビートを待たずに直接投稿します。
\* \*\*エージェントのコンテキストが不要\*\*: メインセッションがアイドルまたは圧縮されていても実行されます。
\* \*\*ワンショットサポート\*\*: 正確な将来のタイムスタンプのための `--at`。
### クロンの例: 毎日の朝のブリーフィング
```bash
openclaw cron add \
--name "Morning briefing" \
--cron "0 7 \* \* \*" \
--tz "America/New\_York" \
--session isolated \
--message "今日のブリーフィングを生成: 天気、カレンダー、トップメール、ニュースの要約。" \
--model opus \
--announce \
--channel whatsapp \
--to "+15551234567"
```
これはニューヨーク時間の午前7時に正確に実行され、品質のためにOpusを使用し、WhatsAppに直接要約をアナウンスします。
### クロンの例: ワンショットリマインダー
```bash
openclaw cron add \
--name "Meeting reminder" \
--at "20m" \
--session main \
--system-event "リマインダー: スタンドアップミーティングは10分後に始まります。" \
--wake now \
--delete-after-run
```
完全なCLIリファレンスについては [Cron jobs](/automation/cron-jobs) を参照してください。
## 決定フローチャート
```
タスクは正確な時間に実行する必要がありますか？
YES -> クロンを使用
NO -> 続行...
タスクはメインセッションからの隔離が必要ですか？
YES -> クロン（孤立）を使用
NO -> 続行...
このタスクは他の定期チェックとバッチ処理できますか？
YES -> ハートビートを使用（HEARTBEAT.mdに追加）
NO -> クロンを使用
これはワンショットリマインダーですか？
YES -> `--at` 付きのクロンを使用
NO -> 続行...
異なるモデルや思考レベルが必要ですか？
YES -> `--model/--thinking` 付きのクロン（孤立）を使用
NO -> ハートビートを使用
```
## 両方の組み合わせ
最も効率的なセットアップは\*\*両方\*\*を使用します：
1. \*\*ハートビート\*\*がルーチン監視（受信トレイ、カレンダー、通知）を30分ごとに1回のバッチターンで処理します。
2. \*\*クロン\*\*が正確なスケジュール（毎日のレポート、毎週のレビュー）とワンショットリマインダーを処理します。
### 例: 効率的な自動化セットアップ
\*\*HEARTBEAT.md\*\*（30分ごとにチェック）:
```md
# ハートビートチェックリスト
- 緊急メールのために受信トレイをスキャン
- 次の2時間のイベントのためにカレンダーを確認
- 保留中のタスクを確認
- 8時間以上静かな場合は軽いチェックイン
```
\*\*クロンジョブ\*\*（正確なタイミング）:
```bash
# 毎朝7時のブリーフィング
openclaw cron add --name "Morning brief" --cron "0 7 \* \* \*" --session isolated --message "..." --announce
# 毎週月曜日の午前9時のプロジェクトレビュー
openclaw cron add --name "Weekly review" --cron "0 9 \* \* 1" --session isolated --message "..." --model opus
# ワンショットリマインダー
openclaw cron add --name "Call back" --at "2h" --session main --system-event "クライアントに電話をかける" --wake now
```
## Lobster: 承認を伴う決定論的ワークフロー
Lobsterは、決定論的な実行と明示的な承認を必要とする\*\*マルチステップツールパイプライン\*\*のためのワークフローランタイムです。
タスクが単一のエージェントターンを超える場合、再開可能なワークフローを人間のチェックポイントで持ちたいときに使用します。
### Lobsterが適している時
\* \*\*マルチステップ自動化\*\*: 固定されたツール呼び出しのパイプラインが必要で、一度きりのプロンプトではありません。
\* \*\*承認ゲート\*\*: 副作用は承認されるまで一時停止し、その後再開されるべきです。
\* \*\*再開可能な実行\*\*: 一時停止したワークフローを以前のステップを再実行せずに続行できます。
### ハートビートとクロンとの組み合わせ
\* \*\*ハートビート/クロン\*\*は、実行が\*いつ\*行われるかを決定します。
\* \*\*Lobster\*\*は、実行が開始された後に\*どのステップ\*が行われるかを定義します。
スケジュールされたワークフローの場合、クロンまたはハートビートを使用してエージェントターンをトリガーし、Lobsterを呼び出します。
アドホックワークフローの場合は、Lobsterを直接呼び出します。
### 操作ノート（コードから）
\* Lobsterはツールモードで\*\*ローカルサブプロセス\*\*（`lobster` CLI）として実行され、\*\*JSONエンベロープ\*\*を返します。
\* ツールが `needs_approval` を返す場合、`resumeToken` と `approve` フラグを使用して再開します。
\* ツールは\*\*オプションのプラグイン\*\*です; `tools.alsoAllow: ["lobster"]`を介して追加的に有効にすることを推奨します。
\* Lobsterは `PATH` に `lobster` CLI が利用可能であることを期待します。
使用法と例については [Lobster](/tools/lobster) を参照してください。
## メインセッション vs 孤立セッション
ハートビートとクロンはどちらもメインセッションと相互作用できますが、異なる方法で行います：
| | ハートビート | クロン（メイン） | クロン（孤立） |
| ------- | ------------------------------- | ------------------------ | -------------------------- |
| セッション | メイン | メイン（システムイベント経由） | `cron:` |
| 履歴 | 共有 | 共有 | 各実行ごとに新鮮 |
| コンテキスト | 完全 | 完全 | なし（クリーンスタート） |
| モデル | メインセッションモデル | メインセッションモデル | オーバーライド可能 |
| 出力 | `HEARTBEAT_OK` でない場合は配信 | ハートビートプロンプト + イベント | 要約をアナウンス（デフォルト） |
### メインセッションクロンを使用する時
`--session main` を `--system-event` と共に使用する場合：
\* リマインダー/イベントがメインセッションコンテキストに表示されること
\* エージェントが次のハートビート中に完全なコンテキストで処理すること
\* 別の孤立した実行が不要であること
```bash
openclaw cron add \
--name "Check project" \
--every "4h" \
--session main \
--system-event "プロジェクトの健康チェックの時間です" \
--wake now
```
### 孤立クロンを使用する時
`--session isolated` を使用する場合：
\* 以前のコンテキストがないクリーンスレート
\* 異なるモデルまたは思考設定
\* 要約を直接チャンネルにアナウンス
\* メインセッションを混乱させない履歴
```bash
openclaw cron add \
--name "Deep analysis" \
--cron "0 6 \* \* 0" \
--session isolated \
--message "毎週のコードベース分析..." \
--model opus \
--thinking high \
--announce
```
## コストの考慮事項
| メカニズム | コストプロファイル |
| --------------- | ------------------------------------------------------- |
| ハートビート | N分ごとに1ターン; HEARTBEAT.mdのサイズに応じてスケール |
| クロン（メイン） | 次のハートビートにイベントを追加（孤立したターンなし） |
| クロン（孤立） | ジョブごとにフルエージェントターン; より安価なモデルを使用可能 |
\*\*ヒント\*\*:
\* トークンオーバーヘッドを最小限に抑えるために `HEARTBEAT.md` を小さく保つ。
\* 複数のクロンジョブの代わりに、ハートビートに似たチェックをバッチ処理する。
\* 内部処理のみを希望する場合はハートビートで `target: "none"` を使用。
\* ルーチンタスクには安価なモデルを使用した孤立クロンを使用。
