# Cron ジョブ
# Cron jobs (Gateway scheduler)
> **Cron vs Heartbeat?** 詳細については [Cron vs Heartbeat](/automation/cron-vs-heartbeat) を参照してください。
CronはGatewayに組み込まれたスケジューラです。ジョブを永続化し、適切な時間にエージェントを起動し、オプションでチャットに出力を返すことができます。
「毎朝これを実行する」や「20分後にエージェントを起こす」といった場合、cronがそのメカニズムです。
トラブルシューティング: [/automation/troubleshooting](/automation/troubleshooting)
## TL;DR
* Cronは**Gateway内**で実行されます（モデル内ではありません）。
* ジョブは`~/.openclaw/cron/`に永続化されるため、再起動してもスケジュールが失われません。
* 実行スタイルは2つあります：
  * **メインセッション**：システムイベントをキューに入れ、次のハートビートで実行します。
  * **アイソレート**：`cron:`内で専用のエージェントターンを実行し、配信（デフォルトでは発表またはなし）します。
* ウェイクアップはファーストクラス：ジョブは「今起こす」と「次のハートビート」のリクエストができます。
* Webhookの投稿はジョブごとに`delivery.mode = "webhook"` + `delivery.to = ""`で行われます。
* `cron.webhook`が設定されている場合、`notify: true`の保存されたジョブにはレガシーフォールバックが残ります。これらのジョブをWebhook配信モードに移行してください。
## クイックスタート（実行可能）
一度限りのリマインダーを作成し、存在を確認してすぐに実行します：
```bash
openclaw cron add \
--name "Reminder" \
--at "2026-02-01T16:00:00Z" \
--session main \
--system-event "Reminder: check the cron docs draft" \
--wake now \
--delete-after-run
openclaw cron list
openclaw cron run 
openclaw cron runs --id 
```
配信付きの定期的なアイソレートジョブをスケジュールします：
```bash
openclaw cron add \
--name "Morning brief" \
--cron "0 7 * * *" \
--tz "America/Los_Angeles" \
--session isolated \
--message "Summarize overnight updates." \
--announce \
--channel slack \
--to "channel:C1234567890"
```
## ツールコールの同等物（Gateway cronツール）
標準的なJSONの形状と例については、[ツールコールのためのJSONスキーマ](/automation/cron-jobs#json-schema-for-tool-calls)を参照してください。
## Cronジョブの保存場所
CronジョブはデフォルトでGatewayホストの`~/.openclaw/cron/jobs.json`に永続化されます。
Gatewayはファイルをメモリに読み込み、変更時に書き戻しますので、手動編集はGatewayが停止しているときのみ安全です。変更には`openclaw cron add/edit`またはcronツールコールAPIを使用することをお勧めします。
## 初心者向けの概要
Cronジョブを次のように考えてください：**いつ**実行するか + **何を**するか。
1. **スケジュールを選択**
   * 一度限りのリマインダー → `schedule.kind = "at"` (CLI: `--at`)
   * 繰り返しジョブ → `schedule.kind = "every"` または `schedule.kind = "cron"`
   * ISOタイムスタンプにタイムゾーンが省略されている場合、**UTC**として扱われます。
2. **実行場所を選択**
   * `sessionTarget: "main"` → 次のハートビートでメインコンテキストで実行します。
   * `sessionTarget: "isolated"` → `cron:`内で専用のエージェントターンを実行します。
3. **ペイロードを選択**
   * メインセッション → `payload.kind = "systemEvent"`
   * アイソレートセッション → `payload.kind = "agentTurn"`
オプション：一度限りのジョブ（`schedule.kind = "at"`）は成功後にデフォルトで削除されます。保持するには`deleteAfterRun: false`を設定します（成功後に無効になります）。
## 概念
### ジョブ
Cronジョブは次の情報を持つ保存されたレコードです：
* **スケジュール**（いつ実行するか）
* **ペイロード**（何をするか）
* オプションの**配信モード**（`announce`、`webhook`、または`none`）
* オプションの**エージェントバインディング**（`agentId`）：特定のエージェントの下でジョブを実行します。欠落または不明な場合、Gatewayはデフォルトのエージェントにフォールバックします。
ジョブは安定した`jobId`によって識別されます（CLI/Gateway APIで使用されます）。
エージェントツールコールでは、`jobId`が標準です。レガシーの`id`は互換性のために受け入れられます。
一度限りのジョブは成功後に自動的に削除されます。保持するには`deleteAfterRun: false`を設定します。
### スケジュール
Cronは3つのスケジュールの種類をサポートしています：
* `at`：一度限りのタイムスタンプ（`schedule.at`経由、ISO 8601）
* `every`：固定間隔（ms）
* `cron`：5フィールドのcron式（または秒を含む6フィールド）で、オプションのIANAタイムゾーンを指定できます。
Cron式は`croner`を使用します。タイムゾーンが省略されている場合、Gatewayホストのローカルタイムゾーンが使用されます。
多くのGatewayでの時間の最初の負荷スパイクを減らすために、OpenClawは定期的な時間の最初の表現（例えば`0 * * * *`、`0 */2 * * *`）に対して最大5分の決定論的なジョブごとのスタッガーウィンドウを適用します。固定時間の表現（例えば`0 7 * * *`）は正確なままです。
任意のcronスケジュールに対して、`schedule.staggerMs`を使用して明示的なスタッガーウィンドウを設定できます（`0`は正確なタイミングを維持します）。CLIショートカット：
* `--stagger 30s`（または`1m`、`5m`）で明示的なスタッガーウィンドウを設定します。
* `--exact`で`staggerMs = 0`を強制します。
### メインとアイソレートの実行
#### メインセッションジョブ（システムイベント）
メインジョブはシステムイベントをキューに入れ、オプションでハートビートランナーを起動します。
これらは`payload.kind = "systemEvent"`を使用する必要があります。
* `wakeMode: "now"`（デフォルト）：イベントが即座にハートビートの実行をトリガーします。
* `wakeMode: "next-heartbeat"`：イベントが次の予定されたハートビートを待ちます。
これは通常のハートビートプロンプトとメインセッションコンテキストを希望する場合に最適です。
詳細は [Heartbeat](/gateway/heartbeat) を参照してください。
#### アイソレートジョブ（専用のcronセッション）
アイソレートジョブはセッション`cron:`内で専用のエージェントターンを実行します。
主な挙動：
* プロンプトはトレース可能性のために`[cron:]`で接頭辞が付けられます。
* 各実行は**新しいセッションID**で開始されます（以前の会話の持ち越しはありません）。
* デフォルトの挙動：`delivery`が省略されている場合、アイソレートジョブは要約を発表します（`delivery.mode = "announce"`）。
* `delivery.mode`は何が起こるかを選択します：
  * `announce`：ターゲットチャネルに要約を配信し、メインセッションに簡単な要約を投稿します。
  * `webhook`：完了したイベントペイロードを`delivery.to`にPOSTします。完了したイベントに要約が含まれている場合。
  * `none`：内部専用（配信なし、メインセッションの要約なし）。
* `wakeMode`はメインセッションの要約が投稿されるタイミングを制御します：
  * `now`：即座のハートビート。
  * `next-heartbeat`：次の予定されたハートビートを待ちます。
アイソレートジョブは、メインチャット履歴をスパムしないようにするための騒がしい、頻繁な、または「バックグラウンド作業」に使用します。
### ペイロードの形状（何が実行されるか）
2つのペイロードの種類がサポートされています：
* `systemEvent`：メインセッション専用、ハートビートプロンプトを通じてルーティングされます。
* `agentTurn`：アイソレートセッション専用、専用のエージェントターンを実行します。
一般的な`agentTurn`フィールド：
* `message`：必須のテキストプロンプト。
* `model` / `thinking`：オプションのオーバーライド（以下を参照）。
* `timeoutSeconds`：オプションのタイムアウトオーバーライド。
配信設定：
* `delivery.mode`：`none` | `announce` | `webhook`。
* `delivery.channel`：`last`または特定のチャネル。
* `delivery.to`：チャネル特有のターゲット（発表）またはWebhook URL（Webhookモード）。
* `delivery.bestEffort`：発表配信が失敗した場合でもジョブが失敗しないようにします。
発表配信は実行のためのメッセージングツールの送信を抑制します。代わりに`delivery.channel`/`delivery.to`を使用してチャットをターゲットにします。`delivery.mode = "none"`の場合、メインセッションに要約は投稿されません。
アイソレートジョブの`delivery`が省略されている場合、OpenClawはデフォルトで`announce`に設定します。
#### 発表配信フロー
`delivery.mode = "announce"`の場合、cronはアウトバウンドチャネルアダプタを介して直接配信します。
メインエージェントはメッセージを作成または転送するために起動されません。
挙動の詳細：
* コンテンツ：配信はアイソレート実行のアウトバウンドペイロード（テキスト/メディア）を通常のチャンク化とチャネルフォーマットで使用します。
* ハートビート専用の応答（実際のコンテンツがない`HEARTBEAT_OK`）は配信されません。
* アイソレート実行がすでに同じターゲットにメッセージを送信している場合、配信は重複を避けるためにスキップされます。
* 配信ターゲットが欠落または無効な場合、`delivery.bestEffort = true`でない限りジョブは失敗します。
* `delivery.mode = "announce"`の場合のみ、メインセッションに短い要約が投稿されます。
* メインセッションの要約は`wakeMode`を尊重します：`now`は即座のハートビートをトリガーし、`next-heartbeat`は次の予定されたハートビートを待ちます。
#### Webhook配信フロー
`delivery.mode = "webhook"`の場合、cronは完了したイベントペイロードを`delivery.to`に投稿します。完了したイベントに要約が含まれている場合。
挙動の詳細：
* エンドポイントは有効なHTTP(S) URLでなければなりません。
* Webhookモードではチャネル配信は試みられません。
* Webhookモードではメインセッションの要約は投稿されません。
* `cron.webhookToken`が設定されている場合、認証ヘッダーは`Authorization: Bearer `です。
* 非推奨のフォールバック：`notify: true`の保存されたレガシージョブは、設定されている場合、`cron.webhook`に投稿されます。警告が表示され、`delivery.mode = "webhook"`に移行できます。
### モデルと思考のオーバーライド
アイソレートジョブ（`agentTurn`）はモデルと思考レベルをオーバーライドできます：
* `model`：プロバイダー/モデル文字列（例：`anthropic/claude-sonnet-4-20250514`）またはエイリアス（例：`opus`）
* `thinking`：思考レベル（`off`、`minimal`、`low`、`medium`、`high`、`xhigh`；GPT-5.2 + Codexモデルのみ）
注意：メインセッションジョブでも`model`を設定できますが、共有メインセッションモデルが変更されます。予期しないコンテキストのシフトを避けるために、アイソレートジョブのみにモデルオーバーライドを推奨します。
解決優先度：
1. ジョペイロードオーバーライド（最優先）
2. フック固有のデフォルト（例：`hooks.gmail.model`）
3. エージェント設定のデフォルト
### 配信（チャネル + ターゲット）
アイソレートジョブは、トップレベルの`delivery`設定を介してチャネルに出力を配信できます：
* `delivery.mode`：`announce`（チャネル配信）、`webhook`（HTTP POST）、または`none`。
* `delivery.channel`：`whatsapp` / `telegram` / `discord` / `slack` / `mattermost`（プラグイン） / `signal` / `imessage` / `last`。
* `delivery.to`：チャネル特有の受取人ターゲット。
`announce`配信はアイソレートジョブ（`sessionTarget: "isolated"`）のみに有効です。
`webhook`配信はメインジョブとアイソレートジョブの両方に有効です。
`delivery.channel`または`delivery.to`が省略されている場合、cronはメインセッションの「最後のルート」（エージェントが最後に応答した場所）にフォールバックできます。
ターゲット形式のリマインダー：
* Slack/Discord/Mattermost（プラグイン）ターゲットは、曖昧さを避けるために明示的な接頭辞（例：`channel:`、`user:`）を使用する必要があります。
* Telegramのトピックは、`:topic:`形式を使用する必要があります（以下を参照）。
#### Telegram配信ターゲット（トピック / フォーラムスレッド）
Telegramは`message_thread_id`を介してフォーラムトピックをサポートしています。cron配信の場合、トピック/スレッドを`to`フィールドにエンコードできます：
* `-1001234567890`（チャットIDのみ）
* `-1001234567890:topic:123`（推奨：明示的なトピックマーカー）
* `-1001234567890:123`（省略形：数値サフィックス）
接頭辞付きターゲット（例えば`telegram:...` / `telegram:group:...`）も受け入れられます：
* `telegram:group:-1001234567890:topic:123`
## ツールコールのためのJSONスキーマ
Gateway `cron.*`ツールを直接呼び出す際にこれらの形状を使用します（エージェントツールコールまたはRPC）。
CLIフラグは`20m`のような人間の期間を受け入れますが、ツールコールは`schedule.at`にISO 8601文字列を、`schedule.everyMs`にミリ秒を使用する必要があります。
### cron.addパラメータ
一度限りのメインセッションジョブ（システムイベント）：
```json
{
"name": "Reminder",
"schedule": { "kind": "at", "at": "2026-02-01T16:00:00Z" },
"sessionTarget": "main",
"wakeMode": "now",
"payload": { "kind": "systemEvent", "text": "Reminder text" },
"deleteAfterRun": true
}
```
定期的なアイソレートジョブで配信あり：
```json
{
"name": "Morning brief",
"schedule": { "kind": "cron", "expr": "0 7 * * *", "tz": "America/Los_Angeles" },
"sessionTarget": "isolated",
"wakeMode": "next-heartbeat",
"payload": {
"kind": "agentTurn",
"message": "Summarize overnight updates."
},
"delivery": {
"mode": "announce",
"channel": "slack",
"to": "channel:C1234567890",
"bestEffort": true
}
}
```
ノート：
* `schedule.kind`：`at`（`at`）、`every`（`everyMs`）、または`cron`（`expr`、オプションの`tz`）。
* `schedule.at`はISO 8601を受け入れます（タイムゾーンはオプション；省略時はUTCとして扱われます）。
* `everyMs`はミリ秒です。
* `sessionTarget`は`"main"`または`"isolated"`でなければならず、`payload.kind`と一致する必要があります。
* オプションのフィールド：`agentId`、`description`、`enabled`、`deleteAfterRun`（`at`の場合はデフォルトでtrue）、`delivery`。
* `wakeMode`は省略時に`"now"`がデフォルトです。
### cron.updateパラメータ
```json
{
"jobId": "job-123",
"patch": {
"enabled": false,
"schedule": { "kind": "every", "everyMs": 3600000 }
}
}
```
ノート：
* `jobId`は標準です；互換性のために`id`が受け入れられます。
* パッチでエージェントバインディングをクリアするには`agentId: null`を使用します。
### cron.runおよびcron.removeパラメータ
```json
{ "jobId": "job-123", "mode": "force" }
```
```json
{ "jobId": "job-123" }
```
## ストレージと履歴
* ジョブストア：`~/.openclaw/cron/jobs.json`（Gateway管理のJSON）。
* 実行履歴：`~/.openclaw/cron/runs/.jsonl`（JSONL、自動的にサイズと行数でプルーニングされます）。
* アイソレートcron実行セッションは`sessions.json`にあり、`cron.sessionRetention`（デフォルト`24h`）でプルーニングされます；無効にするには`false`を設定します。
* ストアパスのオーバーライド：設定内の`cron.store`。
## 設定
```json5
{
cron: {
enabled: true, // デフォルトはtrue
store: "~/.openclaw/cron/jobs.json",
maxConcurrentRuns: 1, // デフォルトは1
webhook: "https://example.invalid/legacy", // 保存されたnotify:trueジョブのための非推奨のフォールバック
webhookToken: "replace-with-dedicated-webhook-token", // Webhookモード用のオプショナルベアラートークン
sessionRetention: "24h", // 期間文字列またはfalse
runLog: {
maxBytes: "2mb", // デフォルトは2_000_000バイト
keepLines: 2000, // デフォルトは2000
},
},
}
```
実行ログのプルーニング挙動：
* `cron.runLog.maxBytes`：プルーニング前の最大実行ログファイルサイズ。
* `cron.runLog.keepLines`：プルーニング時に最新のN行のみを保持します。
* 両方とも`cron/runs/.jsonl`ファイルに適用されます。
Webhookの挙動：
* 推奨：各ジョブに対して`delivery.mode: "webhook"`を設定し、`delivery.to: "https://..."`を指定します。
* Webhook URLは有効な`http://`または`https://` URLでなければなりません。
* 投稿されると、ペイロードはcron完了イベントのJSONです。
* `cron.webhookToken`が設定されている場合、認証ヘッダーは`Authorization: Bearer `です。
* `cron.webhookToken`が設定されていない場合、`Authorization`ヘッダーは送信されません。
* 非推奨のフォールバック：保存されたレガシージョブは、存在する場合に`cron.webhook`を使用します。
cronを完全に無効にする：
* `cron.enabled: false`（設定）
* `OPENCLAW_SKIP_CRON=1`（環境変数）
## メンテナンス
Cronには2つの組み込みメンテナンスポイントがあります：アイソレート実行セッションの保持と実行ログのプルーニング。
### デフォルト
* `cron.sessionRetention`：`24h`（実行セッションのプルーニングを無効にするには`false`を設定）
* `cron.runLog.maxBytes`：`2_000_000`バイト
* `cron.runLog.keepLines`：`2000`
### 仕組み
* アイソレート実行はセッションエントリ（`...:cron::run:`）とトランスクリプトファイルを作成します。
* 収穫者は`cron.sessionRetention`より古い期限切れの実行セッションエントリを削除します。
* セッションストアで参照されなくなった削除された実行セッションについて、OpenClawはトランスクリプトファイルをアーカイブし、同じ保持ウィンドウで古い削除されたアーカイブを削除します。
* 各実行の追加後、`cron/runs/.jsonl`はサイズチェックされます：
* ファイルサイズが`runLog.maxBytes`を超える場合、最新の`runLog.keepLines`行にトリミングされます。
### 高ボリュームスケジューラのパフォーマンス注意点
高頻度のcronセットアップは大きな実行セッションと実行ログのフットプリントを生成する可能性があります。メンテナンスは組み込まれていますが、緩い制限は回避可能なIOとクリーンアップ作業を引き起こす可能性があります。
注意すべきこと：
* 多くのアイソレート実行を持つ長い`cron.sessionRetention`ウィンドウ
* 大きな`runLog.maxBytes`と組み合わせた高い`cron.runLog.keepLines`
* 同じ`cron/runs/.jsonl`に書き込む多くの騒がしい定期ジョブ
対策：
* デバッグ/監査のニーズが許す限り短い`cron.sessionRetention`を維持
* 中程度の`runLog.maxBytes`と`runLog.keepLines`で実行ログを制限
* 不要なチャッターを避ける配信ルールで騒がしいバックグラウンドジョブをアイソレートモードに移動
* 定期的に`openclaw cron runs`で成長を確認し、ログが大きくなる前に保持を調整
### カスタマイズ例
実行セッションを1週間保持し、より大きな実行ログを許可：
```json5
{
cron: {
sessionRetention: "7d",
runLog: {
maxBytes: "10mb",
keepLines: 5000,
},
},
}
```
アイソレート実行セッションのプルーニングを無効にし、実行ログのプルーニングを維持：
```json5
{
cron: {
sessionRetention: false,
runLog: {
maxBytes: "5mb",
keepLines: 3000,
},
},
}
```
高ボリュームのcron使用に調整（例）：

## CLIクイックスタート
一度限りのリマインダー（UTC ISO、成功後に自動削除）：
```bash
openclaw cron add \
--name "Send reminder" \
--at "2026-01-12T18:00:00Z" \
--session main \
--system-event "Reminder: submit expense report." \
--wake now \
--delete-after-run
```
一度限りのリマインダー（メインセッション、即座に起こす）：
```bash
openclaw cron add \
--name "Calendar check" \
--at "20m" \
--session main \
--system-event "Next heartbeat: check calendar." \
--wake now
```

エージェント選択（マルチエージェントセットアップ）：
```bash
# エージェント "ops" にジョブを固定（そのエージェントが欠落している場合はデフォルトにフォールバック）
openclaw cron add --name "Ops sweep" --cron "0 6 * * *" --session isolated --message "Check ops queue" --agent ops
# 既存のジョブのエージェントを切り替えまたはクリア
openclaw cron edit  --agent ops
openclaw cron edit  --clear-agent
```
手動実行（forceはデフォルト、`--due`を使用して期限が来たときのみ実行）：
```bash
openclaw cron run 
openclaw cron run  --due
```
既存のジョブを編集（フィールドをパッチ）：
```bash
openclaw cron edit  \
--message "Updated prompt" \
--model "opus" \
--thinking low
```
既存のcronジョブを正確にスケジュール通りに実行（スタッガーなし）：
```bash
openclaw cron edit  --exact
```
実行履歴：
```bash
openclaw cron runs --id  --limit 50
```
ジョブを作成せずに即時システムイベント：
```bash
openclaw system event --mode now --text "Next heartbeat: check battery."
```

## トラブルシューティング
### “何も実行されない”
* cronが有効になっているか確認：`cron.enabled`および`OPENCLAW_SKIP_CRON`。
* Gatewayが継続的に実行されているか確認（cronはGatewayプロセス内で実行されます）。
* `cron`スケジュールの場合：タイムゾーン（`--tz`）とホストタイムゾーンを確認します。
### 繰り返しジョブが失敗後に遅延し続ける
* OpenClawは連続エラーの後に繰り返しジョブに指数的なリトライバックオフを適用します：
30秒、1分、5分、15分、次に60分の間隔でリトライします。
* バックオフは次の成功した実行の後に自動的にリセットされます。
* 一度限りの（`at`）ジョブは、端末実行（`ok`、`error`、または`skipped`）の後に無効になり、リトライしません。
### Telegramが間違った場所に配信する
* フォーラムトピックの場合、`-100…:topic:`を使用して明示的かつ明確にします。
* ログや保存された「最後のルート」ターゲットに`telegram:...`接頭辞が表示される場合、それは正常です；cron配信はそれらを受け入れ、トピックIDを正しく解析します。
### サブエージェントの発表配信リトライ
* サブエージェントの実行が完了すると、Gatewayはリクエスタセッションに結果を発表します。
* 発表フローが`false`を返す場合（例：リクエスタセッションがビジー）、Gatewayは最大3回リトライします。`announceRetryCount`で追跡します。
* `endedAt`から5分以上経過した発表は強制的に期限切れにされ、古いエントリが無限にループするのを防ぎます。
* ログに繰り返し発表配信が表示される場合は、高い`announceRetryCount`値を持つエントリがサブエージェントレジストリにあるか確認してください。
