## 最初の60秒で何かが壊れている場合

1. **クイックステータス（最初のチェック）**

   ```bash
   openclaw status
   ```

   ローカルの迅速な要約: OS + アップデート、Gateway/サービスの到達可能性、エージェント/セッション、プロバイダーの設定 + ランタイムの問題（Gatewayが到達可能な場合）。

2. **共有可能なレポート（安全に共有可能）**

   ```bash
   openclaw status --all
   ```

   ログのテール付きの読み取り専用診断（トークンは非表示）。

3. **デーモン + ポート状態**

   ```bash
   openclaw gateway status
   ```

   スーパーバイザーのランタイムとRPCの到達可能性、プローブ対象のURL、サービスが使用した可能性のある設定を表示します。

4. **詳細プローブ**

   ```bash
   openclaw status --deep
   ```

   Gatewayのヘルスチェック + プロバイダープローブを実行します（到達可能なGatewayが必要）。詳細は[Health](/gateway/health)を参照してください。

5. **最新のログをテールする**

   ```bash
   openclaw logs --follow
   ```

   RPCがダウンしている場合は、次のコマンドにフォールバックします：

   ```bash
   tail -f "$(ls -t /tmp/openclaw/openclaw-*.log | head -1)"
   ```

   ファイルログはサービスログとは別です。詳細は[Logging](/logging)と[Troubleshooting](/gateway/troubleshooting)を参照してください。

6. **ドクターを実行する（修復）**

   ```bash
   openclaw doctor
   ```

   設定/状態を修復/移行し、ヘルスチェックを実行します。詳細は[Doctor](/gateway/doctor)を参照してください。

7. **Gatewayスナップショット**

   ```bash
   openclaw health --json
   openclaw health --verbose   # エラー時にターゲットURL + 設定パスを表示
   ```

   実行中のGatewayにフルスナップショットを要求します（WS専用）。詳細は[Health](/gateway/health)を参照してください。

## クイックスタートと初回セットアップ

### どうしても詰まった場合、最も早く解決する方法は？

自分のマシンを**見ることができる**ローカルAIエージェントを使用してください。これはDiscordで尋ねるよりもはるかに効果的です。なぜなら、ほとんどの「詰まった」ケースは**ローカルの設定や環境の問題**であり、リモートのヘルパーはそれを検査できないからです。

* **Claude Code**: [https://www.anthropic.com/claude-code/](https://www.anthropic.com/claude-code/)
* **OpenAI Codex**: [https://openai.com/codex/](https://openai.com/codex/)

これらのツールはリポジトリを読み取り、コマンドを実行し、ログを検査し、マシンレベルのセットアップ（PATH、サービス、権限、認証ファイル）を修正するのに役立ちます。ハッカブル（git）インストールを介して**フルソースチェックアウト**を提供してください：

```bash
curl -fsSL https://openclaw.ai/install.sh | bash -s -- --install-method git
```

これにより、OpenClawが**gitチェックアウトからインストール**されるため、エージェントはコード + ドキュメントを読み取り、実行中の正確なバージョンについて推論できます。後で`--install-method git`なしでインストーラーを再実行することで、常に安定版に戻ることができます。

ヒント: エージェントに修正を**計画し監督**させてから、必要なコマンドのみを実行させてください。これにより、変更が小さくなり、監査が容易になります。

実際のバグや修正を発見した場合は、GitHubに問題を報告するか、PRを送信してください：
[https://github.com/openclaw/openclaw/issues](https://github.com/openclaw/openclaw/issues)
[https://github.com/openclaw/openclaw/pulls](https://github.com/openclaw/openclaw/pulls)

次のコマンドから始めてください（助けを求める際に出力を共有してください）：

```bash
openclaw status
openclaw models status
openclaw doctor
```

これらのコマンドの説明：

* `openclaw status`: Gateway/Agentのヘルス + 基本設定の迅速なスナップショット。
* `openclaw models status`: プロバイダーの認証 + モデルの可用性をチェックします。
* `openclaw doctor`: 一般的な設定/状態の問題を検証し修復します。

他の便利なCLIチェック: `openclaw status --all`, `openclaw logs --follow`,
`openclaw gateway status`, `openclaw health --verbose`。

クイックデバッグループ: [最初の60秒で何かが壊れている場合](#first-60-seconds-if-somethings-broken)。
インストールドキュメント: [Install](/install), [Installer flags](/install/installer), [Updating](/install/updating)。

### OpenClawをインストールしてセットアップする推奨方法は？

リポジトリはソースから実行し、オンボーディングウィザードを使用することを推奨しています：

```bash
curl -fsSL https://openclaw.ai/install.sh | bash
openclaw onboard --install-daemon
```

ウィザードはUIアセットを自動的にビルドすることもできます。オンボーディング後、通常はポート**18789**でGatewayを実行します。

ソースから（貢献者/開発者）：

```bash
git clone https://github.com/openclaw/openclaw.git
cd openclaw
pnpm install
pnpm build
pnpm ui:build # 初回実行時にUI依存関係を自動インストール
openclaw onboard
```

まだグローバルインストールがない場合は、`pnpm openclaw onboard`を介して実行してください。

### オンボーディング後にダッシュボードを開くにはどうすればよいですか？

ウィザードはオンボーディング直後にブラウザを開き、クリーン（トークンなし）のダッシュボードURLを表示し、サマリーにもリンクを印刷します。そのタブを開いたままにしてください。起動しなかった場合は、印刷されたURLを同じマシンにコピー/ペーストしてください。

### localhostとリモートでダッシュボードトークンを認証するにはどうすればよいですか？

**ローカルホスト（同じマシン）:**

* `http://127.0.0.1:18789/`を開きます。
* 認証を求められた場合は、`gateway.auth.token`（または`OPENCLAW_GATEWAY_TOKEN`）からトークンをControl UI設定にペーストします。
* Gatewayホストから取得します: `openclaw config get gateway.auth.token`（または生成します: `openclaw doctor --generate-gateway-token`）。

**ローカルホスト以外:**

* **Tailscale Serve**（推奨）: バインドループバックを維持し、`openclaw gateway --tailscale serve`を実行し、`https://<magicdns>/`を開きます。`gateway.auth.allowTailscale`が`true`の場合、アイデンティティヘッダーはControl UI/WebSocket認証を満たします（トークンなし、信頼されたGatewayホストを前提とします）；HTTP APIは依然としてトークン/パスワードを必要とします。
* **Tailnetバインド**: `openclaw gateway --bind tailnet --token "<token>"`を実行し、`http://<tailscale-ip>:18789/`を開き、ダッシュボード設定にトークンをペーストします。
* **SSHトンネル**: `ssh -N -L 18789:127.0.0.1:18789 user@host`を実行し、`http://127.0.0.1:18789/`を開き、Control UI設定にトークンをペーストします。

バインドモードと認証の詳細については、[Dashboard](/web/dashboard)と[Web surfaces](/web)を参照してください。

### どのランタイムが必要ですか？

Node **>= 22**が必要です。`pnpm`が推奨されます。Gatewayには**Bunは推奨されません**。

### Raspberry Piで動作しますか？

はい。Gatewayは軽量で、ドキュメントでは**512MB-1GB RAM**、**1コア**、約**500MB**のディスクが個人使用に十分であるとされています。また、**Raspberry Pi 4で実行可能です**。

追加の余裕（ログ、メディア、他のサービス）が必要な場合は、**2GBを推奨**しますが、これは厳密な最小要件ではありません。

ヒント: 小型のPi/VPSがGatewayをホストでき、ノートパソコンや電話上で**ノード**をペアリングしてローカルの画面/カメラ/キャンバスまたはコマンド実行を行うことができます。詳細は[Nodes](/nodes)を参照してください。

### Raspberry Piのインストールに関するヒントはありますか？

短いバージョン: 動作しますが、粗い部分があることを期待してください。

* **64ビット**のOSを使用し、Node >= 22を維持してください。
* **ハッカブル（git）インストール**を優先して、ログを確認し、迅速に更新できるようにしてください。
* チャンネル/スキルなしで開始し、1つずつ追加してください。
* 奇妙なバイナリの問題に直面した場合、通常は**ARMの互換性**の問題です。

ドキュメント: [Linux](/platforms/linux), [Install](/install)。

### 「Wake up my friend」のオンボーディングがハッチしない場合、どうすればよいですか？

その画面はGatewayが到達可能で認証されていることに依存します。TUIは最初のハッチ時に自動的に「Wake up, my friend!」を送信します。その行が表示され、**応答がない**場合、トークンが0のままであれば、エージェントは実行されていません。

1. Gatewayを再起動します：

```bash
openclaw gateway restart
```

2. ステータス + 認証を確認します：

```bash
openclaw status
openclaw models status
openclaw logs --follow
```

3. まだハングしている場合は、次を実行します：

```bash
openclaw doctor
```

Gatewayがリモートの場合、トンネル/Tailscale接続が正常であり、UIが正しいGatewayを指していることを確認してください。詳細は[Remote access](/gateway/remote)を参照してください。

### 新しいマシンMac miniにセットアップを移行できますか？オンボーディングをやり直さずに

はい。**状態ディレクトリ**と**ワークスペース**をコピーし、ドクターを一度実行します。これにより、ボットは「まったく同じ」状態（メモリ、セッション履歴、認証、チャンネル状態）を維持します。**両方**の場所をコピーする限り：

1. 新しいマシンにOpenClawをインストールします。
2. 古いマシンから`$OPENCLAW_STATE_DIR`（デフォルト: `~/.openclaw`）をコピーします。
3. ワークスペース（デフォルト: `~/.openclaw/workspace`）をコピーします。
4. `openclaw doctor`を実行し、Gatewayサービスを再起動します。

これにより、設定、認証プロファイル、WhatsAppの資格情報、セッション、メモリが保持されます。リモートモードの場合、Gatewayホストがセッションストアとワークスペースを所有していることを忘れないでください。

**重要:** ワークスペースをGitHubにコミット/プッシュするだけの場合、**メモリ + ブートストラップファイル**をバックアップしていますが、**セッション履歴や認証**はバックアップしていません。それらは`~/.openclaw/`の下に存在します（例: `~/.openclaw/agents/<agentId>/sessions/`）。

関連: [Migrating](/install/migrating), [Where things live on disk](/help/faq#where-does-openclaw-store-its-data),
[Agent workspace](/concepts/agent-workspace), [Doctor](/gateway/doctor),
[Remote mode](/gateway/remote)。

### 安定版とベータ版の違いは何ですか？

**安定版**と**ベータ版**は**npm dist-tags**であり、別のコードラインではありません：

* `latest` = 安定版
* `beta` = テスト用の早期ビルド

私たちは**ベータ**にビルドを出荷し、テストし、ビルドが堅牢であれば、その同じバージョンを**`latest`に昇格させます**。これが、ベータ版と安定版が**同じバージョン**を指すことができる理由です。

変更点を確認してください：
[https://github.com/openclaw/openclaw/blob/main/CHANGELOG.md](https://github.com/openclaw/openclaw/blob/main/CHANGELOG.md)

**開発版**は`main`（git）の移動する先端です；公開時にはnpm dist-tagの`dev`を使用します。


```bash

```

Windowsインストーラー（PowerShell）：
[https://openclaw.ai/install.ps1](https://openclaw.ai/install.ps1)

詳細: [Development channels](/install/development-channels)と[Installer flags](/install/installer)。

### Windows install says git not found or openclaw not recognized

二つの一般的なWindowsの問題：

### Telegram what goes in allowFrom

`channels.telegram.allowFrom`は**人間の送信者のTelegramユーザーID**（数値）です。ボットのユーザー名ではありません。

オンボーディングウィザードは`@username`の入力を受け付け、数値IDに解決しますが、OpenClawの認証は数値IDのみを使用します。

より安全（サードパーティボットなし）：

* ボットにDMを送り、`openclaw logs --follow`を実行して`from.id`を読み取ります。

公式ボットAPI：

* ボットにDMを送り、`https://api.telegram.org/bot<bot_token>/getUpdates`を呼び出して`message.from.id`を読み取ります。

サードパーティ（プライバシーが低い）：

* `@userinfobot`または`@getidsbot`にDMを送ります。

詳細は[/channels/telegram](/channels/telegram#access-control-dms--groups)を参照してください。

### HomebrewはLinuxで動作しますか？

はい。HomebrewはLinuxをサポートしています（Linuxbrew）。簡単なセットアップ：

```bash
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
echo 'eval "$(/home/linuxbrew/.linuxbrew/bin/brew shellenv)"' >> ~/.profile
eval "$(/home/linuxbrew/.linuxbrew/bin/brew shellenv)"

```

OpenClawをsystemd経由で実行する場合、サービスのPATHに `/home/linuxbrew/.linuxbrew/bin`（またはあなたのbrewプレフィックス）が含まれていることを確認してください。そうしないと、`brew`でインストールしたツールが非ログインシェルで解決されません。最近のビルドでは、Linuxのsystemdサービスで一般的なユーザービンディレクトリ（例えば `~/.local/bin`, `~/.npm-global/bin`, `~/.local/share/pnpm`, `~/.bun/bin`）を先頭に追加し、設定されている場合は `PNPM_HOME`, `NPM_CONFIG_PREFIX`, `BUN_INSTALL`, `VOLTA_HOME`, `ASDF_DATA_DIR`, `NVM_DIR`, `FNM_DIR` を尊重します。

### hackable gitインストールとnpmインストールの違いは何ですか？

* **Hackable (git) install:** 完全なソースチェックアウト、編集可能、貢献者に最適。
  ローカルでビルドを実行し、コードやドキュメントをパッチできます。
* **npm install:** グローバルCLIインストール、リポジトリなし、「ただ実行する」ために最適。
  更新はnpmのdistタグから行われます。

ドキュメント: [Getting started](/start/getting-started), [Updating](/install/updating)。

### 後でnpmとgitのインストールを切り替えることはできますか？

はい。もう一方のインストールを行い、その後Doctorを実行してゲートウェイサービスが新しいエントリポイントを指すようにします。
これは**データを削除しません** - OpenClawのコードインストールのみが変更されます。あなたの状態
(`~/.openclaw`) とワークスペース (`~/.openclaw/workspace`) はそのままです。

npmからgitへ：

```bash
git clone https://github.com/openclaw/openclaw.git
cd openclaw
pnpm install
pnpm build
openclaw doctor
openclaw gateway restart
```

gitからnpmへ：

```bash
npm install -g openclaw@latest
openclaw doctor
openclaw gateway restart
```

Doctorはゲートウェイサービスのエントリポイントの不一致を検出し、現在のインストールに合わせてサービス設定を書き換える提案をします（自動化では `--repair` を使用）。

バックアップのヒント: [Backup strategy](/help/faq#whats-the-recommended-backup-strategy) を参照してください。

### ゲートウェイをノートパソコンで実行するべきですか、それともVPSで実行するべきですか？

短い答え: **24/7の信頼性が必要な場合はVPSを使用してください**。最低限の摩擦を求めていて、スリープや再起動が許容できる場合は、ローカルで実行してください。

**ノートパソコン（ローカルゲートウェイ）**

* **利点:** サーバーコストなし、ローカルファイルへの直接アクセス、ライブブラウザウィンドウ。
* **欠点:** スリープ/ネットワークドロップ = 切断、OSの更新/再起動が中断、常に起きている必要がある。

**VPS / クラウド**

* **利点:** 常時稼働、安定したネットワーク、ノートパソコンのスリープ問題なし、運用が容易。
* **欠点:** ヘッドレスで実行されることが多い（スクリーンショットを使用）、リモートファイルアクセスのみ、更新にはSSHが必要。

**OpenClaw特有の注意:** WhatsApp/Telegram/Slack/Mattermost（プラグイン）/DiscordはすべてVPSから正常に動作します。唯一の本当のトレードオフは**ヘッドレスブラウザ**対可視ウィンドウです。 [Browser](/tools/browser) を参照してください。

**推奨デフォルト:** 以前にゲートウェイの切断があった場合はVPSを使用してください。ローカルは、Macをアクティブに使用しているときや、可視ブラウザでのローカルファイルアクセスやUI自動化を希望する場合に最適です。

### OpenClawを専用のマシンで実行することはどれほど重要ですか？

必須ではありませんが、**信頼性と隔離のために推奨されます**。

* **専用ホスト（VPS/Mac mini/Pi）:** 常時稼働、スリープ/再起動の中断が少ない、クリーンな権限、運用が容易。
* **共有ノートパソコン/デスクトップ:** テストやアクティブな使用には全く問題ありませんが、マシンがスリープしたり更新されたりするときに一時停止が予想されます。

両方の利点を得たい場合は、ゲートウェイを専用ホストで運用し、ノートパソコンを**ノード**としてローカルのスクリーン/カメラ/execツールにペアリングしてください。 [Nodes](/nodes) を参照してください。
セキュリティガイダンスについては、[Security](/gateway/security) をお読みください。

### 最低限のVPS要件と推奨OSは何ですか？

OpenClawは軽量です。基本的なゲートウェイ + 1つのチャットチャネルの場合：

* **絶対最低:** 1 vCPU, 1GB RAM, 約500MBのディスク。
* **推奨:** 1-2 vCPU, 2GB RAM以上（ログ、メディア、複数のチャネル用の余裕）。Nodeツールやブラウザの自動化はリソースを多く消費する可能性があります。

OS: **Ubuntu LTS**（または他の最新のDebian/Ubuntu）を使用してください。Linuxのインストールパスはそこで最もよくテストされています。

ドキュメント: [Linux](/platforms/linux), [VPS hosting](/vps)。

### OpenClawをVMで実行できますか、要件は何ですか？

はい。VMはVPSと同様に扱います：常に稼働していて、到達可能で、ゲートウェイと有効にするチャネルに十分なRAMが必要です。

基本的なガイダンス：

* **絶対最低:** 1 vCPU, 1GB RAM。
* **推奨:** 複数のチャネル、ブラウザの自動化、またはメディアツールを実行する場合は2GB RAM以上。
* **OS:** Ubuntu LTSまたは他の最新のDebian/Ubuntu。

Windowsを使用している場合、**WSL2が最も簡単なVMスタイルのセットアップ**で、最良のツール互換性があります。 [Windows](/platforms/windows), [VPS hosting](/vps) を参照してください。
macOSをVMで実行している場合は、[macOS VM](/install/macos-vm) を参照してください。

## OpenClaw とは？

### OpenClawとは何ですか？

OpenClawは、あなた自身のデバイス上で実行する個人のAIアシスタントです。すでに使用しているメッセージングプラットフォーム（WhatsApp、Telegram、Slack、Mattermost（プラグイン）、Discord、Google Chat、Signal、iMessage、WebChat）で応答し、サポートされているプラットフォームでは音声とライブCanvasも行うことができます。**Gateway**は常時稼働する制御プレーンであり、アシスタントが製品です。

### 価値提案は何ですか？

OpenClawは「単なるClaudeラッパー」ではありません。これは、**あなた自身のハードウェア**上で有能なアシスタントを実行できる**ローカルファーストの制御プレーン**であり、すでに使用しているチャットアプリからアクセスでき、状態を持つセッション、メモリ、ツールを提供します - ホスティングされたSaaSにワークフローの制御を渡すことなく。

ハイライト：

* **あなたのデバイス、あなたのデータ:** ゲートウェイを好きな場所（Mac、Linux、VPS）で実行し、ワークスペースとセッション履歴をローカルに保つことができます。
* **実際のチャネル、ウェブサンドボックスではない:** WhatsApp/Telegram/Slack/Discord/Signal/iMessageなど、さらにモバイル音声とサポートされているプラットフォームでのCanvas。
* **モデルに依存しない:** Anthropic、OpenAI、MiniMax、OpenRouterなどを使用し、エージェントごとのルーティングとフェイルオーバーを行います。
* **ローカル専用オプション:** ローカルモデルを実行することで、**すべてのデータをデバイスに保持**できます。
* **マルチエージェントルーティング:** チャネル、アカウント、またはタスクごとにエージェントを分け、それぞれに独自のワークスペースとデフォルトを持たせます。
* **オープンソースでハッカブル:** ベンダーロックインなしで、検査、拡張、自己ホストが可能です。

ドキュメント: [Gateway](/gateway), [Channels](/channels), [Multi-agent](/concepts/multi-agent), [Memory](/concepts/memory)。

### セットアップしたばかりですが、最初に何をすべきですか？

良い最初のプロジェクト：

* ウェブサイトを構築する（WordPress、Shopify、またはシンプルな静的サイト）。
* モバイルアプリのプロトタイプを作成する（アウトライン、画面、API計画）。
* ファイルとフォルダを整理する（クリーンアップ、命名、タグ付け）。
* Gmailを接続し、要約やフォローアップを自動化する。

大きなタスクを処理できますが、フェーズに分けてサブエージェントを使用して並行作業を行うと最も効果的です。

### OpenClawの一般的な使用例トップ5は何ですか？

日常の勝利は通常次のようになります：

* **個人のブリーフィング:** あなたが気に入っている受信トレイ、カレンダー、ニュースの要約。
* **リサーチとドラフト作成:** クイックリサーチ、要約、メールやドキュメントの初稿。
* **リマインダーとフォローアップ:** cronまたはheartbeat駆動の通知とチェックリスト。
* **ブラウザ自動化:** フォームの入力、データの収集、ウェブタスクの繰り返し。
* **デバイス間の調整:** スマートフォンからタスクを送信し、ゲートウェイがサーバー上で実行し、チャットで結果を受け取る。

### OpenClawはSaaSのリード生成アウトリーチ広告やブログに役立ちますか？

はい、**リサーチ、資格確認、ドラフト作成**に役立ちます。サイトをスキャンし、ショートリストを作成し、見込み客を要約し、アウトリーチや広告コピーのドラフトを作成できます。

**アウトリーチや広告の実行**については、人間をループに入れておいてください。スパムを避け、地域の法律やプラットフォームポリシーに従い、送信前にすべてをレビューしてください。最も安全なパターンは、OpenClawにドラフトを作成させ、あなたが承認することです。

ドキュメント: [Security](/gateway/security)。

### Web開発におけるClaude Codeとの利点は何ですか？

OpenClawは**個人アシスタント**および調整レイヤーであり、IDEの置き換えではありません。リポジトリ内で最も迅速な直接コーディングループにはClaude CodeやCodexを使用してください。持続的なメモリ、デバイス間アクセス、ツールのオーケストレーションが必要なときにOpenClawを使用してください。

利点：

* セッション間での**持続的なメモリ + ワークスペース**
* **マルチプラットフォームアクセス**（WhatsApp、Telegram、TUI、WebChat）
* **ツールのオーケストレーション**（ブラウザ、ファイル、スケジューリング、フック）
* **常時稼働のゲートウェイ**（VPSで実行し、どこからでもインタラクト）
* ローカルブラウザ/スクリーン/カメラ/exec用の**ノード**

ショーケース: [https://openclaw.ai/showcase](https://openclaw.ai/showcase)

## スキルと自動化

### リポジトリを汚さずにスキルをカスタマイズするにはどうすればよいですか？

リポジトリのコピーを編集するのではなく、管理されたオーバーライドを使用してください。変更を `~/.openclaw/skills/<name>/SKILL.md` に入れるか、`~/.openclaw/openclaw.json` の `skills.load.extraDirs` を介してフォルダを追加します。優先順位は `<workspace>/skills` > `~/.openclaw/skills` > バンドルされているので、管理されたオーバーライドがgitに触れることなく勝ちます。上流に値する編集のみがリポジトリに存在し、PRとして出されるべきです。

### カスタムフォルダからスキルを読み込むことはできますか？

はい。 `~/.openclaw/openclaw.json` の `skills.load.extraDirs` を介して追加のディレクトリを追加します（最も低い優先順位）。デフォルトの優先順位は次のままです：`<workspace>/skills` → `~/.openclaw/skills` → バンドル → `skills.load.extraDirs`。`clawhub`はデフォルトで`./skills`にインストールされ、OpenClawはこれを`<workspace>/skills`として扱います。

### 異なるタスクに異なるモデルを使用するにはどうすればよいですか？

現在サポートされているパターンは次のとおりです：

* **Cronジョブ:** 隔離されたジョブは、ジョブごとに`model`オーバーライドを設定できます。
* **サブエージェント:** タスクを異なるデフォルトモデルを持つ別のエージェントにルーティングします。
* **オンデマンドスイッチ:** `/model`を使用して、いつでも現在のセッションモデルを切り替えます。

[cron jobs](/automation/cron-jobs)、[Multi-Agent Routing](/concepts/multi-agent)、および[Slash commands](/tools/slash-commands)を参照してください。

### ボットが重い作業中にフリーズします。どうやってオフロードしますか？

長いまたは並行タスクには**サブエージェント**を使用してください。サブエージェントは独自のセッションで実行され、要約を返し、メインチャットを応答可能に保ちます。

ボットに「このタスクのためにサブエージェントを生成する」と頼むか、`/subagents`を使用してください。
チャットで`/status`を使用して、ゲートウェイが現在何をしているか（および忙しいかどうか）を確認できます。

トークンのヒント: 長いタスクとサブエージェントは両方ともトークンを消費します。コストが懸念される場合は、`agents.defaults.subagents.model`を介してサブエージェント用に安価なモデルを設定してください。

ドキュメント: [Sub-agents](/tools/subagents)。

### Discordでのスレッドバウンドサブエージェントセッションはどのように機能しますか？

スレッドバインディングを使用します。Discordのスレッドをサブエージェントまたはセッションターゲットにバインドできるため、そのスレッド内のフォローアップメッセージはそのバインドされたセッションに留まります。

基本的なフロー：

* `sessions_spawn`を使用して`thread: true`で生成します（オプションで`mode: "session"`を指定して持続的なフォローアップを行います）。
* または、`/focus <target>`で手動でバインドします。
* `/agents`を使用してバインディング状態を確認します。
* `/session idle <duration|off>`および`/session max-age <duration|off>`を使用して自動フォーカス解除を制御します。
* `/unfocus`を使用してスレッドをデタッチします。

必要な設定：

* グローバルデフォルト: `session.threadBindings.enabled`, `session.threadBindings.idleHours`, `session.threadBindings.maxAgeHours`。
* Discordオーバーライド: `channels.discord.threadBindings.enabled`, `channels.discord.threadBindings.idleHours`, `channels.discord.threadBindings.maxAgeHours`。
* スポーン時の自動バインド: `channels.discord.threadBindings.spawnSubagentSessions: true`を設定します。

ドキュメント: [Sub-agents](/tools/subagents)、[Discord](/channels/discord)、[Configuration Reference](/gateway/configuration-reference)、[Slash commands](/tools/slash-commands)。

### Cronやリマインダーが発火しません。何を確認すべきですか？

Cronはゲートウェイプロセス内で実行されます。ゲートウェイが継続的に実行されていない場合、スケジュールされたジョブは実行されません。

チェックリスト：

* cronが有効であることを確認します（`cron.enabled`）および`OPENCLAW_SKIP_CRON`が設定されていないこと。
* ゲートウェイが24/7稼働していることを確認します（スリープ/再起動なし）。
* ジョブのタイムゾーン設定を確認します（`--tz`対ホストタイムゾーン）。

デバッグ：

```bash
openclaw cron run <jobId> --force
openclaw cron runs --id <jobId> --limit 50
```

ドキュメント: [Cron jobs](/automation/cron-jobs)、[Cron vs Heartbeat](/automation/cron-vs-heartbeat)。
### Linuxにスキルをインストールするには

**ClawHub** (CLI) を使用するか、スキルをワークスペースにドロップします。macOSのスキルUIはLinuxでは利用できません。スキルは [https://clawhub.com](https://clawhub.com) でブラウズできます。

ClawHub CLIをインストールします（パッケージマネージャを1つ選択）：

```bash
npm i -g clawhub
```

```bash
pnpm add -g clawhub
```

### OpenClawはタスクをスケジュールまたはバックグラウンドで継続的に実行できますか？

はい。Gatewayスケジューラを使用します：

* **Cronジョブ**：スケジュールされたまたは定期的なタスク（再起動を跨いで持続）。
* **Heartbeat**： "メインセッション" の定期チェック。
* **孤立したジョブ**：要約を投稿したりチャットに配信する自律エージェント用。

ドキュメント：[Cronジョブ](/automation/cron-jobs)、[Cron vs Heartbeat](/automation/cron-vs-heartbeat)、[Heartbeat](/gateway/heartbeat)。

### LinuxからApple macOS専用のスキルを実行できますか？

直接はできません。macOSスキルは `metadata.openclaw.os` と必要なバイナリによって制限されており、スキルは**Gatewayホスト**で適格な場合にのみシステムプロンプトに表示されます。Linuxでは、`darwin`専用のスキル（例えば `apple-notes`、`apple-reminders`、`things-mac`）は、ゲーティングをオーバーライドしない限り読み込まれません。

サポートされているパターンは3つあります：

**オプションA - MacでGatewayを実行する（最も簡単）。**
macOSバイナリが存在する場所でGatewayを実行し、Linuxから[リモートモード](#how-do-i-run-openclaw-in-remote-mode-client-connects-to-a-gateway-elsewhere)またはTailscale経由で接続します。GatewayホストがmacOSであるため、スキルは通常通り読み込まれます。

**オプションB - macOSノードを使用する（SSHなし）。**
LinuxでGatewayを実行し、macOSノード（メニューバーアプリ）をペアリングし、Macで**Node Run Commands**を「常に尋ねる」または「常に許可する」に設定します。OpenClawは、必要なバイナリがノードに存在する場合、macOS専用のスキルを適格として扱うことができます。エージェントは`nodes`ツールを介してそれらのスキルを実行します。「常に尋ねる」を選択した場合、プロンプトで「常に許可する」を承認すると、そのコマンドが許可リストに追加されます。

**オプションC - SSH経由でmacOSバイナリをプロキシする（高度な方法）。**
Linux上でGatewayを維持しますが、必要なCLIバイナリがMac上で実行されるSSHラッパーに解決されるようにします。次に、スキルをオーバーライドしてLinuxを許可し、適格な状態を維持します。

1. バイナリ用のSSHラッパーを作成します（例：Apple Notes用の`memo`）：

   ```bash
   #!/usr/bin/env bash
   set -euo pipefail
   exec ssh -T user@mac-host /opt/homebrew/bin/memo "$@"
   ```

2. ラッパーをLinuxホストの`PATH`に置きます（例えば`~/bin/memo`）。

3. スキルメタデータをオーバーライドします（ワークスペースまたは`~/.openclaw/skills`）：

   ```markdown
   ---
   name: apple-notes
   description: Manage Apple Notes via the memo CLI on macOS.
   metadata: { "openclaw": { "os": ["darwin", "linux"], "requires": { "bins": ["memo"] } } }
   ---
   ```

4. スキルスナップショットを更新するために新しいセッションを開始します。

### NotionまたはHeyGenの統合はありますか？

現在は組み込まれていません。

オプション：

* **カスタムスキル / プラグイン**：信頼性の高いAPIアクセスに最適（Notion/HeyGenの両方にAPIがあります）。
* **ブラウザ自動化**：コードなしで動作しますが、遅くて壊れやすいです。

クライアントごとにコンテキストを保持したい場合（エージェンシーのワークフロー）、シンプルなパターンは次の通りです：

* クライアントごとに1つのNotionページ（コンテキスト + 優先事項 + アクティブな作業）。
* セッションの開始時にエージェントにそのページを取得させる。

ネイティブ統合を希望する場合は、機能リクエストを開くか、これらのAPIをターゲットにしたスキルを構築してください。

スキルをインストールするには：

```bash
clawhub install <skill-slug>
clawhub update --all
```

ClawHubは現在のディレクトリの`./skills`にインストールされます（または設定されたOpenClawワークスペースにフォールバックします）。OpenClawは次のセッションでそれを`<workspace>/skills`として扱います。エージェント間で共有スキルを使用するには、`~/.openclaw/skills/<name>/SKILL.md`に配置します。一部のスキルはHomebrewを介してインストールされたバイナリを期待します。Linuxの場合、それはLinuxbrewを意味します（上記のHomebrew Linux FAQエントリを参照）。[Skills](/tools/skills)と[ClawHub](/tools/clawhub)を参照してください。

### ブラウザテイクオーバー用のChrome拡張機能をインストールするにはどうすればよいですか？

組み込みのインストーラーを使用し、次にChromeで未パッケージの拡張機能を読み込みます：

```bash
openclaw browser extension install
openclaw browser extension path
```

次に、Chrome → `chrome://extensions` → 「デベロッパーモード」を有効にする → 「未パッケージの読み込み」 → そのフォルダーを選択します。

完全なガイド（リモートGateway + セキュリティノートを含む）：[Chrome拡張機能](/tools/chrome-extension)

GatewayがChromeと同じマシン上で実行されている場合（デフォルトのセットアップ）、通常は**追加のものは必要ありません**。Gatewayが別の場所で実行されている場合、ブラウザマシン上でノードホストを実行して、Gatewayがブラウザアクションをプロキシできるようにします。制御したいタブで拡張機能ボタンをクリックする必要があります（自動的にアタッチされません）。

## サンドボックスとメモリ

### 専用のサンドボックスドキュメントはありますか？

はい。[Sandboxing](/gateway/sandboxing)を参照してください。Docker特有のセットアップ（Docker内のフルGatewayまたはサンドボックスイメージ）については、[Docker](/install/docker)を参照してください。

### Dockerは制限があるように感じます。フル機能を有効にするにはどうすればよいですか？

デフォルトのイメージはセキュリティを重視しており、`node`ユーザーとして実行されるため、システムパッケージ、Homebrew、またはバンドルされたブラウザは含まれていません。より完全なセットアップのために：

* `/home/node`を`OPENCLAW_HOME_VOLUME`で持続させ、キャッシュが生き残るようにします。
* `OPENCLAW_DOCKER_APT_PACKAGES`を使用して、システム依存関係をイメージに焼き込みます。
* バンドルされたCLIを介してPlaywrightブラウザをインストールします：
  `node /app/node_modules/playwright-core/cli.js install chromium`
* `PLAYWRIGHT_BROWSERS_PATH`を設定し、パスが持続することを確認します。

ドキュメント：[Docker](/install/docker)、[Browser](/tools/browser)。

**DMを個人的に保ちながら、1つのエージェントで公開サンドボックスグループを作成できますか？**

はい - あなたのプライベートトラフィックが**DM**であり、公開トラフィックが**グループ**である場合。

`agents.defaults.sandbox.mode: "non-main"`を使用して、グループ/チャネルセッション（非メインキー）がDockerで実行されるようにし、メインDMセッションはホスト上に留まります。次に、サンドボックスセッションで利用可能なツールを`tools.sandbox.tools`を介して制限します。

セットアップの手順 + サンプル設定：[グループ：個人的なDM + 公開グループ](/channels/groups#pattern-personal-dms-public-groups-single-agent)

主要な設定リファレンス：[Gateway設定](/gateway/configuration#agentsdefaultssandbox)

### ホストフォルダーをサンドボックスにバインドするにはどうすればよいですか？

`agents.defaults.sandbox.docker.binds`を`["host:path:mode"]`に設定します（例：`"/home/user/src:/src:ro"`）。グローバル + エージェントごとのバインドはマージされます；エージェントごとのバインドは`scope: "shared"`の場合無視されます。敏感なものには`:ro`を使用し、バインドがサンドボックスのファイルシステムの壁をバイパスすることを忘れないでください。例や安全ノートについては、[Sandboxing](/gateway/sandboxing#custom-bind-mounts)と[Sandbox vs Tool Policy vs Elevated](/gateway/sandbox-vs-tool-policy-vs-elevated#bind-mounts-security-quick-check)を参照してください。

### メモリはどのように機能しますか？

OpenClawのメモリは、エージェントワークスペース内のMarkdownファイルです：

* 日次ノートは`memory/YYYY-MM-DD.md`に保存されます。
* キュレーションされた長期ノートは`MEMORY.md`に保存されます（メイン/プライベートセッションのみ）。

OpenClawはまた、モデルに耐久性のあるノートを書き込むようにリマインドするために**サイレントな事前圧縮メモリフラッシュ**を実行します。これは、ワークスペースが書き込み可能なときのみ実行されます（読み取り専用のサンドボックスではスキップされます）。[Memory](/concepts/memory)を参照してください。

### メモリが物事を忘れ続けます。どうすれば記憶させることができますか？

ボットに**事実をメモリに書き込む**ように依頼してください。長期ノートは`MEMORY.md`に、短期コンテキストは`memory/YYYY-MM-DD.md`に入れます。

これはまだ改善中の領域です。モデルにメモリを保存するようにリマインドすることが助けになります；モデルは何をすべきかを知っています。忘れ続ける場合は、Gatewayが毎回同じワークスペースを使用していることを確認してください。

ドキュメント：[Memory](/concepts/memory)、[Agent workspace](/concepts/agent-workspace)。

### セマンティックメモリ検索にはOpenAI APIキーが必要ですか？

**OpenAIの埋め込み**を使用する場合のみ必要です。Codex OAuthはチャット/完了をカバーし、埋め込みアクセスを**付与しません**。したがって、**Codexでサインインすること（OAuthまたはCodex CLIログイン）**は、セマンティックメモリ検索には役立ちません。OpenAIの埋め込みには、実際のAPIキー（`OPENAI_API_KEY`または`models.providers.openai.apiKey`）が必要です。

プロバイダーを明示的に設定しない場合、OpenClawはAPIキーを解決できるときにプロバイダーを自動的に選択します（認証プロファイル、`models.providers.*.apiKey`、または環境変数）。OpenAIキーが解決される場合はOpenAIを優先し、そうでなければGemini、次にVoyage、次にMistralを優先します。リモートキーが利用できない場合、メモリ検索は構成されるまで無効のままです。ローカルモデルパスが構成されていて存在する場合、OpenClawは`local`を優先します。

ローカルに留まりたい場合は、`memorySearch.provider = "local"`を設定します（オプションで`memorySearch.fallback = "none"`）。Geminiの埋め込みを希望する場合は、`memorySearch.provider = "gemini"`を設定し、`GEMINI_API_KEY`（または`memorySearch.remote.apiKey`）を提供します。**OpenAI、Gemini、Voyage、Mistral、またはローカル**の埋め込みモデルをサポートしています - セットアップの詳細は[Memory](/concepts/memory)を参照してください。

### メモリは永遠に持続しますか？制限はありますか？

メモリファイルはディスク上に存在し、削除するまで持続します。制限はストレージにあり、モデルにはありません。**セッションコンテキスト**はモデルのコンテキストウィンドウによって依然として制限されているため、長い会話は圧縮または切り捨てられることがあります。これがメモリ検索が存在する理由です - それは関連する部分のみをコンテキストに戻します。

ドキュメント：[Memory](/concepts/memory)、[Context](/concepts/context)。

## ディスク上のデータの保存場所

### OpenClawで使用されるすべてのデータはローカルに保存されていますか？

いいえ - **OpenClawの状態はローカルですが、** **外部サービスはあなたが送信するものを見ることができます**。

* **デフォルトでローカル**：セッション、メモリファイル、設定、ワークスペースはGatewayホスト上に存在します（`~/.openclaw` + あなたのワークスペースディレクトリ）。
* **必要に応じてリモート**：モデルプロバイダー（Anthropic/OpenAIなど）に送信するメッセージはそのAPIに送信され、チャットプラットフォーム（WhatsApp/Telegram/Slackなど）はメッセージデータをそのサーバーに保存します。
* **あなたがフットプリントを制御します**：ローカルモデルを使用するとプロンプトがあなたのマシンに保持されますが、チャネルトラフィックは依然としてチャネルのサーバーを通過します。

関連：[Agent workspace](/concepts/agent-workspace)、[Memory](/concepts/memory)。

### OpenClawはデータをどこに保存しますか？

すべてのデータは`$OPENCLAW_STATE_DIR`の下に存在します（デフォルト：`~/.openclaw`）：

| パス                                                            | 目的                                                            |
| --------------------------------------------------------------- | ------------------------------------------------------------------ |
| `$OPENCLAW_STATE_DIR/openclaw.json`                             | メイン設定（JSON5）                                                |
| `$OPENCLAW_STATE_DIR/credentials/oauth.json`                    | レガシーOAuthインポート（初回使用時に認証プロファイルにコピーされる）       |
| `$OPENCLAW_STATE_DIR/agents/<agentId>/agent/auth-profiles.json` | 認証プロファイル（OAuth、APIキー、およびオプションの`keyRef`/`tokenRef`）  |
| `$OPENCLAW_STATE_DIR/secrets.json`                              | `file` SecretRefプロバイダー用のオプションのファイルバックされた秘密ペイロード |
| `$OPENCLAW_STATE_DIR/agents/<agentId>/agent/auth.json`          | レガシー互換性ファイル（静的な`api_key`エントリが削除される）      |
| `$OPENCLAW_STATE_DIR/credentials/`                              | プロバイダーの状態（例：`whatsapp/<accountId>/creds.json`）            |
| `$OPENCLAW_STATE_DIR/agents/`                                   | エージェントごとの状態（agentDir + セッション）                              |
| `$OPENCLAW_STATE_DIR/agents/<agentId>/sessions/`                | 会話の履歴と状態（エージェントごと）                           |
| `$OPENCLAW_STATE_DIR/agents/<agentId>/sessions/sessions.json`   | セッションメタデータ（エージェントごと）                                       |

レガシーの単一エージェントパス：`~/.openclaw/agent/*`（`openclaw doctor`によって移行されます）。

あなたの**ワークスペース**（AGENTS.md、メモリファイル、スキルなど）は別で、`agents.defaults.workspace`を介して構成されます（デフォルト：`~/.openclaw/workspace`）。

### AGENTS.md、SOUL.md、USER.md、MEMORY.mdはどこに保存すべきですか？

これらのファイルは**エージェントワークスペース**に存在し、`~/.openclaw`には存在しません。

* **ワークスペース（エージェントごと）**：`AGENTS.md`、`SOUL.md`、`IDENTITY.md`、`USER.md`、`MEMORY.md`（または`memory.md`）、`memory/YYYY-MM-DD.md`、オプションの`HEARTBEAT.md`。
* **状態ディレクトリ（`~/.openclaw`）**：設定、認証情報、認証プロファイル、セッション、ログ、および共有スキル（`~/.openclaw/skills`）。

デフォルトのワークスペースは`~/.openclaw/workspace`で、次のように構成できます：

```json5
{
  agents: { defaults: { workspace: "~/.openclaw/workspace" } },
}
```

ボットが再起動後に「忘れる」場合は、Gatewayが毎回同じワークスペースを使用していることを確認してください（リモートモードは**Gatewayホストの**ワークスペースを使用し、あなたのローカルラップトップのものではありません）。

ヒント：耐久性のある動作や好みを希望する場合は、ボットに**AGENTS.mdまたはMEMORY.mdに書き込むように依頼**することをお勧めします。チャット履歴に依存するのではなく。

[Agent workspace](/concepts/agent-workspace)と[Memory](/concepts/memory)を参照してください。
### 推奨されるバックアップ戦略

**エージェントワークスペース**を**プライベート**なGitリポジトリに置き、どこかプライベートな場所（例えばGitHubのプライベートリポジトリ）にバックアップしてください。これにより、メモリやAGENTS/SOUL/USERファイルがキャプチャされ、後でアシスタントの「心」を復元できます。

`~/.openclaw`以下の何もコミットしないでください（認証情報、セッション、トークン、または暗号化されたシークレットペイロード）。完全な復元が必要な場合は、ワークスペースと状態ディレクトリを別々にバックアップしてください（上記の移行に関する質問を参照）。

ドキュメント: [エージェントワークスペース](/concepts/agent-workspace)。

### OpenClawを完全にアンインストールするには

専用ガイドを参照してください: [アンインストール](/install/uninstall)。

### エージェントはワークスペースの外で動作できますか

はい。ワークスペースは**デフォルトの作業ディレクトリ**およびメモリアンカーであり、厳密なサンドボックスではありません。相対パスはワークスペース内で解決されますが、絶対パスはサンドボックスが有効でない限り、他のホストの場所にアクセスできます。隔離が必要な場合は、[`agents.defaults.sandbox`](/gateway/sandboxing)またはエージェントごとのサンドボックス設定を使用してください。リポジトリをデフォルトの作業ディレクトリにしたい場合は、そのエージェントの`workspace`をリポジトリのルートにポイントします。OpenClawリポジトリは単なるソースコードですので、エージェントがその中で動作することを意図しない限り、ワークスペースは別に保つべきです。

例（リポジトリをデフォルトの作業ディレクトリとして）:

```json5
{
  agents: {
    defaults: {
      workspace: "~/Projects/my-repo",
    },
  },
}
```

### リモートモードでセッションストアはどこですか

セッション状態は**ゲートウェイホスト**が所有しています。リモートモードの場合、あなたが気にするセッションストアはリモートマシン上にあり、ローカルのラップトップ上にはありません。セッション管理については[セッション管理](/concepts/session)を参照してください。

## 設定の基本

### 設定はどのフォーマットで、どこにありますか

OpenClawは、`$OPENCLAW_CONFIG_PATH`からオプションの**JSON5**設定を読み込みます（デフォルト: `~/.openclaw/openclaw.json`）:

```
$OPENCLAW_CONFIG_PATH
```

ファイルが存在しない場合、安全なデフォルト（デフォルトのワークスペースとして`~/.openclaw/workspace`を含む）を使用します。

### gatewaybind lanまたはtailnetを設定したら、何もリッスンしなくなり、UIが未認証と言っています

非ループバックバインドは**認証が必要**です。`gateway.auth.mode`と`gateway.auth.token`を設定してください（または`OPENCLAW_GATEWAY_TOKEN`を使用します）。

```json5
{
  gateway: {
    bind: "lan",
    auth: {
      mode: "token",
      token: "replace-me",
    },
  },
}
```

注意:

* `gateway.remote.token` / `.password`は、ローカルゲートウェイ認証を単独では有効にしません。
* ローカル呼び出しパスは、`gateway.auth.*`が未設定のときに`gateway.remote.*`をフォールバックとして使用できます。
* コントロールUIは`connect.params.auth.token`を介して認証します（アプリ/UI設定に保存）。トークンをURLに入れるのは避けてください。

### なぜ今、localhostでトークンが必要なのですか

OpenClawはデフォルトでトークン認証を強制します。ループバックを含めてです。トークンが設定されていない場合、ゲートウェイの起動時に自動生成され、`gateway.auth.token`に保存されるため、**ローカルWSクライアントは認証する必要があります**。これにより、他のローカルプロセスがゲートウェイを呼び出すことがブロックされます。

**本当に**オープンループバックが必要な場合は、設定で`gateway.auth.mode: "none"`を明示的に設定してください。Doctorはいつでもトークンを生成できます: `openclaw doctor --generate-gateway-token`。

### 設定を変更した後、再起動する必要がありますか

ゲートウェイは設定を監視し、ホットリロードをサポートしています:

* `gateway.reload.mode: "hybrid"`（デフォルト）: 安全な変更をホット適用し、重要な変更には再起動が必要です。
* `hot`、`restart`、`off`もサポートされています。

### ウェブ検索とウェブフェッチを有効にするにはどうすればよいですか

`web_fetch`はAPIキーなしで動作します。`web_search`はBrave Search APIキーが必要です。**推奨:** `openclaw configure --section web`を実行して、`tools.web.search.apiKey`に保存してください。環境変数の代替: ゲートウェイプロセスのために`BRAVE_API_KEY`を設定します。

```json5
{
  tools: {
    web: {
      search: {
        enabled: true,
        apiKey: "BRAVE_API_KEY_HERE",
        maxResults: 5,
      },
      fetch: {
        enabled: true,
      },
    },
  },
}
```

注意:

* アロウリストを使用している場合は、`web_search`/`web_fetch`または`group:web`を追加してください。
* `web_fetch`はデフォルトで有効です（明示的に無効にされていない限り）。
* デーモンは`~/.openclaw/.env`（またはサービス環境）から環境変数を読み取ります。

ドキュメント: [ウェブツール](/tools/web)。

### 複数のデバイスで専門のワーカーを持つ中央ゲートウェイを実行するにはどうすればよいですか

一般的なパターンは**1つのゲートウェイ**（例: Raspberry Pi）と**ノード**および**エージェント**です:

* **ゲートウェイ（中央）:** チャンネル（Signal/WhatsApp）、ルーティング、セッションを所有します。
* **ノード（デバイス）:** Mac/iOS/Androidが周辺機器として接続し、ローカルツール（`system.run`、`canvas`、`camera`）を公開します。
* **エージェント（ワーカー）:** 特殊な役割のための別々の脳/ワークスペース（例: "Hetzner ops"、"Personal data"）。
* **サブエージェント:** メインエージェントからバックグラウンド作業を生成し、並列処理を行います。
* **TUI:** ゲートウェイに接続し、エージェント/セッションを切り替えます。

ドキュメント: [ノード](/nodes)、[リモートアクセス](/gateway/remote)、[マルチエージェントルーティング](/concepts/multi-agent)、[サブエージェント](/tools/subagents)、[TUI](/web/tui)。

### OpenClawブラウザはヘッドレスで実行できますか

はい。これは設定オプションです:

```json5
{
  browser: { headless: true },
  agents: {
    defaults: {
      sandbox: { browser: { headless: true } },
    },
  },
}
```

デフォルトは`false`（ヘッドフル）です。ヘッドレスは、一部のサイトでボット対策チェックを引き起こす可能性が高くなります。詳細は[ブラウザ](/tools/browser)を参照してください。

ヘッドレスは**同じChromiumエンジン**を使用し、ほとんどの自動化（フォーム、クリック、スクレイピング、ログイン）に対応します。主な違いは以下の通りです:

* 可視のブラウザウィンドウはありません（視覚が必要な場合はスクリーンショットを使用してください）。
* 一部のサイトはヘッドレスモードでの自動化に対して厳格です（CAPTCHA、ボット対策）。
  例えば、X/Twitterはしばしばヘッドレスセッションをブロックします。

### ブラウザ制御にBraveを使用するにはどうすればよいですか

`browser.executablePath`をBraveバイナリ（または任意のChromiumベースのブラウザ）に設定し、ゲートウェイを再起動します。
Braveまたは他のChromiumベースのブラウザを使用するための完全な設定例は、[ブラウザ](/tools/browser#use-brave-or-another-chromium-based-browser)を参照してください。

## リモートゲートウェイとノード

### コマンドはどのようにTelegram、ゲートウェイ、ノード間で伝播しますか

Telegramメッセージは**ゲートウェイ**によって処理されます。ゲートウェイはエージェントを実行し、ノードツールが必要なときにのみ**ゲートウェイWebSocket**を介してノードを呼び出します:

Telegram → ゲートウェイ → エージェント → `node.*` → ノード → ゲートウェイ → Telegram

ノードは受信プロバイダトラフィックを見ません; ノードRPC呼び出しのみを受け取ります。

### エージェントがリモートでホストされている場合、どのようにしてコンピュータにアクセスできますか

短い答え: **コンピュータをノードとしてペアリングします**。ゲートウェイは他の場所で実行されますが、ゲートウェイWebSocketを介してローカルマシン上の`node.*`ツール（画面、カメラ、システム）を呼び出すことができます。

典型的なセットアップ:

1. 常時稼働しているホスト（VPS/ホームサーバー）でゲートウェイを実行します。
2. ゲートウェイホストとコンピュータを同じtailnetに置きます。
3. ゲートウェイWSが到達可能であることを確認します（tailnetバインドまたはSSHトンネル）。
4. macOSアプリをローカルで開き、**SSH経由のリモート**モード（または直接tailnet）で接続して、ノードとして登録できるようにします。
5. ゲートウェイでノードを承認します:

   ```bash
   openclaw nodes pending
   openclaw nodes approve <requestId>
   ```

別途TCPブリッジは必要ありません; ノードはゲートウェイWebSocketを介して接続します。

セキュリティの注意: macOSノードをペアリングすると、そのマシンで`system.run`が許可されます。信頼できるデバイスのみをペアリングし、[セキュリティ](/gateway/security)を確認してください。

ドキュメント: [ノード](/nodes)、[ゲートウェイプロトコル](/gateway/protocol)、[macOSリモートモード](/platforms/mac/remote)、[セキュリティ](/gateway/security)。

### Tailscaleが接続されていますが、応答がありません。どうすればよいですか

基本を確認してください:

* ゲートウェイが実行中: `openclaw gateway status`
* ゲートウェイの健康状態: `openclaw status`
* チャンネルの健康状態: `openclaw channels status`

次に、認証とルーティングを確認します:

* Tailscale Serveを使用している場合は、`gateway.auth.allowTailscale`が正しく設定されていることを確認してください。
* SSHトンネル経由で接続している場合は、ローカルトンネルが稼働しており、正しいポートを指していることを確認してください。
* あなたのアロウリスト（DMまたはグループ）があなたのアカウントを含んでいることを確認してください。

ドキュメント: [Tailscale](/gateway/tailscale)、[リモートアクセス](/gateway/remote)、[チャンネル](/channels)。

### 2つのOpenClawインスタンスはローカルVPSで互いに通信できますか

はい。組み込みの「ボット間」ブリッジはありませんが、いくつかの信頼できる方法で接続できます:

**最も簡単な方法:** 両方のボットがアクセスできる通常のチャットチャンネル（Telegram/Slack/WhatsApp）を使用します。ボットAがボットBにメッセージを送信し、ボットBが通常通りに返信します。

**CLIブリッジ（汎用）:** 他のゲートウェイを呼び出すスクリプトを実行し、`openclaw agent --message ... --deliver`を使用して、他のボットがリッスンしているチャットをターゲットにします。一方のボットがリモートVPS上にある場合は、そのリモートゲートウェイにSSH/Tailscale経由でCLIを向けます（[リモートアクセス](/gateway/remote)を参照）。

例パターン（ターゲットゲートウェイに到達できるマシンから実行）:

```bash
openclaw agent --message "Hello from local bot" --deliver --channel telegram --reply-to <chat-id>
```

ヒント: 2つのボットが無限にループしないようにガードレールを追加してください（メンションのみ、チャンネルアロウリスト、または「ボットメッセージには返信しない」ルール）。

ドキュメント: [リモートアクセス](/gateway/remote)、[エージェントCLI](/cli/agent)、[エージェント送信](/tools/agent-send)。

### 複数のエージェントのために別々のVPSが必要ですか

いいえ。1つのゲートウェイが複数のエージェントをホストでき、それぞれが独自のワークスペース、モデルデフォルト、およびルーティングを持っています。これは通常のセットアップであり、エージェントごとに1つのVPSを実行するよりもはるかに安価で簡単です。

厳格な隔離（セキュリティ境界）や共有したくない非常に異なる設定が必要な場合のみ、別々のVPSを使用してください。それ以外の場合は、1つのゲートウェイを維持し、複数のエージェントまたはサブエージェントを使用してください。

### VPSからSSHする代わりに、個人のラップトップでノードを使用する利点はありますか

はい - ノードはリモートゲートウェイからラップトップに到達するための第一級の方法であり、シェルアクセス以上のものを解放します。ゲートウェイはmacOS/Linux（WindowsはWSL2経由）で実行され、軽量です（小さなVPSまたはRaspberry Piクラスのボックスで十分; 4GB RAMで十分です）、したがって一般的なセットアップは常時稼働のホストとラップトップをノードとして使用することです。

* **受信SSHは必要ありません。** ノードはゲートウェイWebSocketに接続し、デバイスペアリングを使用します。
* **安全な実行制御。** `system.run`は、そのラップトップ上のノードアロウリスト/承認によって制限されます。
* **より多くのデバイツール。** ノードは`system.run`に加えて`canvas`、`camera`、`screen`を公開します。
* **ローカルブラウザ自動化。** ゲートウェイをVPS上に維持し、Chromeをローカルで実行し、Chrome拡張機能とラップトップ上のノードホストを使用して制御を中継します。

SSHはアドホックなシェルアクセスには適していますが、ノードは継続的なエージェントワークフローやデバイス自動化にはより簡単です。

ドキュメント: [ノード](/nodes)、[ノードCLI](/cli/nodes)、[Chrome拡張機能](/tools/chrome-extension)。

### 2台目のラップトップにインストールするべきか、それともノードを追加するべきか

2台目のラップトップで**ローカルツール**（画面/カメラ/exec）だけが必要な場合は、**ノード**として追加してください。それにより、単一のゲートウェイを維持し、設定の重複を避けることができます。ローカルノードツールは現在macOS専用ですが、他のOSへの拡張を計画しています。

**厳格な隔離**や2つの完全に別々のボットが必要な場合のみ、2台目のゲートウェイをインストールしてください。

ドキュメント: [ノード](/nodes)、[ノードCLI](/cli/nodes)、[複数のゲートウェイ](/gateway/multiple-gateways)。

### ノードはゲートウェイサービスを実行しますか

いいえ。**1つのゲートウェイ**のみがホストごとに実行されるべきであり、意図的に隔離されたプロファイルを実行する場合を除きます（[複数のゲートウェイ](/gateway/multiple-gateways)を参照）。ノードはゲートウェイに接続する周辺機器です（iOS/Androidノード、またはメニューバーアプリのmacOS「ノードモード」）。ヘッドレスノードホストとCLI制御については、[ノードホストCLI](/cli/node)を参照してください。

`gateway`、`discovery`、および`canvasHost`の変更には完全な再起動が必要です。

### API RPCを使用して設定を適用する方法はありますか

はい。`config.apply`は、完全な設定を検証し+書き込み、操作の一部としてゲートウェイを再起動します。

### config.applyが設定を消去しました。どうやって回復し、これを避けることができますか

`config.apply`は**全体の設定**を置き換えます。部分的なオブジェクトを送信すると、他のすべてが削除されます。

回復方法:

* バックアップから復元します（gitまたはコピーした`~/.openclaw/openclaw.json`）。
* バックアップがない場合は、`openclaw doctor`を再実行し、チャンネル/モデルを再構成します。
* これが予期しないものであった場合は、バグを報告し、最後に知られている設定やバックアップを含めてください。
* ローカルコーディングエージェントは、ログや履歴から動作する設定を再構築できることがよくあります。

避ける方法:

* 小さな変更には`openclaw config set`を使用します。
* インタラクティブな編集には`openclaw configure`を使用します。

ドキュメント: [設定](/cli/config)、[設定](/cli/configure)、[Doctor](/gateway/doctor)。
### 最初のインストールにおける最小限の適切な設定は何ですか

```json5
{
  agents: { defaults: { workspace: "~/.openclaw/workspace" } },
  channels: { whatsapp: { allowFrom: ["+15555550123"] } },
}
```

これにより、ワークスペースが設定され、ボットをトリガーできる人が制限されます。

### VPSにTailscaleを設定し、Macから接続するにはどうすればよいですか

最小限の手順：

1. **VPSにインストールしてログイン**

   ```bash
   curl -fsSL https://tailscale.com/install.sh | sh
   sudo tailscale up
   ```

2. **Macにインストールしてログイン**
   * Tailscaleアプリを使用し、同じtailnetにサインインします。

3. **MagicDNSを有効にする（推奨）**
   * Tailscale管理コンソールでMagicDNSを有効にし、VPSに安定した名前を付けます。

4. **tailnetホスト名を使用**
   * SSH: `ssh user@your-vps.tailnet-xxxx.ts.net`
   * Gateway WS: `ws://your-vps.tailnet-xxxx.ts.net:18789`

SSHなしでControl UIを使用したい場合は、VPSでTailscale Serveを使用します：

```bash
openclaw gateway --tailscale serve
```

これにより、ゲートウェイがループバックにバインドされ、Tailscaleを介してHTTPSが公開されます。詳細は[Tailscale](/gateway/tailscale)を参照してください。

### MacノードをリモートGateway Tailscale Serveに接続するにはどうすればよいですか

Serveは**Gateway Control UI + WS**を公開します。ノードは同じGateway WSエンドポイントを介して接続します。

推奨設定：

1. **VPSとMacが同じtailnetにあることを確認します**。
2. **macOSアプリをリモートモードで使用**（SSHターゲットはtailnetホスト名にできます）。
   アプリはGatewayポートをトンネルし、ノードとして接続します。
3. **ゲートウェイでノードを承認します**：

   ```bash
   openclaw nodes pending
   openclaw nodes approve <requestId>
   ```

ドキュメント：[Gatewayプロトコル](/gateway/protocol)、[Discovery](/gateway/discovery)、[macOSリモートモード](/platforms/mac/remote)。

## 環境変数と.envの読み込み

### OpenClawは環境変数をどのように読み込みますか

OpenClawは親プロセス（シェル、launchd/systemd、CIなど）からenv varsを読み込み、さらに以下を読み込みます：

* 現在の作業ディレクトリからの`.env`
* グローバルフォールバック`.env`（`~/.openclaw/.env`、別名`$OPENCLAW_STATE_DIR/.env`）

どちらの`.env`ファイルも既存のenv varsを上書きしません。

プロセスenvに存在しない場合にのみ、設定内でインラインenv varsを定義することもできます：

```json5
{
  env: {
    OPENROUTER_API_KEY: "sk-or-...",
    vars: { GROQ_API_KEY: "gsk-..." },
  },
}
```

完全な優先順位とソースについては[/environment](/help/environment)を参照してください。

### サービス経由でGatewayを起動したら環境変数が消えました。どうすればよいですか

一般的な修正方法は2つあります：

1. サービスがシェルenvを継承しない場合でも、`~/.openclaw/.env`に欠落しているキーを入れておきます。
2. シェルインポートを有効にします（オプトインの便利機能）：

```json5
{
  env: {
    shellEnv: {
      enabled: true,
      timeoutMs: 15000,
    },
  },
}
```

これにより、ログインシェルが実行され、欠落している期待されるキーのみがインポートされます（上書きはしません）。環境変数の同等物：
`OPENCLAW_LOAD_SHELL_ENV=1`、`OPENCLAW_SHELL_ENV_TIMEOUT_MS=15000`。

### COPILOTGITHUBTOKENを設定したのに、モデルのステータスがShell envオフと表示されるのはなぜですか

`openclaw models status`は**シェルenvインポート**が有効かどうかを報告します。「Shell env: off」は、あなたのenv varsが欠落していることを意味するのではなく、OpenClawが自動的にログインシェルを読み込まないことを意味します。

Gatewayがサービス（launchd/systemd）として実行されている場合、シェル環境を継承しません。以下のいずれかを行って修正します：

1. トークンを`~/.openclaw/.env`に入れます：

   ```
   COPILOT_GITHUB_TOKEN=...
   ```

2. またはシェルインポートを有効にします（`env.shellEnv.enabled: true`）。

3. または設定の`env`ブロックに追加します（欠落している場合のみ適用されます）。

その後、ゲートウェイを再起動し、再確認します：

```bash
openclaw models status
```

Copilotトークンは`COPILOT_GITHUB_TOKEN`（または`GH_TOKEN` / `GITHUB_TOKEN`）から読み取られます。
詳細は[/concepts/model-providers](/concepts/model-providers)および[/environment](/help/environment)を参照してください。

## セッションと複数のチャット

### 新しい会話を始めるにはどうすればよいですか

`/new`または`/reset`をスタンドアロンメッセージとして送信します。詳細は[セッション管理](/concepts/session)を参照してください。

### 新しいメッセージを送信しない場合、セッションは自動的にリセットされますか

はい。セッションは`session.idleMinutes`（デフォルトは**60**）の後に期限切れになります。**次の**メッセージは、そのチャットキーの新しいセッションIDを開始します。これはトランスクリプトを削除するものではなく、新しいセッションを開始するだけです。

```json5
{
  session: {
    idleMinutes: 240,
  },
}
```

### OpenClawのインスタンスのチームを1人のCEOと多数のエージェントにする方法はありますか

はい、**マルチエージェントルーティング**と**サブエージェント**を介して可能です。1つのコーディネーターエージェントと、それぞれ独自のワークスペースとモデルを持ついくつかのワーカーエージェントを作成できます。

とはいえ、これは**楽しい実験**として見るのが最適です。トークンが多く、通常は1つのボットを使用して別々のセッションを持つよりも効率が悪いことが多いです。私たちが想定する典型的なモデルは、話しかけるボットが1つあり、並行作業のための異なるセッションがあります。そのボットは必要に応じてサブエージェントを生成することもできます。

ドキュメント：[マルチエージェントルーティング](/concepts/multi-agent)、[サブエージェント](/tools/subagents)、[エージェントCLI](/cli/agents)。

### タスクの途中でコンテキストが切り捨てられたのはなぜですか？どうすれば防げますか

セッションコンテキストはモデルウィンドウによって制限されています。長いチャット、大きなツール出力、または多くのファイルが圧縮または切り捨てを引き起こす可能性があります。

役立つこと：

* ボットに現在の状態を要約し、それをファイルに書き込むように依頼します。
* 長いタスクの前に`/compact`を使用し、トピックを切り替えるときに`/new`を使用します。
* 重要なコンテキストをワークスペースに保持し、ボットにそれを読み返すように依頼します。
* 長いまたは並行作業のためにサブエージェントを使用し、メインチャットを小さく保ちます。
* これが頻繁に発生する場合は、より大きなコンテキストウィンドウを持つモデルを選択します。

### OpenClawを完全にリセットするにはどうすればよいですが、インストールは維持しますか

リセットコマンドを使用します：

```bash
openclaw reset
```

非対話型の完全リセット：

```bash
openclaw reset --scope full --yes --non-interactive
```

その後、オンボーディングを再実行します：

```bash
openclaw onboard --install-daemon
```

注意：

* オンボーディングウィザードは、既存の設定を検出した場合に**リセット**も提供します。詳細は[ウィザード](/start/wizard)を参照してください。
* プロファイルを使用している場合（`--profile` / `OPENCLAW_PROFILE`）、各状態ディレクトリをリセットします（デフォルトは`~/.openclaw-<profile>`）。
* 開発リセット：`openclaw gateway --dev --reset`（開発専用；開発設定 + 認証情報 + セッション + ワークスペースを消去します）。

### コンテキストが大きすぎるエラーが発生しています。リセットまたは圧縮するにはどうすればよいですか

以下のいずれかを使用します：

* **圧縮**（会話を保持しますが、古いターンを要約します）：

  ```
  /compact
  ```

  または`/compact <instructions>`で要約をガイドします。

* **リセット**（同じチャットキーの新しいセッションID）：

  ```
  /new
  /reset
  ```

これが続く場合：

* **セッションプルーニング**（`agents.defaults.contextPruning`）を有効にするか調整して、古いツール出力をトリミングします。
* より大きなコンテキストウィンドウを持つモデルを使用します。

ドキュメント：[圧縮](/concepts/compaction)、[セッションプルーニング](/concepts/session-pruning)、[セッション管理](/concepts/session)。

### "LLMリクエストが拒否されました：messages.content.tool\_use.inputフィールドが必要です"と表示されるのはなぜですか？

これはプロバイダの検証エラーです：モデルが必要な`input`なしに`tool_use`ブロックを発行しました。これは通常、セッション履歴が古くなっているか破損していることを意味します（しばしば長いスレッドやツール/スキーマの変更後）。

修正：`/new`（スタンドアロンメッセージ）で新しいセッションを開始します。

### 30分ごとにハートビートメッセージが送信されるのはなぜですか

ハートビートはデフォルトで**30分**ごとに実行されます。調整または無効にします：

```json5
{
  agents: {
    defaults: {
      heartbeat: {
        every: "2h", // または無効にするには "0m"
      },
    },
  },
}
```

`HEARTBEAT.md`が存在するが実質的に空（空白行や`# 見出し`のようなマークダウンヘッダーのみ）である場合、OpenClawはAPI呼び出しを節約するためにハートビートの実行をスキップします。ファイルが欠落している場合、ハートビートはまだ実行され、モデルが何をするかを決定します。

エージェントごとのオーバーライドは`agents.list[].heartbeat`を使用します。ドキュメント：[ハートビート](/gateway/heartbeat)。

### WhatsAppグループにボットアカウントを追加する必要がありますか

いいえ。OpenClawは**あなた自身のアカウント**で実行されるため、グループにいる場合、OpenClawはそれを見ることができます。デフォルトでは、送信者を許可するまでグループの返信はブロックされます（`groupPolicy: "allowlist"`）。

グループの返信を**あなたのみ**がトリガーできるようにしたい場合：

```json5
{
  channels: {
    whatsapp: {
      groupPolicy: "allowlist",
      groupAllowFrom: ["+15551234567"],
    },
  },
}
```

### WhatsAppグループのJIDを取得するにはどうすればよいですか

オプション1（最速）：ログを追跡し、グループにテストメッセージを送信します：

```bash
openclaw logs --follow --json
```

`@g.us`で終わる`chatId`（または`from`）を探します。例：
`1234567890-1234567890@g.us`。

オプション2（すでに設定済み/許可リストにある場合）：設定からグループをリストします：

```bash
openclaw directory groups list --channel whatsapp
```

ドキュメント：[WhatsApp](/channels/whatsapp)、[ディレクトリ](/cli/directory)、[ログ](/cli/logs)。

### OpenClawがグループで返信しないのはなぜですか

一般的な原因は2つあります：

* メンションゲーティングがオン（デフォルト）。ボットを@メンションする必要があります（または`mentionPatterns`に一致させる必要があります）。
* `channels.whatsapp.groups`を`"*"`なしで設定し、グループが許可リストに含まれていません。

詳細は[グループ](/channels/groups)および[グループメッセージ](/channels/group-messages)を参照してください。

### グループ/スレッドはDMとコンテキストを共有しますか

直接チャットはデフォルトでメインセッションに統合されます。グループ/チャンネルはそれぞれ独自のセッションキーを持ち、TelegramのトピックやDiscordのスレッドは別のセッションです。詳細は[グループ](/channels/groups)および[グループメッセージ](/channels/group-messages)を参照してください。

### いくつのワークスペースとエージェントを作成できますか

制限はありません。数十（さらには数百）でも問題ありませんが、以下に注意してください：

* **ディスクの成長：** セッション + トランスクリプトは`~/.openclaw/agents/<agentId>/sessions/`に保存されます。
* **トークンコスト：** より多くのエージェントは、同時にモデルを使用することを意味します。
* **運用オーバーヘッド：** エージェントごとの認証プロファイル、ワークスペース、チャネルルーティング。

ヒント：

* 各エージェントに対して1つの**アクティブ**ワークスペースを保持します（`agents.defaults.workspace`）。
* ディスクが増加した場合は古いセッションをプルーニングします（JSONLを削除するか、エントリを保存します）。
* `openclaw doctor`を使用して、不要なワークスペースやプロファイルの不一致を特定します。

### 複数のボットやチャットを同時に実行できますか？Slackでそれをどのように設定すればよいですか

はい。**マルチエージェントルーティング**を使用して、複数の孤立したエージェントを実行し、チャネル/アカウント/ピアによって受信メッセージをルーティングします。Slackはチャネルとしてサポートされており、特定のエージェントにバインドできます。

ブラウザアクセスは強力ですが、「人間ができることを何でもする」わけではありません - アンチボット、CAPTCHA、MFAは自動化をブロックする可能性があります。最も信頼性の高いブラウザ制御を実現するには、ブラウザを実行するマシン上でChrome拡張リレーを使用し、Gatewayはどこにでも配置します。

ベストプラクティスの設定：

* 常時稼働のGatewayホスト（VPS/Mac mini）。
* 役割ごとに1つのエージェント（バインディング）。
* それらのエージェントにバインドされたSlackチャネル。
* 必要に応じて拡張リレー（またはノード）を介してローカルブラウザを使用します。

ドキュメント：[マルチエージェントルーティング](/concepts/multi-agent)、[Slack](/channels/slack)、[ブラウザ](/tools/browser)、[Chrome拡張](/tools/chrome-extension)、[ノード](/nodes)。

## モデル：デフォルト、選択、エイリアス、切り替え

### デフォルトモデルは何ですか

OpenClawのデフォルトモデルは、あなたが設定したもので：

```
agents.defaults.model.primary
```

モデルは`provider/model`として参照されます（例：`anthropic/claude-opus-4-6`）。プロバイダを省略すると、OpenClawは現在`anthropic`を一時的な非推奨のフォールバックとして仮定しますが、**明示的に**`provider/model`を設定するべきです。

### どのモデルを推奨しますか

**推奨デフォルト：** `anthropic/claude-opus-4-6`。
**良い代替：** `anthropic/claude-sonnet-4-5`。
**信頼性が高い（キャラクターが少ない）：** `openai/gpt-5.2` - Opusとほぼ同じですが、個性が少ないです。
**予算向け：** `zai/glm-4.7`。

MiniMax M2.1には独自のドキュメントがあります：[MiniMax](/providers/minimax)および[ローカルモデル](/gateway/local-models)。

一般的なルール：高リスクの作業には**最良のモデルを使用**し、日常的なチャットや要約には安価なモデルを使用します。エージェントごとにモデルをルーティングし、サブエージェントを使用して長いタスクを並行化できます（各サブエージェントはトークンを消費します）。詳細は[モデル](/concepts/models)および[サブエージェント](/tools/subagents)を参照してください。

強い警告：弱い/過量化されたモデルは、プロンプトインジェクションや安全でない動作に対してより脆弱です。詳細は[セキュリティ](/gateway/security)を参照してください。

さらなるコンテキスト：[モデル](/concepts/models)。

### 自己ホストされたモデル（llamacpp、vLLM、Ollama）を使用できますか

はい。ローカルサーバーがOpenAI互換のAPIを公開している場合、カスタムプロバイダをそれにポイントできます。Ollamaは直接サポートされており、最も簡単な方法です。

セキュリティノート：小型または過度に量子化されたモデルは、プロンプトインジェクションに対してより脆弱です。ツールを使用できるボットには**大きなモデル**を強く推奨します。小型モデルを使用したい場合は、サンドボックス化と厳格なツール許可リストを有効にしてください。

ドキュメント：[Ollama](/providers/ollama)、[ローカルモデル](/gateway/local-models)、[モデルプロバイダ](/concepts/model-providers)、[セキュリティ](/gateway/security)、[サンドボックス化](/gateway/sandboxing)。
### 設定を消さずにモデルを切り替えるにはどうすればよいですか

**モデルコマンド**を使用するか、**モデル**フィールドのみを編集してください。完全な設定の置き換えは避けてください。

安全なオプション:

* チャットでの`/model`（迅速、セッションごと）
* `openclaw models set ...`（モデル設定のみを更新）
* `openclaw configure --section model`（対話式）
* `~/.openclaw/openclaw.json`内の`agents.defaults.model`を編集

全体の設定を置き換えるつもりがない限り、部分オブジェクトで`config.apply`を避けてください。
設定を上書きしてしまった場合は、バックアップから復元するか、`openclaw doctor`を再実行して修復してください。

ドキュメント: [Models](/concepts/models), [Configure](/cli/configure), [Config](/cli/config), [Doctor](/gateway/doctor).

### OpenClaw、Flawd、およびKrillはモデルに何を使用していますか

* **OpenClaw + Flawd:** Anthropic Opus (`anthropic/claude-opus-4-6`) - [Anthropic](/providers/anthropic)を参照。
* **Krill:** MiniMax M2.1 (`minimax/MiniMax-M2.1`) - [MiniMax](/providers/minimax)を参照。

### 再起動せずにモデルを切り替えるにはどうすればよいですか

スタンドアロンメッセージとして`/model`コマンドを使用します:

```
/model sonnet
/model haiku
/model opus
/model gpt
/model gpt-mini
/model gemini
/model gemini-flash
```

利用可能なモデルをリストするには`/model`、`/model list`、または`/model status`を使用できます。

`/model`（および`/model list`）はコンパクトな番号付きピッカーを表示します。番号で選択します:

```
/model 3
```

プロバイダーの特定の認証プロファイルを強制することもできます（セッションごと）:

```
/model opus@anthropic:default
/model opus@anthropic:work
```

ヒント: `/model status`は、どのエージェントがアクティブであるか、どの`auth-profiles.json`ファイルが使用されているか、次に試みられる認証プロファイルを表示します。
また、利用可能な場合は設定されたプロバイダーエンドポイント（`baseUrl`）とAPIモード（`api`）も表示します。

**profileで設定したプロファイルのピン留めを解除するにはどうすればよいですか**

`@profile`サフィックスなしで`/model`を再実行します:

```
/model anthropic/claude-opus-4-6
```

デフォルトに戻したい場合は、`/model`から選択するか、`/model <default provider/model>`を送信します。
どの認証プロファイルがアクティブであるかを確認するには`/model status`を使用します。

### 日常業務にGPT 5.2を、コーディングにCodex 5.3を使用できますか

はい。1つをデフォルトとして設定し、必要に応じて切り替えます:

* **クイックスイッチ（セッションごと）:** 日常業務には`/model gpt-5.2`、コーディングには`/model gpt-5.3-codex`を使用します。
* **デフォルト + スイッチ:** `agents.defaults.model.primary`を`openai/gpt-5.2`に設定し、コーディング時に`openai-codex/gpt-5.3-codex`に切り替えます（またはその逆）。
* **サブエージェント:** コーディングタスクを異なるデフォルトモデルを持つサブエージェントにルーティングします。

[Models](/concepts/models)と[Slash commands](/tools/slash-commands)を参照してください。

### "Model is not allowed"と表示され、その後返信がないのはなぜですか

`agents.defaults.models`が設定されている場合、それは`/model`およびセッションオーバーライドの**許可リスト**になります。そのリストにないモデルを選択すると、次のようになります:

```
Model "provider/model" is not allowed. Use /model to list available models.
```

そのエラーは**通常の返信の代わりに**返されます。修正: モデルを`agents.defaults.models`に追加するか、許可リストを削除するか、`/model list`からモデルを選択してください。

### "Unknown model minimaxMiniMaxM21"が表示されるのはなぜですか

これは、**プロバイダーが構成されていない**ことを意味します（MiniMaxプロバイダーの構成または認証プロファイルが見つからないため）、したがってモデルを解決できません。この検出の修正は**2026.1.12**にあります（執筆時点では未リリース）。

修正チェックリスト:

1. **2026.1.12**にアップグレードする（またはソース`main`から実行）し、その後ゲートウェイを再起動します。
2. MiniMaxが構成されていることを確認します（ウィザードまたはJSON）、またはMiniMax APIキーがenv/authプロファイルに存在することを確認し、プロバイダーを注入できるようにします。
3. 正確なモデルID（大文字と小文字を区別）を使用します: `minimax/MiniMax-M2.1`または`minimax/MiniMax-M2.1-lightning`。
4. 次を実行します:

   ```bash
   openclaw models list
   ```

   そしてリストから選択します（またはチャットで`/model list`を使用します）。

[MiniMax](/providers/minimax)と[Models](/concepts/models)を参照してください。

### MiniMaxをデフォルトとして使用し、複雑なタスクにOpenAIを使用できますか

はい。**MiniMaxをデフォルトとして使用**し、必要に応じて**セッションごとにモデルを切り替え**ます。
フォールバックは**エラー**用であり、「ハードタスク」用ではないため、`/model`または別のエージェントを使用してください。

**オプションA: セッションごとに切り替え**

```json5
{
  env: { MINIMAX_API_KEY: "sk-...", OPENAI_API_KEY: "sk-..." },
  agents: {
    defaults: {
      model: { primary: "minimax/MiniMax-M2.1" },
      models: {
        "minimax/MiniMax-M2.1": { alias: "minimax" },
        "openai/gpt-5.2": { alias: "gpt" },
      },
    },
  },
}
```

その後:

```
/model gpt
```

**オプションB: 別々のエージェント**

* エージェントAのデフォルト: MiniMax
* エージェントBのデフォルト: OpenAI
* エージェントによってルーティングするか、`/agent`を使用して切り替えます。

ドキュメント: [Models](/concepts/models), [Multi-Agent Routing](/concepts/multi-agent), [MiniMax](/providers/minimax), [OpenAI](/providers/openai)。

### opus、sonnet、gptは組み込みショートカットですか

はい。OpenClawは、いくつかのデフォルトのショートハンドを提供します（モデルが`agents.defaults.models`に存在する場合のみ適用されます）:

* `opus` → `anthropic/claude-opus-4-6`
* `sonnet` → `anthropic/claude-sonnet-4-5`
* `gpt` → `openai/gpt-5.2`
* `gpt-mini` → `openai/gpt-5-mini`
* `gemini` → `google/gemini-3-pro-preview`
* `gemini-flash` → `google/gemini-3-flash-preview`

同じ名前で独自のエイリアスを設定した場合、あなたの値が優先されます。

### モデルショートカットのエイリアスを定義または上書きするにはどうすればよいですか

エイリアスは`agents.defaults.models.<modelId>.alias`から取得されます。例:

```json5
{
  agents: {
    defaults: {
      model: { primary: "anthropic/claude-opus-4-6" },
      models: {
        "anthropic/claude-opus-4-6": { alias: "opus" },
        "anthropic/claude-sonnet-4-5": { alias: "sonnet" },
        "anthropic/claude-haiku-4-5": { alias: "haiku" },
      },
    },
  },
}
```

その後、`/model sonnet`（またはサポートされている場合は`/<alias>`）はそのモデルIDに解決されます。

### OpenRouterやZAIのような他のプロバイダーからモデルを追加するにはどうすればよいですか

OpenRouter（トークンごとの支払い; 多くのモデル）:

```json5
{
  agents: {
    defaults: {
      model: { primary: "openrouter/anthropic/claude-sonnet-4-5" },
      models: { "openrouter/anthropic/claude-sonnet-4-5": {} },
    },
  },
  env: { OPENROUTER_API_KEY: "sk-or-..." },
}
```

Z.AI（GLMモデル）:

```json5
{
  agents: {
    defaults: {
      model: { primary: "zai/glm-4.7" },
      models: { "zai/glm-4.7": {} },
    },
  },
  env: { ZAI_API_KEY: "..." },
}
```

プロバイダー/モデルを参照しますが、必要なプロバイダーキーが欠落している場合、ランタイム認証エラー（例: `No API key found for provider "zai"`）が発生します。

**新しいエージェントを追加した後に「プロバイダーのAPIキーが見つかりません」と表示される**

これは通常、**新しいエージェント**に空の認証ストアがあることを意味します。認証はエージェントごとに行われ、次の場所に保存されます:

```
~/.openclaw/agents/<agentId>/agent/auth-profiles.json
```

修正オプション:

* `openclaw agents add <id>`を実行し、ウィザード中に認証を設定します。
* または、メインエージェントの`agentDir`から新しいエージェントの`agentDir`に`auth-profiles.json`をコピーします。

エージェント間で`agentDir`を再利用しないでください。認証/セッションの衝突を引き起こします。

## モデルフェイルオーバーと「すべてのモデルが失敗しました」

### フェイルオーバーはどのように機能しますか

フェイルオーバーは2つの段階で行われます:

1. 同じプロバイダー内での**認証プロファイルのローテーション**。
2. `agents.defaults.model.fallbacks`内の次のモデルへの**モデルフォールバック**。

クールダウンは失敗したプロファイルに適用され（指数バックオフ）、OpenClawはプロバイダーがレート制限されている場合や一時的に失敗している場合でも応答し続けることができます。

### このエラーは何を意味しますか

```
No credentials found for profile "anthropic:default"
```

これは、システムが認証プロファイルID`anthropic:default`を使用しようとしましたが、期待される認証ストアでその資格情報を見つけられなかったことを意味します。

### "No credentials found for profile anthropicdefault"の修正チェックリスト

* **認証プロファイルの場所を確認する**（新しいパスとレガシーパス）
  * 現在: `~/.openclaw/agents/<agentId>/agent/auth-profiles.json`
  * レガシー: `~/.openclaw/agent/*`（`openclaw doctor`によって移行されました）
* **環境変数がゲートウェイによって読み込まれていることを確認する**
  * シェルで`ANTHROPIC_API_KEY`を設定したが、systemd/launchd経由でゲートウェイを実行している場合、それを継承しない可能性があります。`~/.openclaw/.env`に置くか、`env.shellEnv`を有効にします。
* **正しいエージェントを編集していることを確認する**
  * マルチエージェントセットアップでは、複数の`auth-profiles.json`ファイルが存在する可能性があります。
* **モデル/認証の状態を確認する**
  * `openclaw models status`を使用して、設定されたモデルとプロバイダーが認証されているかどうかを確認します。

**"No credentials found for profile anthropic"の修正チェックリスト**

これは、実行がAnthropic認証プロファイルにピン留めされているが、ゲートウェイがその認証ストアでそれを見つけられないことを意味します。

* **セットアップトークンを使用する**
  * `claude setup-token`を実行し、その後`openclaw models auth setup-token --provider anthropic`で貼り付けます。
  * トークンが別のマシンで作成された場合は、`openclaw models auth paste-token --provider anthropic`を使用します。

* **APIキーを代わりに使用したい場合**
  * `ANTHROPIC_API_KEY`を**ゲートウェイホスト**の`~/.openclaw/.env`に置きます。
  * 欠落したプロファイルを強制するピン留めされた順序をクリアします:

    ```bash
    openclaw models auth order clear --provider anthropic
    ```

* **ゲートウェイホストでコマンドを実行していることを確認する**
  * リモートモードでは、認証プロファイルはゲートウェイマシンに存在し、あなたのラップトップには存在しません。

### なぜGoogle Geminiを試みて失敗したのですか

モデル設定にGoogle Geminiがフォールバックとして含まれている場合（またはGeminiショートハンドに切り替えた場合）、OpenClawはモデルフォールバック中にそれを試みます。Googleの資格情報が構成されていない場合、`No API key found for provider "google"`が表示されます。

修正: Googleの認証を提供するか、`agents.defaults.model.fallbacks` / エイリアス内のGoogleモデルを削除/回避して、フォールバックがそこにルーティングされないようにします。

**"LLM request rejected message thinking signature required google antigravity"**

原因: セッション履歴に**署名のない思考ブロック**が含まれています（中断された/部分的なストリームからのことが多いです）。Google Antigravityは思考ブロックに署名を必要とします。

修正: OpenClawは、Google Antigravity Claude用に署名のない思考ブロックを削除します。それでも表示される場合は、**新しいセッション**を開始するか、そのエージェントに対して`/thinking off`を設定します。

## 認証プロファイル: それらは何で、どのように管理するか

関連: [/concepts/oauth](/concepts/oauth)（OAuthフロー、トークンストレージ、マルチアカウントパターン）

### 認証プロファイルとは何ですか

認証プロファイルは、プロバイダーに関連付けられた名前付きの資格情報レコード（OAuthまたはAPIキー）です。プロファイルは次の場所に保存されます:

```
~/.openclaw/agents/<agentId>/agent/auth-profiles.json
```

### 一般的なプロファイルIDは何ですか

OpenClawは、次のようなプロバイダー接頭辞のIDを使用します:

* `anthropic:default`（メールアイデンティティが存在しない場合によく見られる）
* `anthropic:<email>`（OAuthアイデンティティ用）
* あなたが選択したカスタムID（例: `anthropic:work`）

### どの認証プロファイルが最初に試されるかを制御できますか

はい。設定は、プロファイルのオプションメタデータとプロバイダーごとの順序（`auth.order.<provider>`）をサポートしています。これは**秘密を保存しません**; IDをプロバイダー/モードにマッピングし、ローテーション順序を設定します。

OpenClawは、短い**クールダウン**（レート制限/タイムアウト/認証失敗）の場合、一時的にプロファイルをスキップすることがあります。また、長い**無効**状態（請求/クレジット不足）の場合もあります。これを確認するには、`openclaw models status --json`を実行し、`auth.unusableProfiles`を確認します。調整: `auth.cooldowns.billingBackoffHours*`。

また、CLIを介して**エージェントごとの**順序オーバーライドを設定することもできます（そのエージェントの`auth-profiles.json`に保存されます）:

```bash
# 設定されたデフォルトエージェントにデフォルト（--agentを省略）
openclaw models auth order get --provider anthropic

# 単一のプロファイルにロック（この1つだけを試す）
openclaw models auth order set --provider anthropic anthropic:default

# または明示的な順序を設定（プロバイダー内でフォールバック）
openclaw models auth order set --provider anthropic anthropic:work anthropic:default

# オーバーライドをクリア（設定されたauth.order / ラウンドロビンに戻る）
openclaw models auth order clear --provider anthropic
```

特定のエージェントをターゲットにするには:

```bash
openclaw models auth order set --provider anthropic --agent main anthropic:default
```

### OAuthとAPIキーの違いは何ですか

OpenClawは両方をサポートしています:

* **OAuth**は、適用可能な場合、サブスクリプションアクセスを活用します。
* **APIキー**は、トークンごとの課金を使用します。

ウィザードは、AnthropicセットアップトークンとOpenAI Codex OAuthを明示的にサポートし、APIキーを保存できます。
## Gateway: ポート、"すでに実行中"、およびリモートモード

### Gatewayはどのポートを使用しますか

`gateway.port`はWebSocket + HTTP（Control UI、フックなど）のための単一の多重化ポートを制御します。

優先順位:

```
--port > OPENCLAW_GATEWAY_PORT > gateway.port > default 18789
```

### なぜopenclaw gatewayのステータスはRuntime runningと言っているのにRPCプローブが失敗したのか

"running"は**スーパーバイザーの**視点です（launchd/systemd/schtasks）。RPCプローブはCLIが実際にGatewayのWebSocketに接続して`status`を呼び出しているものです。

`openclaw gateway status`を使用し、以下の行を信頼してください：

* `Probe target:`（プローブが実際に使用したURL）
* `Listening:`（ポートで実際にバインドされているもの）
* `Last gateway error:`（プロセスは生きているがポートがリスニングしていない場合の一般的な根本原因）

### なぜopenclaw gatewayのステータスはConfig cliとConfig serviceが異なるのか

あなたが一つの設定ファイルを編集している間に、サービスは別の設定ファイルを実行しています（しばしば`--profile` / `OPENCLAW_STATE_DIR`の不一致）。

修正:

```bash
openclaw gateway install --force
```

サービスが使用することを希望する同じ`--profile` / 環境から実行してください。

### もう一つのGatewayインスタンスがすでにリスニングしているとはどういう意味ですか

OpenClawは、起動時にWebSocketリスナーを即座にバインドすることでランタイムロックを強制します（デフォルトは`ws://127.0.0.1:18789`）。バインドが`EADDRINUSE`で失敗すると、別のインスタンスがすでにリスニングしていることを示す`GatewayLockError`がスローされます。

修正: 他のインスタンスを停止するか、ポートを解放するか、`openclaw gateway --port <port>`で実行します。

### OpenClawをリモートモードで実行するにはどうすればよいですか？クライアントが他の場所のGatewayに接続します。

`gateway.mode: "remote"`を設定し、リモートWebSocket URLを指し、オプションでトークン/パスワードを指定します：

```json5
{
  gateway: {
    mode: "remote",
    remote: {
      url: "ws://gateway.tailnet:18789",
      token: "your-token",
      password: "your-password",
    },
  },
}
```

注意:

* `openclaw gateway`は`gateway.mode`が`local`の時のみ開始されます（またはオーバーライドフラグを渡す場合）。
* macOSアプリは設定ファイルを監視し、これらの値が変更されるとライブでモードを切り替えます。

### Control UIが未認証または再接続を繰り返す場合、どうすればよいですか？

あなたのGatewayは認証が有効な状態で実行されています（`gateway.auth.*`）、しかしUIは一致するトークン/パスワードを送信していません。

事実（コードから）:

* Control UIはトークンをブラウザのlocalStorageキー`openclaw.control.settings.v1`に保存します。

修正:

* 最も早い方法: `openclaw dashboard`（ダッシュボードURLを印刷+コピーし、開こうとします; ヘッドレスの場合はSSHヒントを表示します）。
* まだトークンを持っていない場合: `openclaw doctor --generate-gateway-token`。
* リモートの場合、最初にトンネルを作成: `ssh -N -L 18789:127.0.0.1:18789 user@host` その後 `http://127.0.0.1:18789/`を開きます。
* Gatewayホストで`gateway.auth.token`（または`OPENCLAW_GATEWAY_TOKEN`）を設定します。
* Control UIの設定で同じトークンを貼り付けます。
* まだ行き詰まっていますか？ `openclaw status --all`を実行し、[トラブルシューティング](/gateway/troubleshooting)に従ってください。認証の詳細については[ダッシュボード](/web/dashboard)を参照してください。

### Windowsでターミナルを閉じた場合、OpenClawを再起動するにはどうすればよいですか？

**2つのWindowsインストールモード**があります：

**1) WSL2（推奨）:** GatewayはLinux内で実行されます。

PowerShellを開き、WSLに入り、次に再起動します：

```powershell
wsl
openclaw gateway status
openclaw gateway restart
```

サービスをインストールしていない場合は、フォアグラウンドで開始します：

```bash
openclaw gateway run
```

**2) ネイティブWindows（推奨されません）:** GatewayはWindowsで直接実行されます。

PowerShellを開いて実行します：

```powershell
openclaw gateway status
openclaw gateway restart
```

手動で実行している場合（サービスなし）、次のように使用します：

```powershell
openclaw gateway run
```

ドキュメント: [Windows (WSL2)](/platforms/windows)、[Gatewayサービスランブック](/gateway)。

### Gatewayは起動していますが、応答が届かない場合、何を確認すればよいですか？

まずは簡単なヘルススイープを行います：

```bash
openclaw status
openclaw models status
openclaw channels status
openclaw logs --follow
```

一般的な原因：

* モデル認証が**Gatewayホスト**で読み込まれていない（`models status`を確認）。
* チャンネルのペアリング/許可リストが応答をブロックしている（チャンネル設定 + ログを確認）。
* WebChat/Dashboardが正しいトークンなしで開いている。

リモートの場合は、トンネル/Tailscale接続が正常で、GatewayのWebSocketにアクセスできることを確認してください。

ドキュメント: [チャンネル](/channels)、[トラブルシューティング](/gateway/troubleshooting)、[リモートアクセス](/gateway/remote)。

### Gatewayから切断されました、理由は不明です。どうすればよいですか？

これは通常、UIがWebSocket接続を失ったことを意味します。確認すべきこと：

1. Gatewayは実行中ですか？ `openclaw gateway status`
2. Gatewayは正常ですか？ `openclaw status`
3. UIは正しいトークンを持っていますか？ `openclaw dashboard`
4. リモートの場合、トンネル/Tailscaleリンクは正常ですか？

次に、ログを追尾します：

```bash
openclaw logs --follow
```

ドキュメント: [ダッシュボード](/web/dashboard)、[リモートアクセス](/gateway/remote)、[トラブルシューティング](/gateway/troubleshooting)。

### TelegramのsetMyCommandsがネットワークエラーで失敗しました。何を確認すればよいですか？

まずはログとチャンネルのステータスを確認します：

```bash
openclaw channels status
openclaw channels logs --channel telegram
```

VPS上またはプロキシの背後にいる場合、アウトバウンドHTTPSが許可されていることとDNSが機能していることを確認してください。
Gatewayがリモートの場合、Gatewayホストのログを確認していることを確認してください。

ドキュメント: [Telegram](/channels/telegram)、[チャンルトラブルシューティング](/channels/troubleshooting)。

### TUIが出力を表示しません。何を確認すればよいですか？

まずGatewayにアクセスでき、エージェントが実行できることを確認します：

```bash
openclaw status
openclaw models status
openclaw logs --follow
```

TUI内で`/status`を使用して現在の状態を確認します。チャットチャンネルで応答を期待する場合、配信が有効になっていることを確認してください（`/deliver on`）。

ドキュメント: [TUI](/web/tui)、[スラッシュコマンド](/tools/slash-commands)。

### Gatewayを完全に停止してから再起動するにはどうすればよいですか？

サービスをインストールしている場合：

```bash
openclaw gateway stop
openclaw gateway start
```

これは**監視サービス**を停止/開始します（macOSのlaunchd、Linuxのsystemd）。
Gatewayがバックグラウンドでデーモンとして実行されているときに使用します。

フォアグラウンドで実行している場合は、Ctrl-Cで停止し、次に：

```bash
openclaw gateway run
```

ドキュメント: [Gatewayサービスランブック](/gateway)。

### ELI5 openclaw gateway restartとopenclaw gatewayの違いは？

* `openclaw gateway restart`: **バックグラウンドサービス**（launchd/systemd）を再起動します。
* `openclaw gateway`: このターミナルセッションのためにGatewayを**フォアグラウンド**で実行します。

サービスをインストールしている場合は、Gatewayコマンドを使用します。単発でフォアグラウンド実行したい場合は`openclaw gateway`を使用します。

### 何かが失敗したときに詳細を取得する最も早い方法は？

Gatewayを`--verbose`で起動して、コンソールの詳細を取得します。その後、チャンネル認証、モデルルーティング、およびRPCエラーのためにログファイルを確認します。
### テキストメッセージに対して自律性を持たせることはできますか？それは安全ですか？

私たちは、あなたの個人的なメッセージに対して完全な自律性を持たせることを**推奨しません**。最も安全なパターンは以下の通りです：

* DMを**ペアリングモード**または厳密な許可リストに保つ。
* 代理でメッセージを送信させたい場合は、**別の番号またはアカウント**を使用する。
* 下書きを作成させ、送信前に**承認する**。

実験を行いたい場合は、専用のアカウントで行い、隔離しておいてください。詳細は[セキュリティ](/gateway/security)を参照してください。

### 個人アシスタントタスクに安価なモデルを使用できますか？

はい、**もし**エージェントがチャット専用で、入力が信頼できるものであれば可能です。小規模なモデルは指示のハイジャックに対してより脆弱なので、ツールを使用するエージェントや信頼できないコンテンツを読む際には避けてください。小規模なモデルを使用する必要がある場合は、ツールを制限し、サンドボックス内で実行してください。詳細は[セキュリティ](/gateway/security)を参照してください。

### Telegramでstartを実行したが、ペアリングコードが得られなかった

ペアリングコードは、**未知の送信者**がボットにメッセージを送信し、`dmPolicy: "pairing"`が有効な場合に**のみ**送信されます。単独で`/start`を実行してもコードは生成されません。

保留中のリクエストを確認してください：

```bash
openclaw pairing list telegram
```

即時アクセスが必要な場合は、送信者IDを許可リストに追加するか、そのアカウントの`dmPolicy: "open"`を設定してください。

### WhatsAppは私の連絡先にメッセージを送信しますか？ペアリングはどのように機能しますか？

いいえ。デフォルトのWhatsApp DMポリシーは**ペアリング**です。未知の送信者はペアリングコードのみを受け取り、そのメッセージは**処理されません**。OpenClawは、受信したチャットや、あなたがトリガーした明示的な送信にのみ応答します。

ペアリングを承認するには：

```bash
openclaw pairing approve whatsapp <code>
```

保留中のリクエストをリスト表示するには：

```bash
openclaw pairing list whatsapp
```

ウィザードの電話番号プロンプト：これはあなたの**許可リスト/所有者**を設定するために使用され、あなた自身のDMが許可されます。自動送信には使用されません。個人のWhatsApp番号で実行する場合は、その番号を使用し、`channels.whatsapp.selfChatMode`を有効にしてください。

## チャットコマンド、タスクの中止、および「止まらない」

### チャットに内部システムメッセージが表示されないようにするにはどうすればよいですか？

ほとんどの内部またはツールメッセージは、そのセッションで**詳細表示**または**推論**が有効な場合にのみ表示されます。

表示されているチャットで修正してください：

```
/verbose off
/reasoning off
```

まだノイズが多い場合は、Control UIのセッション設定を確認し、詳細表示を**継承**に設定してください。また、`verboseDefault`が`on`に設定されたボットプロファイルを使用していないことを確認してください。

ドキュメント：[思考と詳細表示](/tools/thinking)、[セキュリティ](/gateway/security#reasoning--verbose-output-in-groups)。

### 実行中のタスクを停止またはキャンセルするにはどうすればよいですか？

以下のいずれかを**スタンドアロンメッセージ**として送信してください（スラッシュなし）：

```
stop
stop action
stop current action
stop run
stop current run
stop agent
stop the agent
stop openclaw
openclaw stop
stop don't do anything
stop do not do anything
stop doing anything
please stop
stop please
abort
esc
wait
exit
interrupt
```

これらは中止トリガーです（スラッシュコマンドではありません）。

バックグラウンドプロセス（execツールから）の場合、エージェントに実行を依頼できます：

```
process action:kill sessionId:XXX
```

スラッシュコマンドの概要：詳細は[スラッシュコマンド](/tools/slash-commands)を参照してください。

ほとんどのコマンドは、**スタンドアロン**メッセージとして送信する必要があり、`/`で始まりますが、一部のショートカット（例えば`/status`）は許可リストにある送信者に対してインラインでも機能します。

### TelegramからDiscordにメッセージを送信するにはどうすればよいですか？クロスコンテキストメッセージングは拒否されました

OpenClawはデフォルトで**プロバイダー間**のメッセージングをブロックします。ツールコールがTelegramにバインドされている場合、明示的に許可しない限り、Discordに送信されません。

エージェントのためにプロバイダー間メッセージングを有効にするには：

```json5
{
  agents: {
    defaults: {
      tools: {
        message: {
          crossContext: {
            allowAcrossProviders: true,
            marker: { enabled: true, prefix: "[from {channel}] " },
          },
        },
      },
    },
  },
}
```

設定を編集した後、ゲートウェイを再起動してください。特定のエージェントのみにこれを適用したい場合は、`agents.list[].tools.message`の下に設定してください。

### ボットがラピッドファイアメッセージを無視しているように感じるのはなぜですか？

キューモードは、新しいメッセージが進行中の実行とどのように相互作用するかを制御します。モードを変更するには`/queue`を使用してください：

* `steer` - 新しいメッセージが現在のタスクをリダイレクトします
* `followup` - メッセージを1つずつ実行します
* `collect` - メッセージをバッチ処理し、一度だけ応答します（デフォルト）
* `steer-backlog` - 現在のタスクを誘導し、次にバックログを処理します
* `interrupt` - 現在の実行を中止し、新たに開始します

フォローアップモードのために、`debounce:2s cap:25 drop:summarize`のようなオプションを追加できます。

## スクリーンショット/チャットログからの正確な質問に答える

**Q: "AnthropicのデフォルトモデルはAPIキーで何ですか？"**

**A:** OpenClawでは、認証情報とモデル選択は別々です。`ANTHROPIC_API_KEY`を設定する（または認証プロファイルにAnthropic APIキーを保存する）ことで認証が有効になりますが、実際のデフォルトモデルは`agents.defaults.model.primary`で設定したものです（例えば、`anthropic/claude-sonnet-4-5`や`anthropic/claude-opus-4-6`など）。`No credentials found for profile "anthropic:default"`と表示される場合、ゲートウェイは実行中のエージェントの期待される`auth-profiles.json`でAnthropicの認証情報を見つけられなかったことを意味します。
