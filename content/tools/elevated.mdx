# Elevated モード

# Elevated Mode (/elevated directives)

## 何をするか

* `/elevated on` はゲートウェイホストで実行され、execの承認を保持します（`/elevated ask` と同じ）。
* `/elevated full` はゲートウェイホストで実行され、**自動的に** execを承認します（execの承認をスキップします）。
* `/elevated ask` はゲートウェイホストで実行されますが、execの承認を保持します（`/elevated on` と同じ）。
* `on`/`ask` は `exec.security=full` を強制しません；設定されたセキュリティ/askポリシーが引き続き適用されます。
* エージェントが**サンドボックス化**されている場合のみ動作が変更されます（そうでない場合、execはすでにホスト上で実行されています）。
* ディレクティブ形式: `/elevated on|off|ask|full`, `/elev on|off|ask|full`。
* `on|off|ask|full` のみが受け入れられます；それ以外はヒントを返し、状態を変更しません。

## 何を制御するか（および何を制御しないか）

* **可用性ゲート**: `tools.elevated` はグローバルなベースラインです。 `agents.list[].tools.elevated` はエージェントごとにさらに制限できます（両方が許可する必要があります）。
* **セッションごとの状態**: `/elevated on|off|ask|full` は現在のセッションキーのためのelevatedレベルを設定します。
* **インラインディレクティブ**: メッセージ内の `/elevated on|ask|full` はそのメッセージのみに適用されます。
* **グループ**: グループチャットでは、elevatedディレクティブはエージェントが言及された場合にのみ尊重されます。言及要件をバイパスするコマンドのみのメッセージは、言及されたものとして扱われます。
* **ホスト実行**: elevatedは `exec` をゲートウェイホストに強制します；`full` は `security=full` も設定します。
* **承認**: `full` はexecの承認をスキップします；`on`/`ask` はallowlist/askルールが必要な場合にそれを尊重します。
* **サンドボックス化されていないエージェント**: 位置情報には影響しません；ゲーティング、ロギング、ステータスにのみ影響します。
* **ツールポリシーは引き続き適用される**: もし `exec` がツールポリシーによって拒否されると、elevatedは使用できません。
* **`/exec` とは別**: `/exec` は認可された送信者のためのセッションごとのデフォルトを調整し、elevatedを必要としません。

## 解決順序

1. メッセージ内のインラインディレクティブ（そのメッセージのみに適用されます）。
2. セッションオーバーライド（ディレクティブのみのメッセージを送信することで設定されます）。
3. グローバルデフォルト（設定内の `agents.defaults.elevatedDefault`）。

## セッションデフォルトの設定

* **ディレクティブのみ**のメッセージを送信します（空白は許可されます）、例: `/elevated full`。
* 確認の返信が送信されます（`Elevated mode set to full...` / `Elevated mode disabled.`）。
* elevatedアクセスが無効になっているか、送信者が承認されたallowlistにない場合、ディレクティブは実行可能なエラーで返信し、セッション状態は変更されません。
* 現在のelevatedレベルを確認するには、引数なしで `/elevated`（または `/elevated:`）を送信します。

## 可用性 + allowlists

* 機能ゲート: `tools.elevated.enabled`（デフォルトは、コードがサポートしていても設定によってオフにできます）。
* 送信者allowlist: `tools.elevated.allowFrom` とプロバイダーごとのallowlist（例: `discord`, `whatsapp`）。
* プレフィックスなしのallowlistエントリは、送信者スコープのアイデンティティ値のみと一致します（`SenderId`, `SenderE164`, `From`）；受信者のルーティングフィールドはelevated認可には使用されません。
* 変更可能な送信者メタデータには明示的なプレフィックスが必要です：
  * `name:<value>` は `SenderName` と一致します
  * `username:<value>` は `SenderUsername` と一致します
  * `tag:<value>` は `SenderTag` と一致します
  * `id:<value>`, `from:<value>`, `e164:<value>` は明示的なアイデンティティターゲティングのために利用可能です
* エージェントごとのゲート: `agents.list[].tools.elevated.enabled`（オプション；さらに制限することはできません）。
* エージェントごとのallowlist: `agents.list[].tools.elevated.allowFrom`（オプション；設定されている場合、送信者は**両方**のグローバル + エージェントごとのallowlistに一致する必要があります）。
* Discordフォールバック: `tools.elevated.allowFrom.discord` が省略されている場合、`channels.discord.allowFrom` リストがフォールバックとして使用されます（レガシー: `channels.discord.dm.allowFrom`）。 `tools.elevated.allowFrom.discord`（たとえ `[]` でも）を設定して上書きします。エージェントごとのallowlistは**フォールバック**を使用しません。
* すべてのゲートを通過する必要があります；そうでない場合、elevatedは利用不可として扱われます。

## ロギング + ステータス

* Elevated exec呼び出しは情報レベルでログに記録されます。
* セッションステータスにはelevatedモードが含まれます（例: `elevated=ask`, `elevated=full`）。
