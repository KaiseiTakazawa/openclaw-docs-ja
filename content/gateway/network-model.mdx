# ネットワークモデル

ほとんどの操作は、チャネル接続とWebSocket制御プレーンを所有する単一の長時間実行プロセスであるGateway（`openclaw gateway`）を通過します。

## コアルール

* ホストごとに1つのGatewayを推奨します。これはWhatsApp Webセッションを所有する唯一のプロセスです。救助ボットや厳格な分離のためには、隔離されたプロファイルとポートで複数のGatewayを実行してください。詳細は[複数のGateway](/gateway/multiple-gateways)を参照してください。
* ループバックを最初に: Gateway WSはデフォルトで`ws://127.0.0.1:18789`です。ウィザードはデフォルトでGatewayトークンを生成します。ループバックの場合でも同様です。tailnetアクセスのためには、`openclaw gateway --bind tailnet --token ...`を実行してください。トークンは非ループバックバインドに必要です。
* ノードは必要に応じてLAN、tailnet、またはSSH経由でGateway WSに接続します。レガシーTCPブリッジは非推奨です。
* CanvasホストはGateway HTTPサーバーによって、Gatewayと**同じポート**（デフォルトは`18789`）で提供されます:
  * `/__openclaw__/canvas/`
  * `/__openclaw__/a2ui/`
    `gateway.auth`が設定され、Gatewayがループバックを超えてバインドされると、これらのルートはGateway認証によって保護されます。ノードクライアントは、アクティブなWSセッションに結びついたノードスコープの能力URLを使用します。詳細は[Gateway設定](/gateway/configuration)を参照してください（`canvasHost`、`gateway`）。
* リモート使用は通常SSHトンネルまたはtailnet VPNです。詳細は[リモートアクセス](/gateway/remote)および[発見](/gateway/discovery)を参照してください。
