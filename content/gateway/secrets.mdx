# シークレット管理

# Secrets management

OpenClawは追加的なシークレット参照をサポートしているため、資格情報を設定ファイルにプレーンテキストとして保存する必要はありません。

プレーンテキストも機能します。シークレット参照はオプションです。

## 目標とランタイムモデル

シークレットはメモリ内のランタイムスナップショットに解決されます。

* 解決はアクティベーション中に即時に行われ、リクエストパスでは遅延しません。
* 参照された資格情報の解決に失敗した場合、スタートアップはすぐに失敗します。
* リロードは原子的なスワップを使用します：完全な成功または最後の既知の良好な状態を保持します。
* ランタイムリクエストはアクティブなメモリ内スナップショットから読み取ります。

これにより、シークレットプロバイダーの障害がホットリクエストパスから除外されます。

## オンボーディングリファレンスのプレフライト

オンボーディングがインタラクティブモードで実行され、シークレット参照ストレージを選択すると、OpenClawは保存前に迅速なプレフライトチェックを実行します：

* 環境参照：環境変数名を検証し、オンボーディング中に非空の値が表示されることを確認します。
* プロバイダー参照（`file`または`exec`）：選択したプロバイダーを検証し、提供された`id`を解決し、値のタイプをチェックします。

検証に失敗した場合、オンボーディングはエラーを表示し、再試行を許可します。

## SecretRef契約

どこでも同じオブジェクト形状を使用します：

```json5
{ source: "env" | "file" | "exec", provider: "default", id: "..." }
```

### `source: "env"`

```json5
{ source: "env", provider: "default", id: "OPENAI_API_KEY" }
```

検証：

* `provider`は`^[a-z][a-z0-9_-]{0,63}$`に一致する必要があります。
* `id`は`^[A-Z][A-Z0-9_]{0,127}$`に一致する必要があります。

### `source: "file"`

```json5
{ source: "file", provider: "filemain", id: "/providers/openai/apiKey" }
```

検証：

* `provider`は`^[a-z][a-z0-9_-]{0,63}$`に一致する必要があります。
* `id`は絶対JSONポインタである必要があります（`/...`）。
* セグメント内のRFC6901エスケープ：`~` => `~0`, `/` => `~1`

### `source: "exec"`

```json5
{ source: "exec", provider: "vault", id: "providers/openai/apiKey" }
```

検証：

* `provider`は`^[a-z][a-z0-9_-]{0,63}$`に一致する必要があります。
* `id`は`^[A-Za-z0-9][A-Za-z0-9._:/-]{0,255}$`に一致する必要があります。

## プロバイダー設定

`secrets.providers`の下にプロバイダーを定義します：

```json5
{
  secrets: {
    providers: {
      default: { source: "env" },
      filemain: {
        source: "file",
        path: "~/.openclaw/secrets.json",
        mode: "json", // または "singleValue"
      },
      vault: {
        source: "exec",
        command: "/usr/local/bin/openclaw-vault-resolver",
        args: ["--profile", "prod"],
        passEnv: ["PATH", "VAULT_ADDR"],
        jsonOnly: true,
      },
    },
    defaults: {
      env: "default",
      file: "filemain",
      exec: "vault",
    },
    resolution: {
      maxProviderConcurrency: 4,
      maxRefsPerProvider: 512,
      maxBatchBytes: 262144,
    },
  },
}
```

### 環境プロバイダー

* オプションの許可リストを`allowlist`経由で設定可能。
* 環境値が欠落または空の場合、解決に失敗します。

### ファイルプロバイダー

* `path`からローカルファイルを読み取ります。
* `mode: "json"`はJSONオブジェクトペイロードを期待し、`id`をポインタとして解決します。
* `mode: "singleValue"`は参照ID`"value"`を期待し、ファイルの内容を返します。
* パスは所有権/権限チェックに合格する必要があります。

### Execプロバイダー

* 設定された絶対バイナリパスを実行し、シェルは使用しません。
* デフォルトでは、`command`は通常のファイルを指す必要があります（シンボリックリンクではない）。
* シンボリックリンクコマンドパスを許可するには、`allowSymlinkCommand: true`を設定します（例えば、Homebrewのシム）。OpenClawは解決されたターゲットパスを検証します。
* 信頼できるパッケージマネージャーパスに必要な場合のみ`allowSymlinkCommand`を有効にし、`trustedDirs`と組み合わせます（例えば`["/opt/homebrew"]`）。
* `trustedDirs`が設定されている場合、チェックは解決されたターゲットパスに適用されます。
* タイムアウト、出力なしタイムアウト、出力バイト制限、環境許可リスト、信頼できるディレクトリをサポートします。
* リクエストペイロード（stdin）：

```json
{ "protocolVersion": 1, "provider": "vault", "ids": ["providers/openai/apiKey"] }
```

* レスポンスペイロード（stdout）：

```json
{ "protocolVersion": 1, "values": { "providers/openai/apiKey": "sk-..." } }
```

オプションのIDごとのエラー：

```json
{
  "protocolVersion": 1,
  "values": {},
  "errors": { "providers/openai/apiKey": { "message": "not found" } }
}
```

## Exec統合の例

### 1Password CLI

```json5
{
  secrets: {
    providers: {
      onepassword_openai: {
        source: "exec",
        command: "/opt/homebrew/bin/op",
        allowSymlinkCommand: true, // Homebrewのシンボリックリンクバイナリに必要
        trustedDirs: ["/opt/homebrew"],
        args: ["read", "op://Personal/OpenClaw QA API Key/password"],
        passEnv: ["HOME"],
        jsonOnly: false,
      },
    },
  },
  models: {
    providers: {
      openai: {
        baseUrl: "https://api.openai.com/v1",
        models: [{ id: "gpt-5", name: "gpt-5" }],
        apiKey: { source: "exec", provider: "onepassword_openai", id: "value" },
      },
    },
  },
}
```

### HashiCorp Vault CLI

```json5
{
  secrets: {
    providers: {
      vault_openai: {
```

## 対象フィールド（v1）

### `~/.openclaw/openclaw.json`

* `models.providers.<provider>.apiKey`
* `skills.entries.<skillKey>.apiKey`
* `channels.googlechat.serviceAccount`
* `channels.googlechat.serviceAccountRef`
* `channels.googlechat.accounts.<accountId>.serviceAccount`
* `channels.googlechat.accounts.<accountId>.serviceAccountRef`

### `~/.openclaw/agents/<agentId>/agent/auth-profiles.json`

* `profiles.<profileId>.keyRef` for `type: "api_key"`
* `profiles.<profileId>.tokenRef` for `type: "token"`

OAuth資格情報ストレージの変更は範囲外です。

## 必要な動作と優先順位

* 参照のないフィールド：変更なし。
* 参照のあるフィールド：アクティベーション時に必要。
* プレーンテキストと参照の両方が存在する場合、参照がランタイムで優先され、プレーンテキストは無視されます。

警告コード：

* `SECRETS_REF_OVERRIDES_PLAINTEXT`

## アクティベーショントリガー

シークレットアクティベーションは以下のタイミングで試みられます：

* スタートアップ（プレフライトおよび最終アクティベーション）
* 設定リロードのホット適用パス
* 設定リロードの再起動チェックパス
* `secrets.reload`による手動リロード

アクティベーション契約：

* 成功するとスナップショットが原子的にスワップされます。
* スタートアップの失敗はゲートウェイのスタートアップを中止します。
* ランタイムリロードの失敗は最後の既知の良好なスナップショットを保持します。

## 劣化および回復オペレーター信号

リロード時のアクティベーションが健康な状態の後に失敗した場合、OpenClawは劣化したシークレット状態に入ります。

一回限りのシステムイベントおよびログコード：

* `SECRETS_RELOADER_DEGRADED`
* `SECRETS_RELOADER_RECOVERED`

動作：

* 劣化：ランタイムは最後の既知の良好なスナップショットを保持します。
* 回復：成功したアクティベーションの後に一度だけ発生します。
* すでに劣化している間の繰り返しの失敗は警告をログに記録しますが、イベントをスパムしません。
* スタートアップの失敗は、まだランタイムスナップショットが存在しないため、劣化イベントを発生させません。

## 監査および設定ワークフロー

このデフォルトのオペレーターフローを使用します：

```bash
openclaw secrets audit --check
openclaw secrets configure
openclaw secrets audit --check
```

移行の完全性：

* APIキーを使用するスキルがある場合、`skills.entries.<skillKey>.apiKey`ターゲットを含めます。
* 部分的な移行後に`audit --check`がプレーンテキストの発見を報告し続ける場合、残りの報告されたパスを移行し、監査を再実行します。

### `secrets audit`

発見には以下が含まれます：

* 静止状態のプレーンテキスト値（`openclaw.json`、`auth-profiles.json`、`.env`）
* 解決されていない参照
* 優先順位の影（`auth-profiles`が設定参照より優先される）
* レガシー残留物（`auth.json`、OAuthの範囲外のリマインダー）

### `secrets configure`

インタラクティブヘルパー：

* 最初に`secrets.providers`を設定します（`env`/`file`/`exec`、追加/編集/削除）
* `openclaw.json`内のシークレットを持つフィールドを選択できます
* SecretRefの詳細をキャプチャします（`source`、`provider`、`id`）
* プレフライト解決を実行します
* すぐに適用できます

便利なモード：

* `openclaw secrets configure --providers-only`
* `openclaw secrets configure --skip-provider-setup`

`configure`の適用はデフォルトで：

* 対象プロバイダーに対して`auth-profiles.json`から一致する静的資格情報を削除します
* `auth.json`からレガシー静的`api_key`エントリを削除します
* `<config-dir>/.env`から一致する既知のシークレット行を削除します

### `secrets apply`

保存されたプランを適用します：

```bash
openclaw secrets apply --from /tmp/openclaw-secrets-plan.json
openclaw secrets apply --from /tmp/openclaw-secrets-plan.json --dry-run
```

厳密なターゲット/パス契約の詳細および正確な拒否ルールについては、以下を参照してください：

* [Secrets Apply Plan Contract](/gateway/secrets-plan-contract)

## 一方向の安全ポリシー

OpenClawは意図的に、移行前のプレーンテキストシークレット値を含むロールバックバックアップを作成しません。

安全モデル：

* 書き込みモードの前にプレフライトが成功する必要があります。
* コミットの前にランタイムアクティベーションが検証されます。
* 原子的なファイル置換を使用してファイルを更新し、失敗時には最善の努力でメモリ内の復元を行います。

## `auth.json`の互換性ノート

静的資格情報について、OpenClawのランタイムはもはやプレーンテキストの`auth.json`に依存しません。

* ランタイム資格情報のソースは解決されたメモリ内スナップショットです。
* レガシー`auth.json`の静的`api_key`エントリは発見されたときに削除されます。
* OAuth関連のレガシー互換性動作は別個のものとして残ります。
