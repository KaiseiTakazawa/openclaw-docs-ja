# Renderへのデプロイ

Infrastructure as Codeを使用してOpenClawをRenderにデプロイします。含まれている`render.yaml` Blueprintは、サービス、ディスク、環境変数を宣言的に定義し、コードと一緒にインフラストラクチャをバージョン管理しながら、ワンクリックでデプロイできるようにします。

## 前提条件

* [Renderアカウント](https://render.com)（無料プランあり）
* お好みの[モデルプロバイダー]( /providers)からのAPIキー

## Render Blueprintでデプロイ

[Renderにデプロイ](https://render.com/deploy?repo=https://github.com/openclaw/openclaw)

このリンクをクリックすると：

1. このリポジトリのルートにある`render.yaml` Blueprintから新しいRenderサービスが作成されます。
2. `SETUP_PASSWORD`の設定を求められます。
3. Dockerイメージをビルドしてデプロイします。

デプロイが完了すると、サービスのURLは`https://<service-name>.onrender.com`のパターンに従います。

## Blueprintの理解

Render Blueprintsはインフラストラクチャを定義するYAMLファイルです。このリポジトリの`render.yaml`は、OpenClawを実行するために必要なすべての設定を行います：

```yaml
services:
  - type: web
    name: openclaw
    runtime: docker
    plan: starter
    healthCheckPath: /health
    envVars:
      - key: PORT
        value: "8080"
      - key: SETUP_PASSWORD
        sync: false # デプロイ中にプロンプトが表示されます
      - key: OPENCLAW_STATE_DIR
        value: /data/.openclaw
      - key: OPENCLAW_WORKSPACE_DIR
        value: /data/workspace
      - key: OPENCLAW_GATEWAY_TOKEN
        generateValue: true # セキュアなトークンを自動生成します
    disk:
      name: openclaw-data
      mountPath: /data
      sizeGB: 1
```

使用される主要なBlueprint機能：

| 機能                  | 目的                                                    |
| --------------------- | -------------------------------------------------------- |
| `runtime: docker`     | リポジトリのDockerfileからビルドします                   |
| `healthCheckPath`     | Renderは`/health`を監視し、健康でないインスタンスを再起動します |
| `sync: false`         | デプロイ中に値を求めます（秘密情報）                     |
| `generateValue: true` | 暗号的に安全な値を自動生成します                         |
| `disk`                | 再デプロイを生き延びる永続ストレージ                     |

## プランの選択

| プラン      | スピンダウン         | ディスク          | 最適な用途                      |
| ----------- | ------------------- | ----------------- | ------------------------------- |
| 無料        | 15分アイドル後      | 利用不可          | テスト、デモ                    |
| スターター  | 決してスピンダウンしない | 1GB以上           | 個人利用、小規模チーム          |
| スタンダード+ | 決してスピンダウンしない | 1GB以上           | 本番、複数チャネル              |

Blueprintのデフォルトは`starter`です。無料プランを使用するには、フォークした`render.yaml`の`plan: free`を変更してください（ただし、永続ディスクがないため、各デプロイ時に設定がリセットされることに注意してください）。

## デプロイ後

### セットアップウィザードの完了

1. `https://<your-service>.onrender.com/setup`に移動します。
2. `SETUP_PASSWORD`を入力します。
3. モデルプロバイダーを選択し、APIキーを貼り付けます。
4. オプションでメッセージングチャネル（Telegram、Discord、Slack）を設定します。
5. **セットアップを実行**をクリックします。

### コントロールUIへのアクセス

ウェブダッシュボードは`https://<your-service>.onrender.com/openclaw`で利用できます。

## Renderダッシュボードの機能

### ログ

**ダッシュボード → あなたのサービス → ログ**でリアルタイムのログを表示します。フィルタリング可能な項目：

* ビルドログ（Dockerイメージの作成）
* デプロイログ（サービスの起動）
* ランタイムログ（アプリケーションの出力）

### シェルアクセス

デバッグのために、**ダッシュボード → あなたのサービス → シェル**を通じてシェルセッションを開きます。永続ディスクは`/data`にマウントされています。

### 環境変数

**ダッシュボード → あなたのサービス → 環境**で変数を変更します。変更は自動的に再デプロイをトリガーします。

### 自動デプロイ

元のOpenClawリポジトリを使用している場合、RenderはOpenClawを自動デプロイしません。更新するには、ダッシュボードから手動でBlueprintの同期を実行してください。

## カスタムドメイン

1. **ダッシュボード → あなたのサービス → 設定 → カスタムドメイン**に移動します。
2. ドメインを追加します。
3. 指示に従ってDNSを設定します（CNAMEを`*.onrender.com`に設定）。
4. Renderは自動的にTLS証明書をプロビジョニングします。

## スケーリング

Renderは水平および垂直スケーリングをサポートしています：

* **垂直**：プランを変更してCPU/RAMを増やします。
* **水平**：インスタンス数を増やします（スタンダードプラン以上）。

OpenClawの場合、通常は垂直スケーリングで十分です。水平スケーリングにはスティッキーセッションまたは外部状態管理が必要です。

## バックアップと移行

いつでも設定とワークスペースをエクスポートできます：

```
https://<your-service>.onrender.com/setup/export
```

これにより、任意のOpenClawホストに復元できるポータブルバックアップがダウンロードされます。

## トラブルシューティング

### サービスが起動しない

Renderダッシュボードでデプロイログを確認してください。一般的な問題：

* `SETUP_PASSWORD`が欠落 — Blueprintがこれを求めますが、設定されていることを確認してください。
* ポートの不一致 — `PORT=8080`がDockerfileの公開ポートと一致していることを確認してください。

### 冷却スタートが遅い（無料プラン）

無料プランのサービスは15分の非アクティブ後にスピンダウンします。スピンダウン後の最初のリクエストは、コンテナが起動する間に数秒かかります。常時稼働するにはスタータープランにアップグレードしてください。

### 再デプロイ後のデータ損失

これは無料プランで発生します（永続ディスクなし）。有料プランにアップグレードするか、定期的に`/setup/export`を介して設定をエクスポートしてください。

### ヘルスチェックの失敗

Renderは30秒以内に`/health`から200の応答を期待します。ビルドが成功してもデプロイが失敗する場合、サービスの起動に時間がかかっている可能性があります。以下を確認してください：

* エラーログのビルドログ
* `docker build && docker run`でローカルでコンテナが実行されるかどうか
