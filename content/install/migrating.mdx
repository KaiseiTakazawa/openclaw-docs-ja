# 移行ガイド

# OpenClawを新しいマシンに移行する

このガイドでは、**オンボーディングをやり直さずに** OpenClaw Gatewayを1台のマシンから別のマシンに移行します。

移行は概念的にはシンプルです：

* **状態ディレクトリ** (`$OPENCLAW_STATE_DIR`, デフォルト: `~/.openclaw/`) をコピーします — これには設定、認証、セッション、およびチャネルの状態が含まれます。
* **ワークスペース** (`~/.openclaw/workspace/`がデフォルト) をコピーします — これにはエージェントファイル（メモリ、プロンプトなど）が含まれます。

しかし、**プロファイル**、**権限**、および**部分コピー**に関しては一般的な落とし穴があります。

## 始める前に（移行するもの）

### 1) 状態ディレクトリを特定する

ほとんどのインストールはデフォルトを使用します：

* **状態ディレクトリ:** `~/.openclaw/`

ただし、次のような場合は異なる場合があります：

* `--profile <name>` （通常は `~/.openclaw-<profile>/` になります）
* `OPENCLAW_STATE_DIR=/some/path`

不明な場合は、**古い**マシンで次のコマンドを実行します：

```bash
openclaw status
```

出力に `OPENCLAW_STATE_DIR` / プロファイルの言及があるか確認してください。複数のゲートウェイを実行している場合は、各プロファイルについて繰り返してください。

### 2) ワークスペースを特定する

一般的なデフォルト：

* `~/.openclaw/workspace/` （推奨ワークスペース）
* あなたが作成したカスタムフォルダ

ワークスペースには、`MEMORY.md`、`USER.md`、および `memory/*.md` のようなファイルが存在します。

### 3) 保存するものを理解する

**状態ディレクトリ**と**ワークスペース**の両方をコピーすると、次のものを保持します：

* Gatewayの設定 (`openclaw.json`)
* 認証プロファイル / APIキー / OAuthトークン
* セッション履歴 + エージェントの状態
* チャネルの状態（例：WhatsAppのログイン/セッション）
* あなたのワークスペースファイル（メモリ、スキルノートなど）

**ワークスペース**のみをコピーした場合（例：Git経由）、次のものは保持されません：

* セッション
* 認証情報
* チャネルのログイン情報

それらは `$OPENCLAW_STATE_DIR` の下にあります。

## 移行手順（推奨）

### ステップ0 — バックアップを作成する（古いマシン）

**古い**マシンで、まずゲートウェイを停止して、ファイルがコピー中に変更されないようにします：

```bash
openclaw gateway stop
```

（オプションですが推奨）状態ディレクトリとワークスペースをアーカイブします：

```bash
# プロファイルやカスタムの場所を使用している場合はパスを調整してください
cd ~
tar -czf openclaw-state.tgz .openclaw

tar -czf openclaw-workspace.tgz .openclaw/workspace
```

複数のプロファイル/状態ディレクトリ（例：`~/.openclaw-main`、`~/.openclaw-work`）がある場合は、それぞれをアーカイブします。

### ステップ1 — 新しいマシンにOpenClawをインストールする

**新しい**マシンで、CLI（必要に応じてNodeも）をインストールします：

* 詳細: [Install](/install)

この段階では、オンボーディングが新しい `~/.openclaw/` を作成しても問題ありません — 次のステップで上書きします。

### ステップ2 — 状態ディレクトリ + ワークスペースを新しいマシンにコピーする

**両方**をコピーします：

* `$OPENCLAW_STATE_DIR` （デフォルト `~/.openclaw/`）
* あなたのワークスペース（デフォルト `~/.openclaw/workspace/`）

一般的なアプローチ：

* `scp` でtarボールを転送して展開
* SSH経由で `rsync -a`
* 外部ドライブ

コピー後は、次のことを確認してください：

* 隠しディレクトリが含まれていること（例：`.openclaw/`）
* ゲートウェイを実行しているユーザーのファイル所有権が正しいこと

### ステップ3 — Doctorを実行する（移行 + サービス修復）

**新しい**マシンで：

```bash
openclaw doctor
```

Doctorは「安全で退屈な」コマンドです。サービスを修復し、設定の移行を適用し、不一致について警告します。

次に：

```bash
openclaw gateway restart
openclaw status
```

## 一般的な落とし穴（およびそれを避ける方法）

### 落とし穴: プロファイル / 状態ディレクトリの不一致

古いゲートウェイをプロファイル（または `OPENCLAW_STATE_DIR`）で実行し、新しいゲートウェイが異なるものを使用している場合、次のような症状が見られます：

* 設定変更が反映されない
* チャネルが欠落 / ログアウトされる
* 空のセッション履歴

修正: 移行した**同じ**プロファイル/状態ディレクトリを使用してゲートウェイ/サービスを実行し、次を再実行します：

```bash
openclaw doctor
```

### 落とし穴: `openclaw.json` のみをコピーする

`openclaw.json` だけでは不十分です。多くのプロバイダーは状態を次の場所に保存します：

* `$OPENCLAW_STATE_DIR/credentials/`
* `$OPENCLAW_STATE_DIR/agents/<agentId>/...`

常に `$OPENCLAW_STATE_DIR` フォルダ全体を移行してください。

### 落とし穴: 権限 / 所有権

rootとしてコピーしたり、ユーザーを変更した場合、ゲートウェイが認証情報/セッションを読み取れない可能性があります。

修正: 状態ディレクトリ + ワークスペースがゲートウェイを実行しているユーザーによって所有されていることを確認してください。

### 落とし穴: リモート/ローカルモード間の移行

* UI（WebUI/TUI）が**リモート**ゲートウェイを指している場合、リモートホストがセッションストア + ワークスペースを所有します。
* ノートパソコンを移行してもリモートゲートウェイの状態は移動しません。

リモートモードの場合は、**ゲートウェイホスト**を移行してください。

### 落とし穴: バックアップ内の秘密

`$OPENCLAW_STATE_DIR` には秘密（APIキー、OAuthトークン、WhatsAppの認証情報）が含まれています。バックアップは本番の秘密のように扱ってください：

* 暗号化して保存
* 安全でないチャネルで共有を避ける
* 露出が疑われる場合はキーをローテーションする

## 検証チェックリスト

新しいマシンで確認してください：

* `openclaw status` がゲートウェイが実行中であることを示す
* あなたのチャネルがまだ接続されている（例：WhatsAppが再ペアリングを必要としない）
* ダッシュボードが開き、既存のセッションを表示する
* あなたのワークスペースファイル（メモリ、設定）が存在する
